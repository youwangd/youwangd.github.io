I Built a System-Design Wiki So My Agents Stop Hallucinating Architecture

The agent told me to put Redis in front of a write-heavy path. Cited nothing. Had no idea Netflix published a post-mortem two years ago about exactly why that breaks at scale. I got tired of arguing with it. So I built it a wiki.

Your agent doesn't know system design

It knows tokens that appear near the phrase "system design." Not the same thing. In 2025 nobody cared because the code was mostly CRUD. In 2026 the agent is writing whole services, and "mostly right" means your circuit breaker has no jitter, your sharding key picks a hot cell, your event schema has no versioning story. Every one is a production outage. Every one is documented on someone's engineering blog. The agent read those posts during training. It doesn't remember them.

The obvious fix is RAG. I tried it. It's worse than you think — chunks come back context-free, rerankers hallucinate relevance, and the agent cites a Meta video-encoding post to justify a database decision. RAG retrieves. It doesn't compile.

Karpathy's LLM wiki post names the alternative: distill sources into a persistent, interlinked markdown graph. The agent reads the wiki, not the corpus. I built one for system design. Two months in: 3,210 pages, 280 sources, auto-ingest every 6 hours, static site on Cloudflare Pages. Live at sysdesign-wiki.pages.dev.

The whole thing fits in a cron job

No database. No vector store. Three directories and a rule: raw is immutable, wiki is the product, nothing else exists.

sysdesign-wiki/
├── raw/      // scraped articles, one dir per company
├── wiki/     // the compiled knowledge — the product
│   ├── concepts/ patterns/ systems/ companies/
│   └── sources/   // one page per ingested article
└── tools/    // scraper, ingest, lint, build

The publish pipeline is 30 lines of bash in a cron:

#!/usr/bin/env bash
set -euo pipefail
cd /home/dyouwang/sysdesign-wiki
exec 9>/tmp/sysdesign-wiki.lock
flock -n 9 || exit 0

python tools/rss_pull.py              # 29 feeds → raw/
python tools/ingest.py --incremental  # raw/ → wiki/sources/
python tools/gen_category_indexes.py
python tools/gen_landing_page.py

git add -A
git diff --cached --quiet || {
  git commit -m "auto-ingest $(date -u +%FT%H:%MZ)"
  git push
}

Cloudflare Pages rebuilds on push. Site is live in 90 seconds. The curator agent — running via sage — reads each raw article and writes a structured source page: one-paragraph distillation, 3-5 takeaways, [[wiki-links]] to the concept and pattern pages it touches. Every non-obvious claim cites its source. When companies disagree, a ## Contradiction section names both positions. No fake consensus.

Why this beats RAG

Same prompt, two runs: "design a 100k events/sec write path, justify with industry precedent."

RAG pulled six chunks — a LinkedIn post on consumer lag, a Stripe idempotency fragment, an AWS tuning guide, and two unrelated Netflix video-encoding posts because "high throughput" appeared in both. The design sounded authoritative and was internally incoherent. It cited the video-encoding post to defend a DB choice.

Wiki opened patterns/write-ahead-buffering.md, systems/kafka.md, concepts/backpressure.md. Already cross-linked. Every claim already cited. The agent wrote four paragraphs with six citations that all checked out, and defended a choice by pointing at a ## Contradiction section: "I picked LinkedIn's position because your traffic profile matches their case study."

Same model both times. The wiki had already done the thinking. That's the difference.

Compiled knowledge is the moat

A corpus doesn't compound. Bigger corpus, bigger haystack. A wiki compounds — when Uber, Square, and Shopify all publish on idempotency, patterns/idempotency.md doesn't quadruple, it gets sharper. "Stripe does X" becomes "the industry converged on X because Y, except at Uber scale where Z." Every future agent task inherits that sharpening for free.

OpenAI's Frontier team shipped a million lines of zero-human-written code by making AGENTS.md a table of contents and putting the encyclopedia in docs/. That directory is a wiki — they just didn't call it one. The durable artifact wasn't the code the agent shipped. It was the scaffolding a human compiled so the agent could navigate instead of memorize.

The moat for serious agent work isn't the model, the prompt, or the harness. It's the compiled knowledge the agent reads before it writes a line of code. A bigger context window won't save you. Only a compiled, cited, current knowledge base will.

Minimum viable version

You don't need 3,210 pages. Start with ten.

mkdir -p wiki/{concepts,patterns,systems,sources}
cat > AGENTS.md <<'EOF'
You maintain a knowledge base on <your domain>.
Read wiki/index.md first. Cite sources for every non-obvious claim.
When sources conflict, add a ## Contradiction section.
EOF

Write ten concept pages by hand, 500 words each — one-line summary, 3 takeaways, 3 citations. Point your agent at the directory. Watch it stop hallucinating. Automate the ingest when you run out of patience.

Your job is to compile knowledge. The agent's job is to use it. Stop arguing with it about the circuit breaker. Write the page.
