I Built a System-Design Wiki So My Agents Stop Hallucinating Architecture

The agent was confidently telling me to put a Redis cache in front of a write-heavy path. It cited nothing. It had no idea Netflix had published a post-mortem two years ago explaining exactly why that pattern falls over at their scale. The agent didn't know, because the agent had read the internet once, a year ago, compressed it into weights, and now hallucinates whenever the weights are thin. I got tired of arguing with it. So I built it a wiki.

The premise nobody in AI-coding discourse wants to say out loud

Your agent does not know system design. It knows tokens that appear near the phrase "system design." That is not the same thing.

In April 2025 nobody cared about the distinction — the code you wrote was mostly CRUD and the agent mostly got it right. In April 2026 the agent is writing entire services, and "mostly right" means your circuit breaker has no jitter, your sharding key picks a hot cell, and your event schema has no versioning story. Every one of those is a real production outage waiting to happen. Every one is documented somewhere in Netflix, Uber, Stripe, or Cloudflare's engineering blog. The agent has seen those posts during training. It does not remember them.

The obvious answer is RAG — dump engineering blogs into a vector DB, retrieve on demand. I tried it. It's worse than you think. The chunks come back context-free, the rerankers hallucinate relevance, and your agent ends up citing a Meta post about video encoding to justify a decision about database replication. RAG retrieves. It does not compile.

Karpathy has been pushing a different idea he calls the LLM wiki: don't retrieve raw documents, distill them into a persistent, interlinked knowledge graph in markdown. The agent reads the wiki, not the corpus. The wiki is the artifact. The corpus is just the source of truth it was compiled from.

I built one for software system design. Two months in, 3,210 pages, 280 curated sources, hourly auto-ingest, deployed on Cloudflare Pages. Live at sysdesign-wiki.pages.dev. Here's what it actually takes and why I think this is the shape knowledge will take in the AI-coding era.

Three layers, one rule

The architecture is boring on purpose:

sysdesign-wiki/
├── raw/         // immutable scraped articles, one per company
│   ├── netflix/ uber/ meta/ aws/ cloudflare/ stripe/
│   └── _feeds.yaml
├── wiki/        // the compiled knowledge base — the product
│   ├── concepts/    // sharding, consensus, CAP, CRDTs, backpressure
│   ├── patterns/    // circuit-breaker, saga, CQRS, bulkhead
│   ├── systems/     // Kafka, Spanner, Dynamo, TAO, Haystack, Envoy
│   ├── companies/   // per-company stack + lessons summary
│   └── sources/     // one page per ingested article
└── tools/       // scraper, rss poller, ingest, lint, shrink

The rule: raw is immutable, wiki is the product, nothing else exists. No database. No vector store. No search backend beyond what mkdocs ships. When I want to know "what does Netflix do for regional failover," I don't query embeddings. I open wiki/concepts/regional-failover.md and it's three paragraphs with four citations linking back to raw/netflix/. That's it.

Every non-obvious claim cites its source: (Source: [[sources/2026-01-15-netflix-chaos]]). When two companies disagree — and they do, constantly — there's a ## Contradiction section that names both positions. No synthesis pretends consensus exists where it doesn't.

The ingest pipeline is 30 lines of bash

I'm going to show you the whole thing because people keep writing Kubernetes-shaped solutions for problems that fit in a cron job.

# cron_publish.sh — runs every 6 hours
#!/usr/bin/env bash
set -euo pipefail

cd /home/dyouwang/sysdesign-wiki
exec 9>/tmp/sysdesign-wiki.lock
flock -n 9 || exit 0   # no overlapping runs

python tools/rss_pull.py              # 29 feeds → raw/
python tools/ingest.py --incremental  # raw/ → wiki/sources/
python tools/gen_category_indexes.py  # auto-index pages
python tools/gen_landing_page.py      # update homepage stats

git add -A
git diff --cached --quiet || {
  git commit -m "auto-ingest $(date -u +%FT%H:%MZ)"
  git push
}

Cloudflare Pages does the rest. A push triggers a build, mkdocs renders the static site, the search index gets post-processed to fit under the 25 MiB per-file limit, and the site is live in 90 seconds. No server. No daemon. The most expensive part of this system is the GitHub free tier.

The ingest script does the interesting part: it pulls RSS for 29 engineering blogs, and for the 16 blogs that block feed readers (Meta, Google, LinkedIn are the usual suspects), an HTML scraper with a residential user-agent does the job. Each article lands as a raw markdown file. Then a curator agent — running via sage in the background — reads the raw file and writes a structured source page: title, one-paragraph distillation, 3-5 takeaways, and a list of [[wiki-link]] references to the concept/pattern/system pages the article touches.

The curator does one thing that matters more than anything else: it applies a scope filter before distilling. Off-topic articles get rejected on sight. Product launches pretending to be engineering posts. ML research thinly disguised as infra content. Vendor PR. All skipped with a one-line rationale logged to wiki/log.md. Without this filter, the wiki would drown in noise within a week.

Why this beats RAG for architecture decisions

Here is a real prompt I ran twice — once with RAG over the raw corpus, once with the agent reading the compiled wiki:

I'm building a service that ingests 100k events/sec and writes to a primary DB. Design the write path. Justify every choice with industry precedent.

RAG version pulled six chunks: a LinkedIn blog about Kafka consumer lag, a fragment of a Stripe idempotency post, two paragraphs from an AWS DynamoDB tuning guide, and two completely unrelated Netflix posts about video encoding because the phrase "high throughput" appeared in both. The agent wrote a design that sounded authoritative and was internally incoherent. When I asked it to defend a specific choice, it cited the video encoding post. I would have shipped this if I hadn't been paying attention.

Wiki version opened wiki/patterns/write-ahead-buffering.md, wiki/systems/kafka.md, and wiki/concepts/backpressure.md. Every page was already cross-linked. Every claim already had a citation. The agent produced a design in four paragraphs with six citations and every one checked out. When I asked it to defend a choice, it pointed at a ## Contradiction section that named LinkedIn's position versus Uber's position and said "I chose LinkedIn's because your traffic profile matches their case study."

The difference isn't the model. Same model both times. The difference is that the wiki had already done the thinking. RAG retrieves fragments. The wiki hands over the compiled answer.

The compounding that nobody talks about

Here's the thing about a compiled knowledge base versus a corpus: corpora don't compound. A bigger corpus is just a bigger haystack. Every retrieval still starts from zero.

A wiki compounds. Every new source that gets ingested doesn't add noise — it gets distilled into existing pages. The first time I added an article about Stripe's idempotency keys, it created wiki/patterns/idempotency.md. The fourth time — when Uber, Square, and Shopify all published on the same pattern — the page didn't quadruple in length. It got sharper. Disagreements got named. The canonical trade-off got cleaner. The page went from "Stripe does X" to "the industry converged on X because Y, except at Uber scale where Z." That's a knowledge gain that a corpus can't give you.

And the agent, reading that page, inherits the sharpening. The curator did the work once. Every future agent task benefits.

What this means for anyone building with agents in 2026

I am going to say this bluntly because I think the industry is under-indexing on it. The moat for anyone doing serious work with AI coding agents is not the model, not the prompt, not the harness. It is the compiled knowledge the agent reads before it starts work.

OpenAI's Frontier team shipped a million lines of zero-human-written code by writing AGENTS.md files that were a table of contents, not an encyclopedia. The encyclopedia lived in a structured docs/ directory. That directory is a wiki. They didn't call it that, but that's what it is. And they were very clear: the durable artifact the team built wasn't the code the agent shipped. It was the scaffolding — the quality scores, the execution plans, the design docs. The stuff a human compiled so the agent could navigate instead of memorize.

Every serious agent operator I know is converging on the same shape:

  • A short AGENTS.md that is a map, not a manual (100 lines, not 2000)
  • A structured knowledge base the agent reads when it needs depth (wiki, docs/, design-docs/)
  • An ingest pipeline that keeps the knowledge base current without a human babysitter
  • A lint that catches drift between the code and the knowledge base

If you are shipping agents and you have none of these, your agent is hallucinating architecture right now and you are shipping it to production. The model will not save you. The harness will not save you. A bigger context window will not save you. Only a compiled, cited, current knowledge base will.

The minimum viable version

You don't need 3,210 pages. You need the ten concepts that matter for your system. Start there:

# a wiki in 20 minutes
mkdir -p wiki/{concepts,patterns,systems,sources}
cat > AGENTS.md <<'EOF'
You are the maintainer of a knowledge base on <your domain>.
Read wiki/index.md first. Every non-obvious claim cites a source.
When sources conflict, add a ## Contradiction section.
EOF
echo "# Index" > wiki/index.md
# now write 10 concept pages by hand, ~500 words each
# each page has: one-line summary, 3 takeaways, 3 citations
# this is more useful than any RAG you will build

That's the MVP. Ten pages, hand-written, cited. Point your agent at the directory. Watch it stop hallucinating. Then automate the ingest when you run out of patience writing pages.

The agent's job is to use knowledge. Your job is to compile it. The tooling is cheap. The weights are not the bottleneck. The bottleneck is whether the thing the agent reads before it writes a single line of code reflects what you actually know about how your systems fail.

Stop arguing with the agent about the circuit breaker. Write the page.
