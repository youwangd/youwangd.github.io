Harness Engineering: When the Code You Write Isn't Code

OpenAI's Frontier team shipped a million lines of code. Zero written by humans. 1,500 pull requests. Agents running six hours autonomously while the engineers slept. The headline is impressive. But the headline is also the wrong thing to focus on. The real story is what the humans did write: AGENTS.md files, skill definitions, linter rules, quality scores, execution plans. The scaffolding. And after building my own agent harness for the past two months, I'm convinced: the scaffolding is the product.

The 1M LOC Experiment

Ryan Lopopolo's harness engineering post dropped in February and became the defining document of a new discipline. The constraint was deliberate: no human-written code. Not because the engineers couldn't code, but because the constraint forced them to build everything an agent needs to succeed independently.

The numbers tell a story of compounding returns:

  • Month 1: 10x slower than writing code by hand. Painful. The agent lacked building blocks.
  • Month 3: 3.5 PRs per engineer per day. Humans still reviewing every PR.
  • Month 5: 5-10 PRs per engineer per day. Agents reviewing each other's code. Humans doing post-merge review only.

The inflection wasn't a better model. It was better scaffolding. Every time the agent failed, the team asked: "What capability, context, or structure is missing?" — and then built it. Not as a prompt tweak. As a durable artifact in the repository.

On the Latent Space podcast, Lopopolo described the mindset shift: "I can't really have deep code-level opinions about things. It's as if I'm group tech-leading a 500-person organization." The job became systems design, not code authorship.

AGENTS.md Is a Table of Contents, Not a Manual

This is the part most people get wrong. They hear "give the agent context" and write a 2,000-line AGENTS.md with every rule, convention, and architectural decision crammed in. OpenAI tried this. It failed.

A giant instruction file crowds out the task, the code, and the relevant docs — so the agent either misses key constraints or starts optimizing for the wrong ones. When everything is "important," nothing is.

Their solution: treat AGENTS.md as the table of contents, not the encyclopedia. Roughly 100 lines. Pointers to deeper sources of truth in a structured docs/ directory. Progressive disclosure — the agent starts with a small, stable entry point and learns where to look next.

I learned this the hard way building AGENTS.md for FeesDomainService at work. The first draft was a wall of text. The agent would either ignore half of it or get confused by contradictions between sections written weeks apart. What worked: a short map with links to canonical sources. The agent navigates. It doesn't memorize.

The structure that works:

AGENTS.md          // ~100 lines, the map
ARCHITECTURE.md    // domain boundaries, dependency rules
docs/
├── design-docs/   // indexed, versioned, with verification status
├── exec-plans/    // active plans, completed plans, tech debt
├── references/    // llms.txt files for key dependencies
├── QUALITY_SCORE.md
├── RELIABILITY.md
└── SECURITY.md

The key insight: anything the agent can't access in-context doesn't exist. That Slack discussion where the team aligned on an architectural pattern? If it's not in the repo, it's invisible. Same as it would be to a new hire joining three months later.

The Scaffolding IS the Product

Here's where it gets interesting. OpenAI's team has six skills. That's it. When something in the development loop isn't covered, their first move is to encode it in an existing skill — not create a new one. They change agent behavior more cheaply than they change human behavior.

I've been running MeshClaw — my own agent harness — for two months now. The pattern is identical. The things that compound aren't the code the agents produce. It's the meta-layer:

Scaffolding LayerWhat It DoesWhy It Compounds
SkillsReusable agent capabilities (blog-writer, web-search, qlib)Every new skill makes all future tasks cheaper
MemoryPersistent lessons, project context, semantic factsAgent stops repeating mistakes across sessions
Cron jobsScheduled autonomous work (research, monitoring, publishing)Value accrues while you sleep
HooksTriggers that refresh context on code changesKnowledge stays current without manual updates
Linters & testsMechanical enforcement of tasteEncode once, enforce on every line forever

The code the agents write? Disposable. Lopopolo said it directly: "I have way less attachment to the code as it is written. If it's garbage, I can just throw it away." Symphony — their Elixir-based orchestrator — has a "rework" state that completely trashes the work tree and starts from scratch. The code is ephemeral. The scaffolding is permanent.

This maps to something @badlogicgames (creator of libGDX, now building Pi) tweeted: "My next project is the terraform of cloud agents." Infrastructure for agent fleets. Not the agents themselves — the layer that manages them. The scaffolding.

Ghost Libraries and the Death of Dependencies

One of the wildest ideas from the podcast: ghost libraries. Instead of distributing software as packages, you distribute it as specs. A high-fidelity specification that a coding agent can reproduce locally. No dependency hell. No transitive vulnerabilities. No waiting for upstream patches.

Symphony itself is distributed this way. The flow:

  1. Agent reads the spec from the reference repo
  2. Agent implements the spec in a fresh worktree
  3. A second agent reviews the implementation against the spec
  4. Loop until the implementation matches with high fidelity

Brett Taylor (OpenAI's chairman) responded: "Software dependencies are going away." Lopopolo agreed — for low-to-medium complexity dependencies. Instead of pulling in a generic p-limit package, they have the agent reimplement exactly what they need, tightly integrated with their own telemetry, with 100% test coverage.

This is the end of bullshit plugins. The bloat that comes from being liberal in what you accept. When code is free, you can afford to be specific.

What This Means for You

You don't need OpenAI's budget or Codex access to start. The pattern is universal:

1. Start with a map, not a manual. Your AGENTS.md should be 100 lines that point to deeper docs. If you're writing rules in AGENTS.md, you're doing it wrong — write them in dedicated files and link to them.

2. Encode taste as lints, not docs. Documentation rots. Linters don't. When you catch an agent making a mistake, don't just fix it — write a lint rule with an error message that teaches the agent how to fix it next time. OpenAI writes custom linter error messages specifically for agent consumption.

3. Treat every agent failure as a missing capability. The question is never "how do I prompt this better?" It's "what tool, context, or structure would make this trivially solvable?" Then build that thing.

4. Make the build loop fast. OpenAI enforced a one-minute build time ceiling. When builds got slow, they retooled — from Make to Bazel to Turbo to NX — in a week. Because the agent doesn't have opinions about build systems. It just needs speed.

5. Invest in the meta-layer. Skills, memory, scheduled jobs, hooks. These are the things that compound. The code they produce is disposable. The scaffolding that produces the code is your actual product.

We're entering an era where the best engineers don't write the most code — they build the best environments for agents to write code in. The scaffolding is the product. Everything else is tokens.
