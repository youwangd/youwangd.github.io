The Friction Is Your Judgment: Why Agentic Speed Without Steering Produces Invisible Debt

Last week I pushed a CR through 8 rounds of automated static analysis. 42 comments. Hardcoded account IDs, region inconsistencies, security heading leaks, missing crypto guardrails. Every round, I wanted to skip the review and just ship. Every round, the friction caught something real. Armin Ronacher — the creator of Flask — gave a talk at AI Engineer London that crystallized why that friction matters: "The moments where you want to skip thinking are exactly the moments where thinking matters most."

The Psychology Trap

Here's what actually happens when you start coding with agents. Day one: magic. You describe a feature, the agent writes it, tests pass, you ship. Day two: you describe two features. Day five: you're approving PRs you haven't fully read because the agent "seems to get it." Day ten: something breaks in production and you can't explain why, because you stopped understanding your own codebase three days ago.

Ronacher nails the mechanism: agents produce output fast, and output feels like progress. Your brain rewards velocity. Shipping feels good. So you stop questioning, stop designing, stop saying "wait." You're not lazy — you're responding to a reward signal that's been hijacked.

The old balance was natural. You wrote code, then you reviewed code. Creation and evaluation were interleaved. Now every engineer has an agent amplifying their output, but responsibility hasn't been amplified. As Ronacher puts it:

You are outnumbered by entities that cannot carry responsibility.

This isn't a tooling problem. It's a cognitive one. The agent doesn't know when to stop, and once you're in the flow, neither do you.

The Entropy Spiral

The engineering consequence is worse than the psychological one, because it compounds silently.

Agents are optimized to produce code that runs. They over-protect locally: catching exceptions broadly, falling back to defaults, swallowing errors. A service silently running with wrong config is worse than a service that refuses to start — but the agent doesn't know that. It optimizes for green CI, not for correctness.

Ronacher describes the spiral:

Messier codebase → worse agent recall → more duplication → more mess.

Entropy is self-reinforcing. The agent pattern-matches against what's already there. If what's already there is three different ways to handle auth, the agent will invent a fourth. If error handling is inconsistent, the agent will be inconsistently inconsistent. The codebase drifts, and the drift accelerates because the agent uses the drifted code as its reference.

I've seen this firsthand. On a service with 50+ CDK stacks, the agent would copy patterns from the nearest similar stack — including bugs. A wrong region string (us-west-2 instead of eu-south-2) propagated through three files before static analysis caught it. The agent wasn't wrong per se — it was faithfully reproducing what it saw.

Designing Friction Back In

The fix isn't slowing down. It's making speed safe. Ronacher and Poncela propose three categories of mechanical friction:

1. Lint Rules as Guardrails

Not style lints. Judgment lints. Rules that force the agent (and you) to think:

  • No bare catch-alls — forces explicit error handling instead of swallowing failures
  • No raw SQL outside the abstraction layer — preserves the query interface
  • No dynamic imports — keeps the dependency graph legible
  • Unique function names enforced — nudges toward discovery over duplication
  • erasableSyntaxOnly in TypeScript — constrains what the agent can generate

The key insight: lint error messages are agent prompts. When a lint rule fires, the agent reads the error message and uses it to fix the code. Write your lint messages for agent consumption. Not "style violation on line 42" but "this function duplicates AuthHelper.validateToken() — use the existing helper instead."

2. PR Review Splitting

Not all review is equal. Ronacher splits it cleanly:

Immediately Actionable → AgentCall-outs → Human
Style violationsDatabase migrations
Mechanical bugsNew dependencies
Clear engineering rule breaksAuth/permissions changes
Backwards-incompatible API changes
Irreversible destructive operations

The agent handles the mechanical stuff. The human gets pulled in only for decisions that require judgment about consequences the agent can't model. This isn't about trust — it's about routing attention to where it has the highest leverage.

3. Agent-Legible Codebases

Your codebase is infrastructure for the agent. Design it that way:

  • Modularization with clear boundaries — agents work in one area without corrupting another
  • Known patterns and conventions — agents pattern-match; give them patterns worth matching
  • Simple core, complexity pushed to layers above
  • No hidden magic — if the agent can't see it, it can't respect it

Ronacher identifies where agents excel (compact API surface, tight constraints, stable core) versus where they struggle (interacting concerns like flags + permissions + billing, features that span multiple files, context that doesn't fit in one window). The implication: refactor your codebase to maximize the first category and minimize the second.

The .md Future and Its Trap

Guillermo Rauch (@rauchg) tweeted: "Design will become autonomous. Think of it as DESIGN.md." 1,178 likes. The vision: everything becomes markdown files for agents. AGENTS.md, DESIGN.md, ARCHITECTURE.md, QUALITY_SCORE.md.

He's right about the direction. But there's a trap nobody's talking about: who reviews the .md?

MCP adoption hit an inflection point this spring. As @dexhorthy noted: "In Jan/Feb nobody had their MCP story sorted out. Since then engineers expect agents to use gdocs, figma, glean, atlassian." Adoption without friction means rubber-stamped PRs at scale. The agent reads DESIGN.md, generates code that matches the spec, the PR passes automated review, and nobody notices that DESIGN.md itself was wrong.

Meanwhile, on r/LocalLLaMA, someone asked: "Are you guys actually using local tool calling or is it a collective prank?" The grassroots skepticism is healthy. The gap between "agents can use tools" and "agents use tools reliably enough to trust" is where all the interesting engineering lives.

What 8 Rounds of AutoSDE Taught Me

I spent a week pushing CR-267392534 through automated static analysis. AGENTS.md for a production service. Here's the timeline:

  • Round 1: 6 comments. Hardcoded AWS account IDs in documentation (security risk).
  • Round 2: 4 comments. ZAZ region inconsistency — us-west-2 where it should be eu-south-2.
  • Round 3: 5 comments. Security heading leaking internal infrastructure details.
  • Round 4: 5 comments. Missing crypto guardrails for serialization helpers.
  • Rounds 5-8: Diminishing returns — Known Gaps cleanup, git diff formatting, description mismatches.

Every single round, my instinct was to skip it. "It's just docs." "The account IDs are in internal docs anyway." "The region thing is cosmetic." But it wasn't cosmetic. The wrong region string would have confused every agent that read that file for context. The hardcoded account IDs would have been a security finding in the next audit.

The friction of iterative review IS the judgment. Not a substitute for judgment — the actual mechanism by which judgment gets applied. Without the friction, I would have shipped a document that actively made agents worse at their jobs.

The Uncomfortable Truth

Ronacher closes with something that deserves to be uncomfortable:

With agents, you can accumulate in days the debt that used to take months. That makes it much harder to judge the actual state of the codebase.

Speed is not the problem. Speed without steering is the problem. And steering requires friction — not the wasteful kind (waiting for CI, fighting merge conflicts, navigating bureaucratic review processes), but the productive kind: lint rules that encode taste, PR callouts that route human attention, modular architectures that limit blast radius, and iterative review loops that catch what you're too fast to notice.

Some friction is waste. Some friction is judgment. The skill of this era is knowing which is which — and having the discipline to keep the friction that makes you think, even when every instinct says skip it and ship.
