MCP Isn't Dead — It's Overprescribed: What 13 MCP Servers and 46 Skills Side-by-Side Actually Taught Us

On Saturday, Quandri Engineering posted "MCP is dead" and rode it to 195 points and 174 comments on Hacker News. The numbers are devastating: 77 MCP tool definitions across 4 servers eat 21,077 tokens — 10.5% of Claude's 200K window, 16.5% of GPT-4o's — before any work happens. A single Linear issue lookup costs ~65× more tokens via MCP than via curl (12,957 vs 200). Eric Holmes's original Feb manifesto adds the latency: 3× slower per call, 9.4× slower on the first. The takeaway rippling through every dev Slack right now is "rip out MCP, switch to CLI + Skills." Our runtime ships 13 MCP servers and 46 skills concurrently. After eight months of running them next to each other in anger, I can tell you the takeaway is half right — and the half that's wrong is the half being repeated.

Read the update box, not the headline

The lede everyone is sharing is the 21,077-token figure from Quandri's post. It's a real number measured against a real config. Linear MCP alone: 42 tools, 12,807 tokens. Their full stack: 84,308 characters of JSON schema injected at session start, every session, whether or not the agent ends up touching any of it. The HN thread caught fire because every developer who'd ever wired up an MCP server had felt the bloat without being able to point to a number. Now they could.

What almost nobody is sharing is the update box at the top of Quandri's own post:

"Claude Code has rolled out Tool Search with Deferred Loading, which loads MCP tool schemas on-demand and reduces context usage by 85%+. The context bloat described in Problem 1 is largely addressed for users on current Claude Code versions."

Claude Code shipped Deferred Loading two weeks before the post went viral. The headline data point in the obituary describes a state of the world that no longer exists for the most popular agent runtime on the market. The architectural fix shipped in the same month as the architectural complaint. "MCP is dead" is now an obituary for a corpse that walked off the table.

That doesn't make Quandri wrong. It makes them narrow. The cost they measured was real, but it was a load-time cost — and the runtimes are now load-time-aware. The remaining question isn't whether MCP costs too much in 2026. It's which servers earn their tokens, on which task, on which day. That question doesn't have a "rip it all out" answer. It has a four-case answer.

The actual config, with receipts

Here's the live mcpServers block from ~/.kiro/agents/meshclaw.json on this machine, today:

amazon-sharepoint-mcp     aws-outlook-mcp        browser
builder-mcp               clix                   design-inspector-mcp
mcp-mermaid               meshclaw-core          meshclaw-cron
nanobanana                qmd                    slack-mcp
spec-studio-mcp

13 servers. Plus 46 skills under ~/.meshclaw/skills/, most of which wrap a CLI: git, gh, aws, brazil, cr, kubectl, jq, opencli, plantuml. The two populations live in the same agent. They are not in competition. They are differently shaped tools for differently shaped problems, and after running them side-by-side for eight months I can name exactly four cases where MCP earns its tokens. Outside those cases, a Skill that shells out to a CLI wins on every metric I can measure: tokens, latency, debuggability, blast radius.

The four cases where MCP wins

Case 1: There is no CLI. Period.

The clearest case, and the one Holmes's original manifesto explicitly concedes. design-inspector-mcp is a 59-tool surface for an internal Amazon design service. There is no di command on disk anywhere. Same for spec-studio-mcp, parts of builder-mcp (Amazon CRUX/Pipelines/Mechanic), nanobanana (image gen via Gemini Files API). For these, MCP isn't a transport choice — it's the only programmatic surface that exists. Holmes's directive — "if you're a company investing in an MCP server but don't have an official CLI, ship a good API, then ship a good CLI" — is correct as advice to the service owner. As advice to the agent author working on Tuesday with the surface that exists, it's useless. You wire up the MCP and budget the tokens.

Case 2: The auth flow would burn more tokens to re-derive than the schema costs to inject

This is the case the "just use curl" camp keeps missing. slack-mcp handles Slack Web API authentication, rate-limiting (Slack's tier-based 1-50 req/min ceilings), retry semantics, pagination cursors, and channel/user ID resolution. If you replaced it with curl + a Skill, every task that touched Slack would re-derive that machinery from prose, fail interestingly the first three times, and bleed tokens in error-recovery loops. The schema cost is paid once at session start; the auth cost would be paid per invocation. Same logic for aws-outlook-mcp (Microsoft Graph token refresh dance), amazon-sharepoint-mcp (sharepoint URL mangling), the corporate Builder Toolbox internals.

The principle: load-time tokens beat per-invocation tokens when the per-invocation cost is high and the call rate is high. If you only ever read Slack twice a week, sure, write a Skill. If your agent reads Slack on every other turn, the MCP wins on cumulative cost even at the 21K-token load-time price.

Case 3: The safety boundary lives server-side

This is the case that doesn't show up in benchmark posts because benchmarks don't measure agents misbehaving. Look at our mochi-slack skill:

## CRITICAL SAFETY RULE
Mochi can only READ Slack. All write tools are
disabled at the MCP config level.
You CANNOT send messages, post reactions, create
channels, or modify anything on Slack.

Why does that work? Because the MCP server's allowedTools whitelist physically does not include post_message, edit_message, add_reaction, etc. The agent cannot call them no matter what the prompt says, no matter how a hostile message tries to jailbreak it. The skill prose is documentation of a guarantee enforced one layer down. If I implemented the same restriction in a Skill that called the Slack CLI, the guarantee would be a polite request to a stochastic model.

Same shape: a read-only DB MCP that only loads SELECT-tool definitions. A filesystem MCP scoped to one directory. A production-API MCP that only exposes the GET endpoints. The MCP boundary is a capability boundary, not just a transport boundary, and capability boundaries are how you build agent systems that don't post in the wrong channel at 3 AM.

Case 4: Fleet auth uniformity

The least sexy case and probably the most consequential at scale. When N users need to use the same service, MCP centralizes credential management. Each user wires the MCP once; the agent never sees a credential. Without MCP, every user's Skill ends up with its own version of the auth dance, drifts independently, breaks on the same Tuesday in different ways. For a solo runtime — me, this laptop — Case 4 is rare. For a team rolling out an agent platform across 50 engineers, Case 4 is most of the value, and the load-time token cost is irrelevant compared to not having to onboard 50 individual auth setups.

Everywhere else, the Skill-with-CLI wins

For git, gh, aws, brazil, cr, jq, kubectl, anything with a man page the LLM has ingested — write a Skill that shells out. Three reasons, in order of importance:

  1. The model already knows the CLI. Claude has seen git rebase ten million times in training. The schema cost of an MCP git_rebase(branch, onto, options) tool is pure waste — the model is going to translate it back into git rebase in its head anyway. CLIs are LLM-native because they were authored for the same audience LLMs trained on: humans reading man pages.
  2. Errors are debuggable. When git push fails, you get a stderr message a human has read before. When an MCP git_push tool fails, you get a JSON-RPC error code wrapped in a server log on a different process. The "every MCP server adds a process layer" critique Holmes makes is real — and it bites hardest in failure mode, not in the happy path.
  3. Skills compose; MCPs don't. A Skill can git diff | jq | gh pr create in one shell line. The same flow across three MCPs is three sequential tool calls with three round-trips and three rounds of context updates. Pipes are an underrated tool for token efficiency.

The Skill files I trust most in our setup are the ones that read like a senior engineer's runbook: here's the command, here are the gotchas, here's the recovery path. The MCP servers I trust most are the ones that exist because no senior engineer's runbook could exist — because there's no CLI to write a runbook for.

The debate isn't two camps. It's three.

The "MCP is dead" framing collapses three distinct positions, which is why the developer reaction has been so fragmented:

CampPositionRepresentative
CLI maximalistsShip a CLI, agents handle the rest. MCP is architecturally wrong.Eric Holmes
Skill-pattern advocatesLoad tool definitions on demand via Skills + CLI; reserve MCP for what genuinely needs a server.Quandri, Anthropic Skills doc
Protocol fixersKeep MCP, fix the ergonomics. Deferred Loading proves this path works.Claude Code team

People who say "MCP is dead" online are usually arguing CLI > MCP for their specific service, not making a protocol claim. The protocol-fixer camp is the one shipping the boring, undramatic answer that's actually working: keep the protocol, defer the loading, whitelist the tools, and be honest about what each server is paying for in tokens.

The configuration discovery that came three years early

Buried in our agent config is a line we shipped months before this debate trended:

"includeMcpJson": false

That flag stops kiro-cli from auto-loading every MCP server in ~/.kiro/settings/mcp.json. Every server we want, we add explicitly to meshclaw.json's mcpServers + tools + allowedTools. We did this because the default ("load every MCP I have configured, expose every tool") made context bleed visible at our scale. The 21K-token measurement Quandri popularized is what happens when you don't do this. The single line of config above is, I'd argue, the actual fix to "MCP is dead" — not switching to CLIs, but explicitly whitelisting which servers and which tools you opt into, per agent, per use case.

And just to keep us honest: MCP isn't always smooth even inside the four cases. Last week I burned half an afternoon on design-inspector-mcp hitting OutdatedVersionException on parallel mutations because DI uses optimistic concurrency the protocol spec doesn't mention. Real fleets discover MCP failure modes that don't show up in marketing diagrams. "MCP is the right answer here" and "this server is operationally mature" are different sentences. The four-case framework tells you when to reach for MCP. It does not absolve you from auditing the specific server you reach for.

Five rules I'd staple to the wall

  1. If the service has a CLI you can apt install or brew install, write a Skill that calls it. Skip MCP. Your model already knows the CLI; the schema injection is pure tax.
  2. If the service is internal/proprietary and there is no CLI, MCP — but pin the version and budget the schema cost. Read the JSON definitions and decide if all tools earn their tokens, or if you should whitelist a subset.
  3. If the safety boundary is "agent can read but never write," enforce it at the MCP-config level, not in prompt prose. A whitelist of allowedTools is a real guarantee. A "please don't post" in the system prompt is a coin flip.
  4. Always set includeMcpJson: false (or your runtime's equivalent). The "load every MCP I have configured" default is the actual root of the 21K-token bleed. Whitelist explicitly. Per agent.
  5. For Claude Code users: confirm Tool Search with Deferred Loading is on. It's the architectural answer to the May 30th critique, and it's already shipping. The protocol fix is in the box.

What's actually dead

Here's what I think the next 18 months look like. The protocol isn't going anywhere — there are already too many proprietary services with no CLI for it to die, and Deferred Loading proves the runtime can fix the ergonomics without breaking the wire format. What dies is the "MCP supported" bullet point on every SaaS landing page since Q1 2026, regardless of whether the server is stable, scoped, or scoped well. The marketing-driven proliferation — the overprescription — is the part of MCP that's terminal. The protocol survives, in a smaller, more honest configuration.

"MCP is dead" is the new "Kubernetes is dead" is the new "microservices are dead." All three statements have been true in spirit and false in fact, simultaneously, for years. The honest version of all three is: this thing was applied too broadly, and the over-application is what's collapsing — not the underlying technology. Build for the four cases. Skip it for the other 80%. Whitelist explicitly. Read the update box.

And if the answer to your MCP-or-CLI question is genuinely ambiguous, just look at where the safety boundary lives. If it's in your prompt, you don't have a safety boundary. If it's in the server's allowedTools, you do. That's the cleanest test I know.
