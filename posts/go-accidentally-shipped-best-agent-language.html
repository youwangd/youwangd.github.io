Go Accidentally Shipped the Best Agent Coding Language — Here's Why

On Monday, Mitchell Hashimoto — the guy who built HashiCorp's entire product line and then spent three years evangelizing Zig — tweeted that he's writing Go again, and that go doc and gopls are "agent superpowers." 2,398 likes. The quiet part: he's not praising Go. He's describing a language whose 2012-era design decisions happen to be the exact interface an LLM agent needs. Nobody planned this. That's what makes it interesting.

The 2,398-Like Tweet

Here's the text, verbatim:

"I'm writing Go again... go doc and gopls are like agent superpowers and its shocking how productive agents are out of the box at writing good Go code versus other languages I've used (including the JS ecosystem). Also, Go + Zig is a good mix." — @mitchellh, 2026-04-21

Read that carefully. Hashimoto isn't saying Go is a good language — he already knew that. He's saying agents are shockingly productive at writing Go without any special setup. Out of the box. Compared to JS, which has an order of magnitude more training data, more libraries, more Stack Overflow answers. The surprise isn't Go's quality. It's the delta.

I went and looked at his go-libghostty repo to see what he's actually doing. There are two custom agent skills in .agents/skills/: discovering-upstream-apis and upstream-api-feedback. Both are built around querying Go's tooling. Not prompt engineering. Not RAG. Just teaching the agent which go subcommand answers which question.

What Agents Actually Do All Day

Forget benchmarks for a second. Watch an agent edit a real codebase. Its inner loop, every single turn, looks like this:

  1. Find the function I'm supposed to change.
  2. Read its signature and docstring — what does it promise?
  3. Find its callers — what will I break?
  4. Edit.
  5. Verify it still compiles.
  6. Run the relevant tests.

That's six queries. Each one has to resolve against the codebase in a way the agent can consume. The question isn't "does the language have great tooling" — every serious language has tooling. The question is: can a naive agent, with no setup, get the answer from one CLI command?

The 2012 Design Decisions Paying Off in 2026

Rob Pike and Ken Thompson weren't thinking about LLMs in 2012. They were thinking about Google-scale monorepos and new-hire onboarding. But the decisions they made — boring, almost religiously boring — turned out to be what a context-starved agent needs:

  • One way to format code. gofmt is mandatory and non-configurable. An agent never has to infer style. Style doesn't exist as a decision.
  • Docs are first-class. Godoc comments live above the function. go doc reads them. There's no "did the team use Sphinx or MkDocs or nothing?"
  • One build command. go build. Not make, not cmake, not ./gradlew --refresh-dependencies clean build -x test.
  • Flat, discoverable stdlib. 150 packages, each answering an obvious question. net/http, not @tanstack/router-react-async-experimental.
  • Errors as values. The agent sees the error path because it's in the return signature. No hidden control flow through try/except.

Every one of these was a trade-off against expressiveness. Every one of them makes the code boring to read and easy to machine-inspect. That was an aesthetic choice in 2012. It's an alignment choice in 2026.

The Python Counter-Argument (and Why It's Weaker Than It Looks)

The obvious objection: "But agents are trained on Python. More Python = more competent on Python." True, but this is a one-time advantage. Training data is a fixed cost. Tooling legibility is a per-query cost. Every time your agent re-enters the codebase, it pays again.

I live this at work. SageCLI is Python + shell. Our production monorepo has a Kotlin CDK package. When I ask an agent to modify a Python function, it reads 3–5 files hunting the call graph. When I ask it to modify the Kotlin CDK, gradle dependencies plus explicit imports let it work from one file. The Python agent burns 40K tokens exploring. The Kotlin agent burns 8K and ships.

Armin Ronacher — the guy who wrote Flask — made a similar argument on Pragmatic Engineer last October: the production AI stack is bifurcating. TypeScript and Rust/Go for the infrastructure (agents, runtimes, protocols), Python stays on the library bench. And look at what's shipping. Cloudflare's agent SDK? Go. Anthropic's mcp-go? Go. HashiCorp's entire line? Go. The MCP spec now has ten official SDKs — Go is tier one alongside Python and TypeScript.

DHH disagrees, loudly. His tweet from 04-20 (652 likes): "AI models are still not good enough to one-shot implementations you'd want to merge to master on large, critical applications." He's right — and also missing the point. Nobody's one-shotting. The argument is about the iterative inner loop, not the final artifact. Ruby's problem isn't that it's a bad language. It's that bundle and rails and rspec and rubocop and sorbet are five separate tools with five separate config surfaces. An agent has to learn each project's dialect before it can even start.

What to Build If Your Language Isn't Go

Most of us don't get to pick the language. I don't. SageCLI is Python because SageCLI has to run where the user's Python already runs. So what do you do?

Build the meta-command your language didn't ship with. Specifically:

  1. One project doctor command that aggregates format, lint, type-check, and test into a single agent-queryable output. Make it a Makefile target, a just recipe, whatever — just make it one command with stable output.
  2. One canonical dep-graph query. If your agent has to choose between pipdeptree, uv tree, poetry show --tree, you've already lost. Pick one, alias it, document it.
  3. AGENTS.md that points to these commands. Not a prose description of your codebase. A command reference the agent can grep.

I spent the last week doing exactly this on a Kotlin CDK package at work. Wrote AGENTS.md, ran it through eight rounds of AutoSDE static analysis, accumulated 40+ comments, fixed every one. The content of that document matters less than what the process did: it forced the codebase to become agent-queryable. Every "the agent can't figure this out" was a signal that a human couldn't either, given only the CLI.

That's the lesson hiding inside Hashimoto's tweet. Go didn't win because Rob Pike predicted LLMs. Go won because he hated ambiguity, and an LLM is the ultimate stress test for ambiguity. The language that makes the fewest choices for you — and exposes the choices it does make through a clean CLI — is the language your agent will feel "shockingly productive" in.

The rest of us just have to build the CLI ourselves.
