Trap-Positive Or Bust: Your Agent Eval's Oracle Is Probably Lying

Two days ago, dexhorthy — HumanLayer's founder — posted an overheard quote that should be the first slide in every agent-eval design review: "you can't build evals for something you don't understand." 35 likes, no thread, no follow-up. He's right, and the cheapest empirical proof you don't understand your own benchmark is this: your oracle silently passes code you know is broken. Not "might pass." Does pass. We just shipped a 20-task agent benchmark and found out three of our oracles were lying to us. The bug rate was 15%. If we hadn't checked, every PASS verdict on every agent run would have been faith-based.

"OH: 'you can't build evals for something you don't understand'"

The failure mode nobody is measuring

The 2026 ACM Computing Surveys paper on test-oracle automation has the cleanest definition I've seen: "A false negative is an incorrect and unexpected program state for which the oracle is true, i.e., a missed fault." Translate that to agent benchmarks: a false negative is the moment your scorer says PASS on output that is, in fact, wrong. The agent didn't earn the point. The harness gave it away.

This is not a hypothetical. arXiv 2601.05542 — "Understanding LLM-Driven Test Oracle Generation" — documents a specific recurring class: oracles that compare with un-overridden equals() or default reference equality, and silently accept wrong outputs because the comparison primitive itself is wrong. The oracle didn't check the property it was supposed to check. It checked some neighbor property that happened to be true on the broken code.

Every benchmark has this attack surface. Every one. SWE-bench, HumanEval, MBPP, your internal agent eval, the one you're about to write. The discourse focuses on what model did the agent run. Almost nobody talks about did the oracle even ask the right question. That gap is where credibility goes to die.

Trap-positive: the cheapest, most-skipped step in eval engineering

The fix is one paragraph long. Before you score a single real agent submission, take every task in your eval and feed its oracle a known-incorrect implementation. Require the oracle to return FAIL. If it doesn't, the oracle is broken, not the candidate. You haven't found a brilliant submission. You've found a hole.

Concretely, in bash, this is what one trap-positive check looks like for a coding task:

# Task: write `head -n 1` in a single shell pipeline.
# Trap-positive: a deliberately wrong implementation that prints ALL lines.
TRAP_IMPL='cat'   # prints everything, NOT just the first line

# Run the oracle against the trap.
verdict=$(./oracle.sh "$TRAP_IMPL" task.json)

# The oracle MUST say FAIL. If it says PASS, the oracle is broken.
if [ "$verdict" != "FAIL" ]; then
  echo "ORACLE BUG: trap-positive returned $verdict, expected FAIL"
  exit 1
fi

That's it. Six lines per task. Run it once before the corpus locks. You catch broken oracles in seconds, not weeks after the paper goes up.

Reverse this: if you cannot construct a trap-positive that the oracle correctly rejects, you do not have an oracle. You have a coin flip with extra steps. Every PASS verdict downstream inherits the same uncertainty as that coin flip — which is to say, none of them are evidence of anything.

Three oracles that lied to us

I'm going to walk through actual bugs from this workspace. We authored 20 trap-task oracles for an agent benchmark called goal-autopilot. We validated all 20. Three failed. Bug rate: 15%. Each one would have produced phantom PASS verdicts on broken submissions.

1. grep-head-1: the oracle was grading its own homework

Task: write a script that finds the first line containing a substring. The oracle ran the candidate, captured stdout, then grep'd for the expected line in the entire working directory. The trap implementation printed nothing at all. The oracle said PASS.

// Broken oracle (simplified)
candidate_output=$(bash "$candidate_script")
if grep -r "$expected_line" "$WORK_DIR"; then
  echo PASS    // matches the EXPECTED file in WORK_DIR
fi

// Why it fails: WORK_DIR contains the test fixtures, including
// task.json which embeds the expected_line as ground truth.
// The oracle was matching its own ground-truth file, not the
// candidate's output.

// Fixed oracle
candidate_output=$(bash "$candidate_script")
if printf '%s\n' "$candidate_output" | grep -qx "$expected_line"; then
  echo PASS
fi

This bug is so easy to write and so hard to spot by inspection. It type-checks. It looks reasonable. The CI pipeline would have lit up green forever. Every single PASS this oracle emitted was faith.

2. concurrent-counter: CPython's GIL hid the race the oracle was supposed to detect

Task: implement a thread-safe counter. The trap implementation was a deliberately broken counter using a non-atomic self.value += 1. The oracle ran 100 threads, each incrementing 1000 times, and asserted the final count was 100,000.

# Trap-positive submission (no lock, supposed to fail)
class BrokenCounter:
    def __init__(self):
        self.value = 0
    def inc(self):
        self.value += 1   # NOT thread-safe, the whole point of the trap

# Oracle assertion
assert counter.value == 100_000   # PASSES on CPython 3.11. Always.

Why? CPython's bytecode interpreter holds the GIL across the load-add-store sequence for plain integer increments. The race technically exists — the spec says it's UB — but in practice on CPython, with these exact operands, it never fires. The oracle could not detect the bug it was designed to detect, on the runtime the benchmark ran on.

Fix: switch the trap workload to operations that do release the GIL — list mutation under iteration, dict update during read — and assert on data-structure invariants the GIL doesn't protect. Different bug, same outcome (corruption), an oracle that actually fires.

The lesson: oracles are platform-coupled even when you don't realize they are. If your trap-positive runs on Python and your oracle's failure mode requires a primitive C-level race, you're testing CPython's GIL, not the candidate.

3. safe-eval-arith: $? captured the wrong process's exit code

Task: implement a sandboxed arithmetic evaluator that rejects shell injection. The oracle piped a malicious payload into the candidate, then checked $? to see if the candidate exited non-zero (signaling rejection).

# Broken oracle
echo "$malicious_payload" | bash "$candidate"
if [ $? -ne 0 ]; then echo PASS; fi

# In a pipeline, $? is the exit code of the LAST command,
# which is `bash "$candidate"`. Looks right? It is — IF you pipe.
# But our trap-positive submission used `read` to consume stdin
# and `read` returns 0 on success even when its body did the wrong
# thing. The candidate exited 0. The oracle said... PASS.

# We thought we were checking "did the sandbox reject the payload."
# We were actually checking "did `read` return 0," which is almost
# always true.

# Fixed oracle
output=$(echo "$malicious_payload" | bash "$candidate" 2>&1)
rc=$?
# Check BOTH that exit code is non-zero AND a rejection message
# appears, so a candidate that silently exits 0 cannot pass.
if [ "$rc" -ne 0 ] && printf '%s' "$output" | grep -q "rejected"; then
  echo PASS
fi

This one took 40 minutes to find. The fix took 90 seconds. That ratio — diagnosis vs patch — is normal for oracle bugs and exactly why people skip the validation step. It feels expensive. It is, until you ship the benchmark and someone with adversarial energy starts asking pointed questions.

The reviewer attack you should pre-empt

Imagine you ship a paper or a blog post or an internal launch deck claiming "agent X scores 73% on benchmark Y." Now imagine a sharp reader — a competing lab, a skeptical PM, a reviewer at NeurIPS — replies with one sentence: "Did you trap-positive your oracles?"

If the answer is "no" or "what's that," your number is a vibe, and now everyone reading the thread knows it. If the answer is "yes — out of 20 oracles we authored, validation caught 3 broken ones, here's the patch log," something else happens. You just demonstrated, with receipts, that your benchmark has higher epistemic standards than 95% of the leaderboards in this field. The number isn't a vibe. The number has a confidence interval that you actually checked.

The cline.bot crew put it well in their "Don't Build Slop" talk this week: agent maturity is now measured at the harness layer, not the model layer. Which model your agent runs is the cattle. Which oracle scores it is the pet. The model swaps weekly. The oracle, if it's broken, poisons every single result you've ever published — past, present, future.

One day of trap-positive engineering. That's the price. It buys back every percentage point of credibility your eval is currently borrowing on faith. Every team I've watched skip this step has eventually had to do a credibility apology — sometimes publicly. Every team that ran the trap-positive pass once, before the corpus locked, never looked back.

Build the oracle. Then break it on purpose. If it says PASS, throw it out and write a real one. That's the whole job.
