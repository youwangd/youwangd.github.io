Your AI Agent Is Burning Money on Tokens. Here's How to Fix It.

Token costs are the hidden tax on AI agents. Most of what your agent sends to the model is noise. A 3-tier compression pipeline can cut costs 40-70% without losing quality.

The Hidden Tax

Every AI team obsesses over prompt optimization. Count the tokens in your system message. Trim unnecessary words. Use shorter variable names. Meanwhile, your agent is burning through thousands of tokens you're not even tracking.

The real money drain isn't your carefully crafted prompts. It's the tool output.

When your agent runs a git diff, that's 2,000 tokens of context. When it searches the web, that's another 5,000 tokens of article text, metadata, and markup. When it reads a documentation page, that's 10,000 more tokens that get fed into every subsequent model call.

Here's the brutal math: most tool output gets processed by the model multiple times. Every follow-up question, every clarification, every decision carries forward all the previous tool results in the context window. That git diff doesn't cost you 2,000 tokens once — it costs you 2,000 tokens times the number of remaining turns in the conversation.

A single web search can inflate your session cost by 10x. Teams spend weeks optimizing prompts to save 200 tokens while ignoring tool output that balloons contexts to 100K+ tokens of mostly irrelevant noise.

Where Tokens Actually Go

Here's the breakdown nobody talks about for a typical long-running agent session:

  • Tool results: ~40% — Git diffs, web search results, file contents, API responses, command output
  • Conversation history: ~30% — Previous messages and responses that accumulate over time
  • System prompt + context: ~20% — Your carefully optimized instructions and workspace context
  • Current request + overhead: ~10% — The actual user message and JSON formatting

Tool results dominate. They're also the most compressible. Yet most teams focus entirely on that 20% system prompt optimization while ignoring the 70% elephant in the room.

The worst part? Most tool output is signal-to-noise disaster. Your agent runs npm test and gets 500 lines where only 3 matter: the tests that failed. It fetches a web page and gets 50KB of article text when the answer was in the first paragraph.

Tier 1: Compress at the Source

The highest-leverage optimization happens before tool results enter your context. Intercept the output and compress it intelligently at ingestion time.

Command Line Output: RTK

RTK is a Rust binary that specializes in compressing bash and CLI output. It understands command patterns and can reduce noise dramatically:

# Before RTK
$ git push origin feature-branch
Enumerating objects: 23, done.
Counting objects: 100% (23/23), done.
Delta compression using up to 8 threads
Compressing objects: 100% (12/12), done.
Writing objects: 100% (13/13), 1.23 KiB | 1.23 MiB/s, done.
Total 13 (delta 8), reused 0 (delta 0), pack-reused 0
remote: Resolving deltas: 100% (8/8), completed with 4 local objects.
remote: Create pull request for feature-branch:
remote:   https://github.com/company/repo/pull/new/feature-branch
To github.com:company/repo.git
   abc123f..def456g  feature-branch -> feature-branch

# After RTK
$ git push origin feature-branch
✓ Pushed feature-branch (13 objects, +5/-2 changes)
→ PR: https://github.com/company/repo/pull/new/feature-branch

That's 200+ tokens down to ~15 tokens. RTK handles grouping (multiple similar lines), deduplication (removes repeated warnings), and extraction (pulls out the essential result).

Web Content: Laconic

Laconic compresses markdown and HTML content with zero information loss:

Laconic strips badges, converts HTML tables to CSV format, removes redundant whitespace, and collapses repeated formatting. It typically achieves 40-68% compression on documentation and web content.

Structural Compression

For output that RTK and Laconic don't handle, implement structural compression rules:

# Before: Log output with repeated entries
2024-03-17 10:15:23 INFO Starting service
2024-03-17 10:15:23 INFO Loading config
2024-03-17 10:15:23 INFO Connecting to database
2024-03-17 10:15:23 ERROR Connection failed: timeout
2024-03-17 10:15:24 INFO Retrying connection
2024-03-17 10:15:24 ERROR Connection failed: timeout
2024-03-17 10:15:25 INFO Retrying connection
2024-03-17 10:15:25 ERROR Connection failed: timeout
2024-03-17 10:15:26 INFO Retrying connection
2024-03-17 10:15:26 ERROR Connection failed: timeout

# After: Collapsed with metadata
INFO Starting service, loading config
ERROR Connection failed: timeout [repeated 4 times over 3 seconds]
INFO Retrying connection [repeated 4 times]

Smart Truncation

When you must truncate, don't just take the first N tokens. Use head+tail preservation:

  • 70% from the beginning — Context and setup information
  • 20% from the end — Error messages and conclusions
  • 10% gap indicator — "[... 2,847 tokens truncated ...]"

Error messages appear at the end of output. Don't cut them off with naive head-only truncation.

Result Deduplication

If your agent calls the same tool with identical arguments, don't re-process the output:

# Instead of re-fetching
GET https://api.example.com/status
{
  "status": "operational",
  "services": [...] // 2000+ tokens
}

# Return a reference
Reference: Same as GET /status call at 14:23:15 (still cached)

Tier 2: Compress the History

As conversations grow, old tool results lose relevance. But you can't just delete them — the agent needs to understand what happened. Instead, implement graduated tombstones based on age.

The Decay Schedule

  • Age 0-2 turns: Keep full result
  • Age 3-5 turns: Replace with preview summary (first 200 chars)
  • Age 6+ turns: Minimal tombstone: "[tool was called: args → result type]"

This is the inverse of "Lost in the Middle" — instead of important information getting buried in long context, old information gracefully degrades while staying accessible.

# Turn 1 (fresh result)
exec "npm test"
FAIL src/components/Button.test.js
  ● Button component
    ✕ should render with correct text (24ms)
      Expected: "Click me"
      Received: "Clck me"
... [full 1,200 token output] ...

# Turn 4 (compressed)
exec "npm test" → FAIL Button.test.js: typo "Clck me" vs "Click me" [+1,150 tokens compressed]

# Turn 7 (tombstone)
exec "npm test" → test failures [result tombstoned]

Protocol Preservation

Critical insight: many AI APIs (like Bedrock Converse, Azure OpenAI) validate the toolResult structure. When you compress, maintain the JSON schema while reducing content:

{
  "toolResult": {
    "toolUseId": "toolu_123",
    "content": "[COMPRESSED] Git push succeeded → PR created [original: 1,847 tokens]",
    "isError": false
  }
}

Don't break the protocol. Compress within it.

Tier 3: Use the Right Model for the Job

Not every task requires your expensive frontier model. Most AI agents use one model for everything — burning premium tokens on background work that could run on cheaper alternatives.

The Cost Pyramid

  • Agent loop + tool dispatch + user responses: Sonnet/GPT-4 (full price) — needs reasoning and tool use
  • Memory extraction: Haiku/Flash (~1/12th cost) — pattern recognition, no tool calls
  • Context summarization: Haiku/Flash — text compression, deterministic task
  • Tombstone generation: Haiku/Flash — template filling

The "Tool Model" pattern from Alma.now separates the control plane (expensive model handles decisions) from the data plane (cheap model handles processing).

# Expensive model
Tool call: web_search("latest React hooks patterns")
Result: [15,000 tokens of search results]
Decision: "I need to extract the key patterns and summarize"

# Cheap model (background task)
Task: "Extract main React hooks patterns from this content"
Input: [15,000 tokens] 
Output: [500 token summary]

# Back to expensive model
Context: "Based on latest patterns: [500 tokens]"
Response: "Here are the modern React hooks approaches..."

Background Processing

Most compression work doesn't need to happen in the main conversation thread. Queue it:

  • Immediate: Show user the raw tool result
  • Background: Cheap model processes/compresses it
  • Future turns: Use compressed version in context

The user sees fast responses. You pay cheap model prices for the heavy lifting.

The Compound Effect

These three tiers don't just add up — they multiply. Here's a realistic before/after for a development session:

# Scenario: Agent debugging a failed deployment

Turn 1: git log --oneline -10
Raw output: 800 tokens
After Tier 1: 120 tokens (85% saved)

Turn 3: kubectl describe pod failing-pod
Raw output: 2,400 tokens  
After Tier 1: 900 tokens (63% saved)

Turn 5: Previous git log result (now age 4)
After Tier 2: 50 tokens (95% saved from original)

Turn 7: Background summary of kubectl output
Tier 3: Processed by Haiku instead of Sonnet
Cost per token: 12x cheaper

Combined result:
• Turn 1: 800 → 120 tokens (Tier 1: 85% saved)
• Turn 5: 120 → 50 tokens (Tier 2: 95% saved total) 
• Turn 7: Processed at 1/12th cost (Tier 3: 92% cost saved)

In a 20-turn session with heavy tool use:

  • Baseline: 50,000 tokens at full model price
  • After all tiers: 15,000 tokens, 60% processed by cheap models
  • Total cost reduction: 70-85% depending on model pricing

The savings compound because compressed results from Tier 1 get further compressed by Tier 2, and background processing from Tier 3 reduces the cost of that compression work.

What Doesn't Work

Be honest about the limitations. Not every compression technique works everywhere:

Proxy-Only Solutions Break

Tools like Kompact and Janus work by sitting between your app and OpenAI's API, compressing requests transparently. They only work if you use OpenAI directly.

If you're on Bedrock, Azure, Claude via Vertex, or any custom endpoint, these proxies are useless. The compression has to be built into your agent architecture, not bolted on outside.

RTK Only Helps Bash

RTK is fantastic for command-line tools, but it doesn't understand web search results, browser automation output, or MCP tool responses. You need compression strategies for each tool type.

Lossy Compression in the Wrong Place

Aggressive compression early in the pipeline can destroy information the agent needs later. Don't compress error messages, stack traces, or API responses that contain structured data the agent needs to parse.

Compress presentation, not data. Compress redundancy, not uniqueness.

Over-Aggressive Tombstoning

Turn every tool result into a one-line summary too quickly, and your agent loses the context it needs for follow-up questions. The decay schedule matters — too fast and you break reasoning, too slow and you waste tokens.

# Too aggressive (breaks reasoning)
Turn 2: exec "pytest tests/auth.py" → tests failed [tombstoned]
Turn 3: User asks "which specific assertions failed?"
Agent: "I don't have that information anymore"

# Proper decay
Turn 2: exec "pytest tests/auth.py" → [full 1,200 token output]
Turn 5: exec "pytest tests/auth.py" → auth tests failed: 3 assertion errors [+1,100 tokens compressed]
Turn 8: exec "pytest tests/auth.py" → test failures [result tombstoned]

Start Here

You don't need to build all three tiers at once. Start with Tier 1 — the highest impact, lowest risk changes:

  1. Audit your tool output. Log token counts by tool type. Find your biggest offenders.
  2. Add RTK for any bash/shell commands. It's a single binary, zero config.
  3. Add Laconic for web content. Works on markdown, HTML, and mixed formats.
  4. Implement head+tail truncation instead of head-only for large outputs.
  5. Add result deduplication for repeated tool calls.

That alone will cut 40-60% of your tool-related token costs. Tier 2 and 3 are advanced optimizations for when you're running heavy agent workloads.

The tools exist. The techniques work. The only question is whether you'll keep burning money on noise while your competitors compress their way to better margins.

Every token you don't send is a token you don't pay for. Start compressing.
