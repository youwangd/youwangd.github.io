The Agent SDK Was on Disk All Along

Dex Horthy threw out claude -p and shipped Shannon: a 196-star wrapper that launches an interactive claude inside tmux and tails the transcript JSONL on disk. No SDK. No bidi-stdio. No streaming buffer logic. Just three Unix tools from the 1990s. The internet handed him 688 hearts in 24 hours, and the comment under the tweet — "actually getting bidi stdio streaming is non-trivial" — is the whole reason this matters.

The tweet that broke the SDK myth

Saturday morning, the founder of HumanLayer — a company whose entire business is orchestrating coding agents — posted this:

"hey surprise — you can just launch interactive in tmux and then tail the jsonl. shipped a small wrapper... ralph loop iterating to full parity rn."

— @dexhorthy, May 14, 2026 · 688 ❤️ in 24h

The follow-up tweet, fewer hearts but more honest, is the part you should tattoo on your monitor:

"everything llm land is like that. context overflow signaling, retries, etc. it's very sad and very horrible."

— @dexhorthy, May 14, 2026

The man who builds agent infrastructure for a living publicly admitted that the official SDK is a tax, and the workaround is a Bash script. 196 stars, 11 forks, days old. This isn't a hot take. It's a confession from inside the foundry.

What Shannon actually does (and refuses to do)

Read the repo and the architecture is so anti-climactic it feels like a prank. Here is the entire control flow:

# What Shannon does, in actual order:
1. tmux new-session -d -s agent-7   # spawn detached tmux
2. tmux send-keys -t agent-7 "claude" Enter   # launch INTERACTIVE claude
3. tmux send-keys -t agent-7 "fix the bug in src/auth.go" Enter
4. tail -f ~/.claude/projects/<hash>/<session>.jsonl | jq .
   # ↑ that's it. that's the SDK.

What's missing from that flow is the entire point. There is:

  • No Anthropic API client. Shannon never makes an HTTPS call. claude does, on its own.
  • No claude -p headless mode. Dex specifically chose interactive. Headless mode is what the official SDK wraps.
  • No bidi-stdio protocol. No stream multiplexing, no half-buffered stdout reader, no "did the model finish or is it still thinking" heuristics.
  • No streaming framework. No SSE parser, no JSON chunk reassembler, no backpressure logic.

The CLI already did the work. Every turn, every tool call, every Read result, every error — Claude Code writes them to ~/.claude/projects/<hash>/<session-uuid>.jsonl as a side effect of running normally. Shannon is a 200-line script that says: "I'll just read what's already on disk."

If you've never opened one of those files, do it now. Pick any recent Claude Code session and run:

Every field your harness needs. Already there. On disk. No SDK call required.

Bidi-stdio is the wrong abstraction layer

The SDK pattern looks like this in your head:

┌─────────┐   stream   ┌──────────────┐   stream   ┌─────────┐
│  Your   │ ─────────→ │  Agent SDK   │ ─────────→ │  Model  │
│ harness │ ←───────── │ (bidi stdio) │ ←───────── │  API    │
└─────────┘            └──────────────┘            └─────────┘
                              ↑
                    You write retry logic,
                    backpressure, framing,
                    parsing, lifecycle here.

You spend a month building that middle box. You debug streaming edge cases. You write tests for partial JSON chunks. You handle the case where stderr arrives before stdout flushes. You learn what line-buffered vs block-buffered means at 2am.

Shannon's picture is different:

┌─────────┐  send-keys  ┌────────┐
│  Your   │ ──────────→ │  tmux  │
│ harness │             │  pane  │
└─────────┘             │   ↓    │
     ↑                  │ claude │ ──→ Anthropic
     │                  │   ↓    │
     │   tail -f        └───┬────┘
     └──────────────────── ─┘
                            ↓
                   ~/.claude/.../session.jsonl
                   (already structured, already complete)

Three boxes. Three Unix abstractions, none of them younger than 1991:

  • tmux gives you process lifecycle. Spawn, attach, detach, kill, survive disconnect. Solved problem.
  • tail -f gives you streaming. Append-only file + inotify = the cleanest event stream in computing. Solved problem.
  • jq gives you parsing. Structured JSON in, anything out. Solved problem.

The "agent SDK" is what happens when you forget your computer is a Unix machine. The OS is the SDK. The agent CLI is just a long-running process that writes a structured log. The "protocol" between your harness and the agent is a file path and an inotify watch.

I'll be honest: this is the bet I made on SageCLI from day one, and I caught a lot of "you should use the SDK" feedback for it. SageCLI's send / ls / clear commands are bash + tmux + JSONL, full stop. No framework. No SDK. No streaming library. The reason wasn't cleverness — it was laziness. The CLI already had everything I needed; building a second copy of it inside a Python wrapper felt obviously wrong. Shannon just validated the same instinct from a totally independent direction, with a much more public stage.

What this means for "agent SDKs" in 2026

Take any coding agent that ships today and grep its codebase for the word session. You'll find one of these:

~/.claude/projects/<hash>/<uuid>.jsonl       # Claude Code
~/.codex/sessions/<date>/<uuid>.jsonl        # OpenAI Codex CLI
~/.cursor/transcripts/<workspace>/*.jsonl    # Cursor
~/.aider/.aider.chat.history.md              # Aider (markdown, but parseable)
~/.opencode/sessions/*.jsonl                 # OpenCode
~/.kiro/.../sessions/*.json                  # Kiro

Every one of these is your agent's transcript, written as a normal byproduct of running. Every one is structured. Most are JSONL. None of them require an SDK to read. The protocol you've been told you need to learn — bidi-stdio framing, MCP transports, ACP message envelopes — is one or two layers above the layer where your transcript already exists for free.

This isn't a knock on ACP or MCP. They solve a real problem: standardizing the wire between an editor and an agent, so JetBrains doesn't have to write 22 integrations for 22 CLIs. That's the LSP layer. Worth the protocol overhead because the alternative is N×M.

But for your own harness — your cron jobs, your background workers, your CI bot, your custom orchestration loop — you are not N×M. You are 1×1. You don't need a protocol; you need a file path. The honest comparison:

My prediction for the next 12 months: every "agent SDK" with traction quietly becomes a thin courtesy wrapper over the on-disk transcript. The Anthropic SDK will stop being the recommended way to drive Claude Code. Vendors will publish transcript schemas instead of SDK clients. The smart wrappers — Shannon, SageCLI, the next dozen we haven't seen yet — will compete on what they do with the JSONL, not on how they got it.

The protocol you've been waiting for already exists. It's called the filesystem. It shipped in 1969. It has excellent uptime.

The takeaway, in one sentence

If your agent harness has a "client" module, delete it, point a tail -f at the JSONL the CLI is already writing, and ship the week you would have spent debugging stdio framing as something a user can actually use.
