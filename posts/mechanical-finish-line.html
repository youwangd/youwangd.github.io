2026-03-22 agents architecture insight

The Only Thing That Matters Is a Mechanical Finish Line

AI agents can plan, retry, and iterate forever. The bottleneck was never intelligence. It's knowing when you're done.

The Pattern Hiding in Plain Sight

In March 2026, Andrej Karpathy released autoresearch — a repo that lets an AI agent autonomously run LLM training experiments overnight. The agent modifies the training code, runs a 5-minute experiment, checks whether the validation loss improved, keeps or reverts, and repeats. You go to sleep, wake up to 100 experiments and a better model.

It got 25,000 stars in five days. Three files. One metric.

The brilliance isn't the agent. It's the metric. val_bpb is a number. Lower is better. No ambiguity. No judgment call. A bash script can check it. The agent never needs to ask "am I done?" — the number tells it.

Now look at where AI has produced superhuman results:

  • AlphaGo / AlphaZero — Win or lose. Binary. Millions of self-play games with mechanical feedback.
  • LLM training — Loss goes down. Benchmarks go up. Training runs for weeks with no human in the loop.
  • Compiler optimization — The output is faster or it isn't. Measure, compare, keep or discard.
  • Protein folding (AlphaFold) — Structure matches experimental data or it doesn't. Angstrom-level metrics.
  • Code generation benchmarks — SWE-bench, HumanEval. Tests pass or they don't.

Every breakthrough has the same structure: a tight loop with a mechanical validator. The agent does work, a deterministic check says pass or fail, and the agent adjusts. No human in the loop. No subjective judgment. Just a number or a boolean.

The Problem with Most Agent Work

Now look at how most people use AI agents for software engineering:

Human: "Fix the login bug"
Agent: *works for 10 minutes* → "Done!"
Human: *checks* → "No it's not. Tests still fail."
Agent: *works again* → "Fixed now!"
Human: *checks* → "Different error this time."
... 20 iterations later ...
Human: "OK finally."

The human is the validator. That's the bottleneck.

Not because the agent is dumb — modern coding agents are remarkably capable. The problem is that the agent has no way to know it's actually done. It runs the code, it looks right to the agent, it says "done." But "looks right" isn't mechanical. It's vibes.

Autoresearch doesn't have this problem because val_bpb doesn't care about vibes. The number goes down or it doesn't. That's the whole feedback loop.

The Insight: Mechanical Validation Is the Unlock

Here's the principle: anything with a mechanical finish line can be optimized by endless turns of AI iteration.

If you can write a function that returns true or false for "is this done?", you can put an AI agent in a loop and walk away. The agent will iterate until it converges or exhausts its retry budget. No human needed.

If you can't write that function — if "done" requires human judgment — then the agent is bounded by how often a human can check its work. The loop speed is limited by human attention, not compute.

This is the dividing line between tasks AI can fully automate and tasks that still need human involvement:

Mechanical Finish LineHuman Judgment Required
Tests pass"The UI feels right"
Benchmark score improves"The copy reads naturally"
Compilation succeeds"The architecture is clean"
API returns expected response"The error message is helpful"
Linter passes"The variable names are intuitive"
Performance within budget"This is the right abstraction"

The left column is where AI agents will eat the world. The right column is where humans stay relevant. The engineering challenge is moving as many things as possible from right to left — turning subjective criteria into measurable checks.

What This Means for Agent Architecture

If mechanical validation is the unlock, then agent systems should be designed around it. The architecture becomes:

1. Human describes goal in natural language
2. System generates mechanical validation checks
3. Human approves the checks (one-time)
4. Worker agent iterates autonomously
5. Mechanical validator checks after each attempt
6. Loop until pass or escalate

The key decisions in this architecture:

The validator must be external to the worker. If the agent validates its own work, it's grading its own homework. A separate process — ideally a deterministic script, not another LLM — provides honest feedback. Autoresearch gets this right: the training script produces a number, the agent reads the number. The agent doesn't get to decide whether the number is good enough.

State lives on disk, not in context. Autoresearch uses results.tsv as the agent's memory and git branches as state management. The agent's context window stays clean because it reconstructs state from files each iteration. For long-running tasks (20+ iterations), this prevents context window exhaustion.

Fresh context per iteration. Each retry cycle should spawn a fresh agent session. Feed it a summary of previous attempts and the latest failure — but don't carry forward 100K tokens of stale debugging history. The code on disk is the source of truth, not the conversation history.

Circle detection matters. If the same failure appears three times in a row, the agent is stuck. Escalate to a human rather than burning through 20 identical retries. The agent can't always break out of local minima on its own.

Practical Example: SageCLI Goal Loop

SageCLI implements this pattern with a --goal flag on its task command:

sage task debug app.py tests.py \
  --goal "All pytest tests pass" \
  --retries 5

What happens under the hood:

  1. An LLM translates the natural language goal into mechanical checks: pytest test_app.py exits 0.
  2. The user approves the checks.
  3. A worker agent (Claude Code via ACP) gets the task and fixes the code.
  4. Sage (a bash script — not an LLM) runs the mechanical checks.
  5. If checks fail, the failure output is fed back to a fresh worker session.
  6. Loop until pass, circle detection triggers, or retries exhaust.

Three things are deliberately mechanical:

  • The validation — bash runs the test command and checks the exit code. No LLM involved.
  • The logging — every cycle is appended to run.md by bash, not by the agent. Deterministic, not prompted.
  • The decision logic — pass/fail/retry/escalate is a bash if-statement, not an LLM judgment call.

The only LLM calls are: (1) the worker doing the actual coding work, and (2) the initial check generation from the natural language goal. Everything else is deterministic.

The Frontier: Making More Things Mechanical

The most impactful engineering work in the AI agent space right now isn't making agents smarter. It's making validation more mechanical.

Visual regression testing turns "the UI looks right" into pixel-diff comparison. Performance budgets turn "the page is fast enough" into a Lighthouse score threshold. Accessibility audits turn "the app is usable" into axe-core violations. Type checkers turn "the code is correct" into compiler output.

Every time you convert a subjective check into a mechanical one, you unlock another domain for autonomous AI iteration. The agent doesn't need to get better — it just needs a clearer finish line.

Autoresearch's val_bpb. AlphaGo's win/loss. Your test suite's exit code. They're all the same thing: a mechanical signal that turns infinite AI iteration from a spinning loop into a convergent search.

The question for any task isn't "is the AI smart enough?" It's "can you define done?"
