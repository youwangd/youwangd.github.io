The Backend Owns the Repo: Gemini CLI's 104K-Star Postmortem

This Wednesday, June 18, Google turns off Gemini CLI for every Pro, Ultra, and free Code Assist user on the planet. The repo has 104K stars. Apache 2.0. Donated to the Linux Foundation before the sunset email went out. Six thousand external PRs from real volunteers, including one contributor with 27 merged commits who described the after-feeling on Hacker News in one sentence: "essentially working for free on a code base that will only be used in enterprises." The license is fine. The repo is still up. The runtime is gone. If you read only one paragraph of this post, read this one: a permissive license on a backend-dependent client is a managed dependency in a costume.

What actually happens Wednesday

I want to start with the deadline, because most takes I've read this week skipped past it to argue about license philosophy. The deadline is the whole point. From Google's May 19 announcement and the Digital Applied migration guide (Jun 9), here is the matrix that lands on engineering teams Wednesday morning:

Tier Status after Jun 18 Migration path
Free Code Assist Cut off Install agy binary. New auth flow. Weekly compute cap.
Code Assist Pro Cut off Same. Plus a 7-day cooldown if you blow the cap.
Code Assist Ultra Cut off Same.
Code Assist Enterprise (paid) Exempt for now Old gemini binary keeps working until further notice.

That last row is doing a lot of structural work. The reason the migration matters is not "Google killed an open-source CLI." Open-source CLIs get killed all the time and nobody writes a post. The reason it matters is that the only people who keep the open-source one running are paying. The 104K stargazers were funding the enterprise sales motion with their attention and their PRs. The repo is staying open exactly long enough for "we're committed to open source" to be technically true, while every actual user of the open thing is force-migrated to the closed thing.

The four red rows in the migration matrix nobody is screenshotting, but should:

  1. MCP config silently breaks. The remote MCP field renamed from url to serverUrl. Old config loads with no error. The first tool invocation a few hours into a session is where you find out. If you have CI agents reading ~/.gemini/mcp_config.json, they pass startup and fail on first action. That is the worst kind of failure mode for an autonomous loop — far enough into the run that the symptom is decoupled from the cause.
  2. ACP stdio mode is not in the launch. If you orchestrate Antigravity from another agent (the entire point of ACP — Agent Client Protocol), there is currently no stdio mode in the new binary, only the interactive TUI. Every orchestrator that bridged Gemini via ACP stdio breaks on day one and waits for an issue thread to resolve.
  3. Quota model regresses. The free tier's "1,000 requests per day" becomes a weekly compute cap. A single chatty session can spend a week's allowance in a couple of hours. Heavy users have already reported up to 168-hour cooldowns. If you hand-rolled a "burst now, cool off tomorrow" mental model, that model is wrong as of Wednesday.
  4. CI binary swap. Anywhere that pinned gemini in .github/workflows/ or a Brazil cron/ needs agy. The arguments aren't quite the same. The behaviour around stdin isn't quite the same. The default model isn't the same.

None of these are unfixable. All four are silent enough to ruin a Wednesday standup if you walk in cold.

The Apache 2.0 mirage

I'll quote the most-cited line from the announcement, because the reason it confused people is that, taken at face value, it's not wrong:

"The Gemini CLI repository will remain open source. We're committed to the open-source community."

Yes. The repo is up. Apache 2.0 is still in LICENSE. The Linux Foundation donation that was announced before the sunset news still went through. You can fork it tomorrow. None of that helps you, because what shipped under Apache 2.0 was the client: the TUI, the prompt rendering, the command parser, the ACP bridge, the MCP plumbing, the auth flow. The model is not in the repo. The agent loop's reasoning happens server-side. The retrieval-augmented stack lives in Google's tenancy. The auth endpoint that any forked binary needs to talk to is a closed Google service that you do not control and cannot stand up.

So what is the actual portability of the donated artefact? Look at the Antigravity CLI public repo as the brutal control case. As of writing, it contains:

  • A README pointing to the closed-source agy binary download.
  • A changelog.
  • An animated demo GIF.

That's it. Zero application code. The "open repo" is a project page. The successor is being launched from day one with the costume off, which at least has the dignity of being honest. And by analogy backwards, the predecessor's open repo is mostly a costume too — open code, closed runtime, and the runtime is where the value lives.

This is not a Google-specific failure mode. This is the structural shape of every backend-dependent CLI that ever shipped under a permissive license. Pick any of them. Strip the model, strip the auth endpoint, strip the proprietary RAG store. What's left is a TUI. A useful TUI. A TUI you can fork, hand-port to a different model, and run against a different auth backend you build yourself. Almost nobody does, because almost nobody has the runtime budget to. The value of the artefact and the cost of replacing the backend are very far apart, and the gap between them is the lock-in nobody wrote into the LICENSE file.

The grep test before you call something portable

Here is the dependency-posture diagnostic I run on any new agent CLI before adopting it in our loop. The whole thing is three commands. Run them before you commit, not after the migration email.

# 1. How much of the runtime is the binary, vs. how much lives behind a closed endpoint?
git clone <repo> /tmp/audit && cd /tmp/audit
grep -rEn 'https://[a-z0-9.-]+\.(google|googleapis|openai|anthropic)\.com' src/ | wc -l

# 2. Does the auth flow work without a vendor account?
grep -rEn 'oauth|client_id|api.key' src/ docs/

# 3. Can the binary speak any non-vendor model out of the box?
grep -rEn 'provider|model_family|backend' src/ config/

If command #1 returns dozens of hits and command #3 returns one provider name, you are looking at a managed dependency in a costume, regardless of the LICENSE file. Apache 2.0 is necessary for portability. It is nowhere near sufficient.

The Wednesday-deadline grep is the tactical version of the same idea. Three commands every team should run before the cutoff:

# a. Where is the binary pinned in CI/cron?
grep -rn 'gemini' .github/ ci/ cron/ scripts/

# b. Does any MCP config still use the old field name?
grep -rn '"url"' ~/.gemini/ mcp_config.json .config/

# c. Anything routing through ACP stdio against gemini?
grep -rn '\-\-acp' . | grep -v node_modules

If any of those return hits, you have work to do this week. Which is fine. The cost of finding out on Tuesday is small. The cost of finding out at 9:14am Wednesday is a sev.

What an indie CLI does differently

I've been building SageCLI on the opposite end of this spectrum for the last quarter, so I'll be specific about the trade. The structural difference is not the license. It's the fact that the runtime contract is owned by the user:

Posture Big-co freemium CLI Indie CLI (SageCLI shape)
Model One vendor's frontier model, hosted Any model that speaks OpenAI/Anthropic/Bedrock — local Ollama works
Auth Closed OAuth to vendor account User's own keys in ~/.aws/credentials or OPENAI_API_KEY
Tool surface Vendor-curated MCP catalog Any MCP server you can run a process for
Sunset blast radius Every user, same Wednesday One user, the day they stop running the binary themselves
What you give up Nothing — until Wednesday You provision your own model, your own auth, your own MCP servers

The honest part of that table is the last row. There is a cost to indie. You do your own provisioning. You pick your own model. You pay your own usage. You don't get the Pro tier's frontier model for free. In return, nobody can sunset you. The two columns aren't "good" and "bad"; they're "managed convenience with a cliff" and "owned plumbing without one." Pick one knowingly.

The version of this trade I keep coming back to: I'd rather pay $40/month for tokens I route myself than pay nothing for a CLI that decides on a Monday in May that my Wednesday in June starts at zero.

The contributor question

I want to end on the part of the Hacker News thread that stuck with me, because it isn't a technical point and it doesn't have a fix. Andrea Alberti, a contributor with 27 merged commits, posted under the announcement thread and asked: "Essentially working for free on a code base that will only be used in enterprises?"

That is the line that should change how you read the next "open source by <big company>" announcement. Six thousand external PRs of free labour, merged into a repo whose closed-source backend is the only thing that made the client interesting, and on Wednesday the client tier with no backend access is empty space and the paid enterprise tier is the only living user. The repo is still open. The labour was real. The use of that labour is now a tier-restricted enterprise-paid product. Apache 2.0 made that legal. Nobody is alleging it's illegal. The structural question is whether contributors knew the structural shape of the deal when they sent the PR.

The answer most of them would give, if asked honestly, is no — because the structural shape of the deal isn't visible from inside the repo. It's visible only from the dependency-posture grep above, run before you click "Fork." That is the small habit I'm trying to build into our team's adoption checklist this quarter. Two minutes of grep before two months of contribution.

The repo isn't the artefact. The runtime is the artefact. Whoever owns the runtime owns the project, regardless of what LICENSE says. On Wednesday, 104K stargazers find out who owned theirs.
