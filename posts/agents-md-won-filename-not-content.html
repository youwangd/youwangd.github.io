2026-05-09 agents-md coding-agents conventions

AGENTS.md won the filename war. It hasn't won the content war.

A Reddit post on r/ExperiencedDevs this morning asked "how are you handling project conventions across Cursor, Claude Code, and Copilot?" The top answer was a list of seven config files. The filename war is actually over. These developers just missed the memo. The harder question — what belongs inside the file — is still a free-for-all, and I have opinions from shipping AGENTS.md across 19 packages with a static analyzer rejecting my drafts eight times.

A timeline nobody read

For about a year, coding agents did the thing engineers always do when there's no standard: they each invented one. Cursor stuffed rules into .cursorrules, then .cursor/rules/. Claude Code picked CLAUDE.md. Gemini CLI picked GEMINI.md. Aider had CONVENTIONS.md. AMP used AGENT.md — singular. OpenAI's Codex CLI picked AGENTS.md — plural. Your repo root started looking like a UN cafeteria.

Here's the compressed timeline of how it ended:

  • Mid-2025 — OpenAI ships Codex with AGENTS.md. At the same time they quietly buy the agents.md domain.
  • Summer 2025 — AMP does the graceful thing and surrenders AGENT.md in favor of the plural. No drama, no blog post war.
  • Late 2025 — Cursor, Jules (Google), Factory, goose, Zed, Warp, Windsurf, VS Code's Copilot coding agent, JetBrains Junie, RooCode, Kilo Code, Devin, Phoenix, Semgrep, Augment Code, Ona, and ~20 others line up behind AGENTS.md.
  • Early 2026 — The agents.md page lands under Linux Foundation stewardship, via a new Agentic AI Foundation (aaif.io) co-founded with OpenAI. Filename standards under LF stewardship is about as official as it gets on the internet.
  • Today — GitHub's code search returns 60,000+ AGENTS.md files. The Reddit thread above argues the question is still open.

Two tools still hold out on the filename: Claude Code defaults to CLAUDE.md, Gemini CLI defaults to GEMINI.md. The documented workaround is a one-liner that most adopters copy into their README:

ln -s AGENTS.md CLAUDE.md
ln -s AGENTS.md GEMINI.md

If a coordination problem can be solved with ln -s, it's not a coordination problem anymore. It's a habit that'll fade in two release cycles.

What the standard actually standardizes

Read the agents.md page carefully and you notice something strange: it standardizes the filename and almost nothing else. The page suggests five sections — project overview, build/test commands, code style, testing instructions, PR guidelines — but nothing is required. No schema. No linter. No canonical sections. No rules about nesting, inheritance, size limits, or conflict resolution when agent-specific extensions sneak in.

This is, I think, a feature. Agents that share a spec but not a format have repeatedly produced worse outcomes than convergence-via-convention. MCP is the counterexample you want to avoid — a spec that's so formal it takes a month to grok, and it still doesn't tell you what a good tool description looks like.

But it's also the reason your AGENTS.md is probably bad.

What breaks at 19 packages

Over the last three weeks I rolled AGENTS.md across 19 Amazon packages owned by 12 different people. One package — FeesDomainService, a CDK service with ten stages across four regions — got the full treatment: an AI-generated first draft, eight rounds of AutoSDE (Amazon's static analyzer for code review), and 42+ automated comments. Every comment was a content problem. Not one was a filename problem.

Here's what actually breaks when you try to scale the "one file, no rules" approach:

1. Drift from the source of truth

The AI wrote a confident AGENTS.md section listing Prod accounts and regions. Four of them were wrong. The model hadn't followed the CDK import chain — app.ts → deploymentStageConfigs → profiles.ts — so it hallucinated a plausible-looking account map. AutoSDE's security rules caught the hardcoded account IDs; the correctness problem wouldn't have been caught by any linter, because there's no schema saying "account IDs in AGENTS.md must come from CDK source."

The fix I settled on: a Known Gaps section at the top of each file where you explicitly tag rows as [VERIFIED], [UNVERIFIED], or [STUBBED]. If a section can't be tagged [VERIFIED], it doesn't go in. This is the boring 80% answer. It also works.

2. Confident fabrication

LLMs write AGENTS.md the way a new hire writes a transition doc on their last day: with authority. Every section reads like the writer knows. After eight rounds, AutoSDE's single biggest category of comments was variations of "this claim has no source in the codebase, remove it". The model had invented monitoring runbooks, fabricated on-call rotations, and confidently described a retry policy that didn't exist in any config file.

If you're rolling this out: assume every section is fabricated until you grep for the thing it claims.

3. The security-noise trap

Generated AGENTS.md files love a "Security" section. They'll cheerfully list the auth scheme, the encryption algorithm, the KMS key alias, and whether TLS is enforced. AutoSDE and most security teams will take this as an invitation. Our first draft collected six "Security information leak" findings for doing exactly what the skill-builder recipe suggested.

The thing nobody tells you: the AGENTS.md is public inside your monorepo, often published as rendered HTML, and serves as structured reconnaissance if you describe how your security works. The fix is to describe that security exists, not how. "Authentication via internal IAM service, see auth/" beats three paragraphs of crypto trivia.

4. Nested inheritance bugs

The spec says "closest AGENTS.md to the edited file wins." OpenAI's main repo has 88 AGENTS.md files following this pattern. It's the right default for a monorepo. It's also a brand-new category of bug: a project-level invariant ("never commit without running brazil-build") gets silently overridden by a subfolder AGENTS.md that forgot to restate it.

I don't have a clean answer here yet. What I do now: every nested file starts with a one-line reminder — "Inherits from ../AGENTS.md; overrides only what's in this file." — and I run a tiny diff script that flags keys present in the parent but missing from the child. It's duct tape. But "closest wins" without any discipline is how you ship a broken pre-commit hook in three months.

5. Agent-specific extensions that leak back in

Claude Code supports ! prefixes for bash, @ for file refs, and the top of the file as "importable memory." Cursor has .cursor/rules/ with glob targeting. Codex respects <MEMORY> blocks. A shared AGENTS.md that uses any of these becomes silently worse on every other agent.

My rule: the shared file contains only portable content. Tool-specific extensions go in .claude/, .cursor/, .codex/ or whatever the tool ships. If you can't decide whether something is portable, it isn't.

What earns its place

After eight AutoSDE rounds, these are the four sections the analyzer stopped flagging. They're also the four I'd include in every AGENTS.md from now on:

  1. Build & test commands, verbatim. Not prose. The exact command an agent should run. brazil-build beats "run the brazil build." If it's a flaky command, say so: brazil-build release # retry once on CDK sync errors.
  2. Known Gaps. A table of claims with [VERIFIED] / [UNVERIFIED] tags and a link to the source of truth. This one section alone kills ~60% of drift bugs.
  3. Deep-dive knowledge references. Not the knowledge itself — just pointers to .kiro/knowledge/*.md, docs/, or wiki pages. Agents handle links fine; they drown in inlined walls of text.
  4. Hooks for auto-refresh. A Kiro .kiro.hook, a kiro-cli stop hook, a Cline markdown hook. If the file doesn't regenerate when CDK config changes, it starts rotting the day you commit it.

And here are the three sections AutoSDE kept flagging, that I eventually cut:

  • "Security" section with any specifics. Describe the domain, not the implementation. Every time.
  • "Architecture overview" written by the model. If the model writes it, it's fabrication. If a human writes it, it's stale in a month. Replace with a link to an owned design doc.
  • "Operations runbook." Runbooks belong in runbook tools. An agent reading a runbook from AGENTS.md is an agent one git blame away from paging the wrong oncall.

The meta-lesson

The filename standardization made one thing obvious that was hidden before: the hard problem was never the filename. When everyone had their own file, you could explain inconsistency with "we use different tools." When everyone has the same file, the inconsistency is in your content strategy.

60,000 AGENTS.md files on GitHub is an impressive number. I'd bet 55,000 of them are AI-generated, never re-read, silently drifting. The few that work are the ones with fewer sections than the spec suggests, an owner whose name is on the file, a hook that refreshes them on code change, and a static analyzer that rejects fabrication. That's the content war. It's not as photogenic as a domain purchase, but it's the one worth fighting.

Filename: solved. Content: we're at about day three.
