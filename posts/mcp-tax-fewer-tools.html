The MCP Tax: Why Loading Every Server Makes Your Agent Worse

A 200K-token context window sounds enormous until you remember the agent has to spend it on something. Apideck instrumented their production stack in April: three MCP servers — GitHub, Slack, Sentry — totaling about 40 tools. Before the human typed a single character, those tool definitions had already consumed 143,000 of the 200,000 available tokens. 72% of the budget, gone, on a menu the model didn't even order from yet. This is the bill nobody on the "awesome MCP servers" trending list is paying attention to. It is also why every serious agent shop in 2026 is quietly cutting tools, not adding them.

The cold ledger

Numbers don't argue. Here is what gets billed against your context window before the first user message arrives, sourced from real production telemetry and from the official MCP server inventories themselves:

Source Configuration Tokens consumed
Apideck production telemetry 3 servers · ~40 tools (GitHub + Slack + Sentry) ~143,000 / 200K window (72%)
Atlassian mcp-compressor announcement Official GitHub MCP server inventory ~17,600 for 94 tools
Atlassian, same post Same 94 tools, behind mcp-compressor proxy ~500 (97% reduction)
Anthropic Tool Search (Feb 2026) Lazy-load tool definitions on-demand ~85% reduction vs eager loading

Two facts you cannot escape after staring at this table for thirty seconds:

  • The token cost is not bounded by how many tools you actually call — it scales with how many you list.
  • Every major vendor that runs MCP at scale is now shipping a layer that hides the inventory from the model. That is not a feature; it is damage control.

When Atlassian publicly reduces a 17,600-token surface to 500 tokens and frames it as a preventative against tool bloat, they are admitting that the default MCP configuration breaks Rovo Dev's prompt budget. When Perplexity's CTO Denis Yarats walked away from MCP for internal production in March citing "context window consumption" as the operational blocker, that is not a hot take. That is a procurement signal.

Three hidden costs your tool-calling success rate will not show you

1. Context bloat — the tax you pay before you do any work

The first cost is straightforward and the easiest to measure. Every tool definition is JSON Schema describing parameters, types, descriptions, defaults, error semantics. The model has to read all of it on every turn. A 17,600-token tool inventory means you have 17,600 fewer tokens of headroom for everything else: system prompt, retrieved documents, conversation history, the user's actual problem. On a long agentic session that compounds across many turns, the inventory is paid for over and over again as part of the context that the next call sees.

The painful version of this is not the synthetic benchmark — it is the long-running session. We have a wiki-curator that runs on a persistent kiro session ingesting one article at a time. Per-article time started at 10–15 minutes. After about two dozen articles it crept to 60–100 minutes. None of the prompts had changed. The model didn't get worse. The context just kept accumulating, the inventory kept getting re-read, and the per-tick cost grew from "fast enough" to "drop dead unusable." The fix was nothing clever: restart the session every N articles. Drain the bucket. The bucket fills back up.

2. Tool-selection drift — bigger menus produce worse choices

The second cost is the one you can't see in your billing dashboard, only in your traces. The arXiv paper "MCP Tool Descriptions Are Smelly" (Feb 2026) measured exactly the failure I keep seeing in the wild: as the toolset grows, tool-selection accuracy drops. The model can't crisply distinguish between forty plausible-sounding options when half their descriptions overlap. get_user vs fetch_user_profile vs read_member. Three different MCP servers, three different teams, three near-identical signatures. The model picks one. It's often the wrong one.

This is the inverse of the standard pitch. The pitch is: more capabilities mean a more capable agent. The data is: more capabilities mean a more confused agent, and the marginal capability you just added is sometimes negative — it crowds out the one you actually needed. Bigger isn't better. It is often worse, even if you ignore tokens entirely.

3. Cold-start races — the tools that arrive too late to register

The third cost is the most invisible, because it doesn't show up as a degraded response. It shows up as tools that vanish entirely. This is the one we burned half a Tuesday afternoon discovering, and I haven't seen anyone else write it up.

Most agent runtimes seal their tool registry at the start of a session. After that seal, any MCP server that arrives late is alive — visible in /mcp, accepting JSON-RPC probes, returning a healthy tool list — but the agent itself has no idea those tools exist. They were not in the registry when the registry was sealed. The race is silent. There is no error. There is just a server that does nothing for the rest of the session.

Post-mortem: the 1-second tool registry race

We had configured scholar-feed-mcp in our agent specs the recommended way:

{
  "mcpServers": {
    "scholar-feed": {
      "command": "npx",
      "args": ["-y", "scholar-feed-mcp@3.7.0"]
    }
  }
}

This is what every MCP README in the world tells you to write. npx -y looks ergonomic — pin the version, fetch on demand, no global install pollution. We started a session. /mcp showed scholar-feed green. Connected. Healthy. The agent saw zero tools from it. None of the 25 advertised methods were callable. No error in the agent log. No error in the MCP transport.

We hit it directly with a JSON-RPC probe and the server happily returned all 25 tools. The server was fine. The agent was fine. They were just not wired together. Took an embarrassing amount of grepping to find the seam.

# Timeline of one session start, measured.
T+0.000s   kiro-cli boots, agent process forks
T+0.420s   MCP transport opens to scholar-feed (npx invoked)
T+1.000s   ### tool registry SEALED — agent locks its tool list ###
T+1.000s   agent sees: 0 scholar-feed tools registered
T+3.150s   npx finishes warming up, scholar-feed-mcp ready
T+3.151s   server announces 25 tools — TOO LATE

The fix was a one-character change in spirit and a config rewrite in practice: stop using npx as a launcher. Install the package globally — npm i -g scholar-feed-mcp@3.7.0 — and point the spec at the absolute binary path with empty args:

{
  "mcpServers": {
    "scholar-feed": {
      "command": "/home/dyouwang/.nvm/versions/node/v22.17.1/bin/scholar-feed-mcp",
      "args": []
    }
  }
}

Cold-start time dropped from ~3.0s to 0.37s. The server announces its tools well before the registry seal at T+1.0s. The agent sees all 25 tools. Same code, same package version, same model. The only thing that changed was that the launcher stopped racing the registry.

This is the kind of pathology that gets worse with more MCP servers, not better. Each npx-style server adds another roll of the dice on whether it warms up before the seal. The more you stack, the more you lose silently. Tools you don't see are still on your bill — you paid for the listing tokens with no return.

The diet

Treat MCP servers like dependencies. The default answer to "should I add this server" is no. Make people justify each addition the way they would a new third-party SDK in a production codebase. Concrete rules that have stopped hurting us:

  • Decompose by domain, not by demo. Pinterest's pattern: a single agent gets a data server or a comms server, never both stacked together with everything in scope. If a workflow needs both, build a thin orchestrator that spawns two scoped agents. You'll trade a tiny coordination cost for a massive context-window dividend.
  • Put a compressor in front of any large server. Atlassian's mcp-compressor, Anthropic's Tool Search, or roll your own: replace the eager 94-tool inventory with two meta-tools — list_tools(filter) and invoke_tool(name, args). The model finds what it needs when it needs it. 97% reduction is not a typo.
  • Audit your allowedTools whitelist ruthlessly. A tool you don't whitelist still costs you the listing tokens — the server still ships its inventory — but earns you nothing. Drop the server entirely if you're not whitelisting any of it. There is no "we might need it later" mode that doesn't cost you tokens today.
  • Ship binaries, not npx launchers. Cold-start matters because tool registries seal early. Global installs and absolute paths win every time over package-manager-as-launcher. npm i -g, uv tool install, anything that resolves to a binary in under 500ms.
  • Restart long-running sessions on a budget. Context accumulates even when no one is looking. Pick a metric — articles processed, turns elapsed, total tokens spent — and force a hard restart at the threshold. The wiki-curator restart trick reclaimed 5–10× per-tick performance. It is the cheapest optimization in the building.

The reflex on the "awesome-X-AI-agents" trending list is that more skills produce a more capable agent. The reflex is exactly backwards. Every server you load costs you context, blurs the model's ability to choose, and risks losing tools to a startup race you'll never see in the logs. @steipete put it cleanly a few days ago: "modular and lean, only add what you need. Fewer skills, fewer tools = your agent works more efficiently." 112 hearts and zero pushback. The market has voted on this one. The "MCP server marketplace" goldrush is going to slam into a 200K-token wall and the survivors will be the agents that brought a knife, not a tackle box.

Your agent is not weak because it lacks tools. It is weak because it has too many.
