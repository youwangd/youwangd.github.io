Models Are Weekly Hot-Swaps Now. Your Context Layer Is the Moat.

On June 1, dexhorthy posted a quote that should have ended every "which model should I use" thread on the internet: "wow the vibe shift from codex to Claude to try opus 4.8 felt pretty quick — most builders I know back on codex and 5.5 full time within a few days. (Same ones who made the switch to codex cli around 5.2/5.3 launch)." 240 likes. Three model migrations in three weeks. The same people. Round-trip. If your strategic plan is "pick the right model," you are now pre-falsified. The half-life of model loyalty is days, and the only thing in your stack that compounds across swaps is the part that has nothing to do with the model.

The four-day round-trip

Read dexhorthy's quote again, slowly. The cohort he's describing made a model switch around May 2-3 (Codex 0.130-ish, when Codex CLI got real). They made another switch on May 28 when Anthropic dropped Opus 4.8. They made a third switch back to Codex/GPT-5.5 by June 1. Three migrations in 28 days. The duration of the longest one — Opus 4.8 → "back on codex" — was measured in days, not sprints.

The Every.to Vibe Check on Opus 4.8, published the same week, says the quiet part out loud: "I'm now switching between Codex and Claude all the time." Multi-runtime is no longer a hedge. It's the default builder posture. The mental model is no longer "my model" — it's "this task, today, on whatever's best."

And here's what makes this a structural shift instead of a fashion cycle: the cluster at the frontier is getting denser, not sparser. Vercel CEO Guillermo Rauch on June 2: "MiniMax M3 is now the leading open model on the Next.js agent evaluations. Right behind Opus & GPT5, but 10× cheaper." 984 likes. M3 is 456B parameters, 1M context, native multimodal, open-weights, released June 1. It is right behind two trillion-dollar-company flagships, while costing ten times less. The differences between #1 and #5 on agent evals are now smaller than the standard deviation of the eval suite. Cursor's Composer 2.5 — flagship coding IDE, second-largest agentic coding userbase on the planet — ships on a fork of Moonshot's Kimi K2, not on GPT, not on Claude. That's the second time Cursor has chosen a non-OpenAI / non-Anthropic default. Twice in six months.

When the quality gap between top models is smaller than the migration cost, builders re-migrate every time the gap flips sign. The frequency of swaps is now bounded by release cadence, and release cadence has gone weekly. Codex shipped three stable releases in six days last week (0.134, 0.135, 0.136-α — one stable every two days). At that cadence, "what model are you on" is a question with the same shelf-life as "what tab do you have open."

What survives the swap

Here is the piece of the conversation that almost nobody is having: when you re-point your CLI from Claude to Codex to Kimi, what actually moves and what stays? I've done this four times in the last six weeks across three machines. The answer is mechanical:

LayerSurvives the swap?Why
AGENTS.md / CLAUDE.md✅ 100%Plain text. Every CLI reads it.
~/.meshclaw/skills/ (46 skills)✅ 100%Markdown + scripts. Runtime-agnostic by design.
MCP server allow-list✅ 100%It's a config object, not a model artifact.
Learned corrections (learn_add)✅ 100%Stored as files in the workspace, replayed at session start.
Prompt template stack / SOPs✅ ~95%Minor wording drift; structure transfers.
Auth tokens / rate-limit awareness✅ 100%OS-level, not model-level.
Tool calling syntax peculiarities⚠️ 80%JSON-schema variants; mostly handled by the CLI.
The model itself❌ 0%That's the literal point. It got swapped.
Per-session context❌ 0%Always ephemeral. Always was.

Look at the column. Everything that took you weeks to build survives. The thing you spent the most public anxiety on — model choice — is the only row that gets nuked. SageCLI ships nine runtime adapters (Claude, Codex, Gemini, Qwen, Kimi, GLM, MiniMax, DeepSeek, Sonnet-on-Bedrock) and they all share one skills directory, one AGENTS.md schema, one MCP probe pipeline. That's not optionality theater. That's an admission that the runtime is the disposable part of the system. We figured it out the same time the field did, and so did Cursor, and so did Codex CLI's plugin layer.

Karpathy used the cattle-vs-pets metaphor for ML training jobs in 2018. It rotated to mean Kubernetes pods. The 2026 rotation is: the model is the cattle. Your context layer is the pet. You name your pets. You version-control them. You back them up. You cry when they die. You don't name your cattle, because next week's herd looks the same and grazes the same field.

Stop optimizing for the model. Optimize for the swap.

If the model layer is interchangeable plumbing, the engineering question is: how cheap can I make the swap? Four concrete moves that pay off across at least one re-migration each:

1. Never inline-couple a runtime in your agent spec.

Every agent definition we ship has runtime as a top-level config field, never a hard-coded model name buried in a prompt. The skills don't say "Claude, please do X." They say "do X." When I flipped my quant agent from Claude to Codex two weeks ago, the diff was one JSON line. When I flipped it to Kimi K2.5 last week to test, also one JSON line. If swapping models in your stack requires touching more than one config field, you have a portability bug, not a model preference.

2. Write skills like instructions, not Claude-flavored prose poems.

The single biggest lock-in I've seen on other people's agents is skills written in a voice. Long, hedged, "you might want to consider..." prose that reads beautifully to Claude and turns into mush on a smaller open model. Here's the same skill written two ways:

// BAD — Claude-flavored. Works great on Sonnet, ignored by Kimi.
When the user asks about deployments, you might want to consider checking
the pipeline state first, especially if it's a production deploy. There
are some nuances around blocked pipelines that you should be aware of...

// GOOD — instruction set. Works on every runtime tested.
On "deployment" intent:
1. Run `pipeline-status <name>` first.
2. If state == BLOCKED, list blockers and stop. Do not proceed.
3. If state == HEALTHY, summarize last 3 deployments.
4. Never trigger a deploy without explicit user confirmation.

The second version transfers. The first version transfers but degrades — and you don't notice the degradation until the new runtime quietly does the wrong thing on a Tuesday afternoon. Treat your skills like RFCs, not chat messages. Numbered steps. Hard constraints in plain language. No vibes.

3. MCP allow-lists are the new IAM. Treat them like infrastructure.

The set of MCP servers your agent can call survives every model migration we've done. It outlived three Claude versions, two Codex CLI rewrites, and the GPT-5 → 5.5 split. Meanwhile, what runtime was on top kept rotating. The allow-list is the part of your stack that's actually compounding value. Audit it like you'd audit IAM. Pin versions. Document why each server is there. Remove servers you stopped using six months ago. This is the highest-leverage maintenance work you're not doing.

4. Treat learned corrections like a database. Not chat history.

I have 60-something learn_add corrections in this workspace right now. They include things like "TT pendingReason is free-text 3-60 chars, no enum" and "kiro-cli seals MCP tool registry ~1s after session start, npx warm-up loses the race, use absolute binary path." Those facts cost me real debugging time. Every one of them survived this week's model swap test. None of them are written in a Claude-specific voice. None of them assume a particular model's quirks. They're facts, replayed at session start regardless of what runtime is loading. That's a database. Version it. Sync it across machines. Never commit it inside a model-specific config file.

The contrarian footnote

There is exactly one thing that will lock you in: post-training on your private corpus. If you're at the scale where you fine-tune or RL-from-feedback the runtime on your own data, switching costs go from zero back to "weeks of re-training." That's why Cursor forked Kimi K2 and RL'd it for Composer 2.5 — the lock-in just moved one layer down, from API-loyalty to checkpoint-loyalty.

For everyone smaller than Cursor — which is everyone reading this — the half-life of any specific model is days. And the AGENTS.md, the skills directory, the MCP allow-list, the learned corrections, the prompt template stack — those compound across every swap. The asset isn't which model you picked this week. The asset is the part of your stack that's still here when next week's model lands.

Build for that. Stop building for the model.
