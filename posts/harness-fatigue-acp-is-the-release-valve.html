2026-05-13 agents acp developer-tools

Harness fatigue is real. ACP is the release valve.

A Saturday morning post on r/LocalLLaMA — title: "I am overwhelmed by Harnesses" — opened with a developer listing nineteen CLI coding agents he had installed, three he was paying for, and the one he actually used (he wouldn't say which). It hit the front page in two hours. The replies are the canary: people aren't drowning in choice, they're drowning in coupling. Every harness is a wire-format-less monolith that owns your session, your permissions, your MCP config, and — quietly — the next year of your tooling decisions. ACP is the first thing that actually unbolts any of that.

The shape of the pain

Let me list what was on the OP's machine, because this is everyone's machine right now: claude-code, opencode, codex, gemini, goose (now aaif-goose), cline, aider, continue, kilo, roo, junie, cagent, fast-agent, kimi, qwen-code, vt-code, openhands, crystal (now Nimbalyst), and the in-house thing his team built last sprint. Six of those changed names in the last 60 days. crystal → Nimbalyst. block/goose → aaif-goose. ruvnet/claude-flow → ruflo. steveyegge/gastown → gastownhall. There is now a curated awesome-cli-coding-agents directory with 311 stars and 67 forks. There is a directory of directories.

The default reaction — including mine, six months ago — is "pick one and commit." That advice is wrong, and the reason it's wrong is the entire point of this post.

What you actually committed to (it isn't the binary)

When you "pick a harness," you're not picking a binary. You're picking, in approximate order of pain when you try to leave:

  1. A session model. Whether parent sessions inherit child permissions, whether external_dir deny lists propagate, whether resuming after a crash gives you the same context.
  2. An MCP config layout — and worse, an MCP error-handling style. Some swallow OAuth failures silently. Some retry forever. Some surface the raw upstream error.
  3. A permission-prompt UX. Tab to approve? Type "y"? Click a modal? Approve-once vs. approve-forever vs. session-scoped — every harness picks differently.
  4. A tool-schema sanitizer. Some models (Kimi, Moonshot) reject schemas with oneOf or recursive refs. Each harness hand-rolls its own normalization, and they normalize differently.
  5. A workspace path convention. Absolute vs. relative, symlink-resolved or not, what happens when you move the repo.

None of those are model choices. None of those should be irreversible. All of them are, today, because there's no protocol between the agent and the editor / shell / IDE — the harness is the protocol, and protocol bugs ship as patch releases.

OpenCode is one of the better-engineered harnesses — they're shipping fixes; the others are shipping the same bugs and not fixing them. And the n+1-th harness, currently being announced on Twitter as I type this, is about to ship all six bugs from scratch.

This is the part the Reddit thread couldn't articulate but everyone felt. The exhaustion isn't from using harnesses. It's from watching twenty teams independently reinvent permission-prompt UX, MCP retry policy, tool-schema sanitization, and session resume — and watching their users absorb the cost as upgrade churn.

ACP is the LSP moment

Microsoft and Red Hat shipped LSP in 2016. Before LSP, every editor had a per-language plugin: Python-mode, Java-mode, Go-mode, each re-implementing find-references, autocomplete, hover, jump-to-definition. Plugins broke whenever language servers updated. Editors specialized — VSCode for TS, IntelliJ for Java, Vim for "I'll just learn ctags." Sound familiar?

LSP collapsed the matrix. O(editors × languages) went to O(editors) + O(languages). Today every serious editor speaks LSP, every language ships a server, and the question "which editor for which language" is mostly settled by taste, not capability.

ACP — the Agent Client Protocol — is the same shape, applied to coding agents:

  • August 2025: Gemini CLI ↔ Zed first integration. Two parties, JSON-RPC, prove the loop.
  • October 2025: JetBrains and Zed announce open collaboration. This is the moment it stops being a Zed thing.
  • May 2026: The public Zed ACP registry lists 22 agents (AgentPool, Augment, Blackbox, Claude, Cline, Codex, cagent, fast-agent, Gemini CLI, Copilot, Goose, Junie, Kimi CLI, Minion, Mistral Vibe, OpenCode, OpenHands, Pi, Qoder, Qwen Code, Stakpak, VT Code) and 12 editors (Zed, JetBrains, AionUi, Emacs, marimo, Neovim, Sidequery, aizen, DeepChat, Tidewave, Web AI SDK, Obsidian).

The Register summarized it crisply on 2026-10-07: "JetBrains Backs Open AI Coding Standard That Could Gnaw at VS Code Dominance." JetBrains is anchor tenant the way Red Hat was for LSP — once a non-Microsoft IDE vendor commits, the network effect tilts.

What you get for free once both sides speak ACP:

  • Permission delegation. The protocol carries the "agent wants to run X, do you allow it" handshake. Editor renders the prompt however it wants. No more harness-specific approve-once-vs-forever modals.
  • Session portability. Sessions are protocol objects with stable IDs. Resume from any client. Switch from Zed to Neovim mid-task. The harness can crash; the session lives in the editor.
  • Streaming tool output. The exact bug OpenCode keeps patching — partial tool output rendering — is a protocol primitive. Once. For everyone.
  • Schema normalization at the edge. The agent advertises tools with a known schema. The editor can mediate. Kimi-vs-Claude schema quirks become a one-shot adapter, not a per-harness rerun.

This is what "LSP moment" means in practice. Not "everyone agreed on a buzzword." It means the surface area each harness has to own collapses, which means the cost of switching collapses, which means the harness layer commoditizes, which means you can finally pick the best agent for the model you want to use, in the editor you already love. The matrix unsticks.

What this looks like from the SageCLI side

I'll cop to this: I built SageCLI as bash, jq, and tmux glue with no framework specifically because we saw the harness-fatigue cliff coming and didn't want to marry one. That decision looked stubborn in 2025 and looks lucky now. We bet that (a) the wire format would arrive, and (b) the value lived in the corpus around the harness, not the harness binary. Both bets are paying.

Concretely, here's what we treat as portable (worth investing in, survives a harness swap) versus ephemeral (do not couple to):

LayerPortable?Why
AGENTS.md + project conventionsYes. The whole point of standardizing the filename last year.Every ACP-speaking agent reads it.
MCP server inventoryYes. MCP is the tool-side protocol the same way ACP is the editor-side one.Same servers, different harnesses, same tools.
Slash-commands / prompt templatesMostly. Harness-specific syntax, but the content moves cleanly.The prompt is yours. The trigger is theirs.
Session model / permission UXNo. Don't optimize for it. Don't memorize a harness's quirks.ACP will eat this. It already is.
Built-in tools (web fetch, file edit, etc.)No. Use MCP servers instead. Then you own the tool, harness doesn't.Built-ins are a lock-in vector dressed as convenience.

The trick is to recognize that you have a protocol-shaped investment surface (AGENTS.md, MCP, prompts) and a vendor-shaped one (everything else). Spend on the protocol-shaped one; refuse to pay tuition to the vendor-shaped one. When the next rebrand drops — and the next one is in the next 30 days, I will bet money on it — you should feel nothing. Swap the binary, keep the corpus.

The Pi escape hatch for non-ACP agents

Worth knowing: even if your favorite agent doesn't speak ACP yet, there's Pi (in the registry above) — an ACP-bridge that lets non-ACP CLIs join the network. Same trick as the LSP-vs-LSIF adapters from 2017. You don't have to wait for every vendor; you just need one bridge per vendor. The "I have to pick" framing assumes a closed set. The actual situation is that every agent is one ACP shim away from being interchangeable.

So what do you tell the OP?

The Reddit thread gets a lot of "just pick Claude Code" and "Cursor is fine" replies. Both are wrong, in the same way "just pick VSCode" was wrong in 2015. The right answer is:

  1. Stop treating harness selection as a one-way door. The whole point of a wire format is reversibility.
  2. Prefer ACP-speaking harnesses, or harnesses one bridge away. Check the Zed registry; it's now the de-facto compatibility list.
  3. Move your investment to AGENTS.md, MCP servers, and prompt templates. Those survive the next four rebrands.
  4. Treat the harness binary like you treat your shell prompt theme. Strong opinion, weakly held, swappable on a Tuesday afternoon.

The "overwhelmed by harnesses" feeling is a real signal. But the cure isn't to pick one and white-knuckle it. The cure is to notice that the layer below the harness — the protocol layer — is finally arriving, and to invest one level deeper than the thing that keeps churning. Anyone who lived through the editor wars knows the rest of this story. The plugins lost. The protocol won. The vendors who got onboard early got to keep their users; the ones who didn't got rebranded into oblivion.

Crystal is now Nimbalyst. Goose is now aaif-goose. The valve is open. Let some pressure out.
