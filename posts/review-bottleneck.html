2026-05-25 ai-agents cli-tools productivity

Reviewing Code Is the New Bottleneck

I spent the last six months building multi-agent fan-out into SageCLI. Inbox, scratch-log, kanban-of-agents, an overnight Ralph loop. All of it was rational at 40 tokens per second. Last week Cerebras started serving GPT-5.3-Codex-Spark at 1,200 tps and quietly published a best-practices doc that reads, paragraph by paragraph, like a list of things I should not have built. The headline isn't that the model got 20× faster. It's that the bottleneck moved — and the playbook flipped with it.

The 20× cliff, in one paragraph

Numbers first, because the rest of the post is downstream of these. Open-weights and frontier code models in 2025 settled around 40–60 output tokens per second. That was the budget every CLI tool, every agent harness, every kanban-of-agents UI was built against. Then Cerebras shipped Codex Spark on the WSE-3, OpenAI plugged in GPT-5.3-Codex-Spark as a research preview, and the steady-state throughput became 1,200+ tokens per second on a real coding model. Not a distilled toy. Not a 7B. The same Codex you're already paying for, served on different silicon.

20× sounds like a number. Lived in for an hour, it's a phase change. Things that used to be expensive become free:

  • Run the full test suite between every task, not every PR.
  • Generate ten UI variants and pick the best one — same wall clock as generating one used to take.
  • Lint, typecheck, and diff-review every save without breaking flow.
  • Re-prompt with full context instead of compacting; the round-trip is cheaper than the compaction.

This isn't a thought experiment. The benchmark page lists the runs. The OpenAI community thread has people copy-pasting their first-impression sessions. The thing is shipping.

The reversal nobody is publishing yet

Here's Cerebras Tip #4, from the same best-practices page, verbatim. Read it twice:

"Slower models pushed us toward parallelism. You'd open 10 tabs, fire off 10 prompts, and wait. With Spark, you're generating thousands of lines a minute. Reviewing code is the new bottleneck, and Spark works best as a pair programmer rather than a delegated workforce. Use 2–3 sessions max."

Every multi-agent CLI built in the last 18 months was an answer to the first sentence of that paragraph. Claude-Squad, Kanbots (which just hit 230 points on HN this week — peak slow-era thinking), ruflo/claude-flow, my own SageCLI fan-out primitives. The pitch was always the same: the model is the slow part, so parallelize it, and use git worktrees + AGENTS.md to keep ten of them from stepping on each other. That was correct. It is no longer correct.

The slow part is now the human reviewer. @aiDotEngineer put it bluntly on X this week: "At 20× the speed, the habits you built around slow generation don't go away. They start shipping bad code 20 times faster." 65 hearts, but the meme is spreading because it's mechanically true. Parallelism only helps when the parallelizable resource is the constrained one. Once review velocity is the constraint, every additional agent is anti-leverage — work piling up in a queue you can't drain.

The Cerebras line "you cannot let that pile up across multiple agents" is the polite version. The honest version is: the kanban-of-agents UI was a coping mechanism for slow inference. We're about to spend the rest of 2026 watching that UI become an antipattern.

What I built that ages well, and what doesn't

I'm going to be specific about my own tool, because vague self-criticism doesn't teach anything. SageCLI has roughly two halves. One ages well. One ages badly.

Ages well: stateless executor + git-as-memory

Cerebras Tip #5 and #6 are about not letting the agent maintain conversational state across tasks. Hand it a clean context every time, externalize memory to git and to disk-resident plan files, and let the executor be a pure function from (repo, task) to (diff, log). SageCLI's msg send / msg ls / msg clear primitives plus --worktree for fresh-context runs already do exactly this — by accident, because I copied the pattern from build systems, not because I anticipated the speedup.

sage send "fix the off-by-one in date_window()"
sage ls --since 5m   # scratch log: what the agent saw, what it tried
sage commit          # diff lands, context window is freed
sage clear --worktree

That loop runs at 40 tps and at 1,200 tps. It runs faster at 1,200 tps. Nothing about the API changes. The AGENTS.md / PLAN.md / VERIFY.md externalization that the whole industry converged on last quarter — same story. Disk is the right place to put state when the model doesn't have one. That doesn't change when the model gets faster; if anything, it matters more, because you're running the loop more often.

Ages well: --then validation chains

This was a nice-to-have at 40 tps. At 1,200 tps it's mandatory.

sage send "add the --strict flag" \
  --then "run pytest -x and post the failures" \
  --then "fix any failures with diff <= 50 LOC" \
  --then "run mypy strict and post the report"

The model can now generate the diff, run the suite, fix the regressions, and re-run mypy in less time than it used to take to generate the diff alone. The cost of skipping validation has not changed; the cost of doing validation has cratered. If your CLI doesn't have a way to express "do this, then verify, then fix what broke" as a single operator-driven command, you're about to look like Make without dependency tracking.

Ages badly: --pattern fan-out for N parallel agents

This is the part that hurts. sage plan --pattern fan-out --n 10 spawns ten worktrees, distributes ten subtasks, and round-robins my attention across ten kanban cards. I built it. I've used it. It was the right shape for the constraint. It is no longer the right shape.

At 1,200 tps, the same ten subtasks run sequentially in a single pair-programming session faster than I can context-switch across ten cards. The fan-out cost isn't the wall clock anymore; it's the review backlog. Ten cards waiting for review is ten times the cognitive load with zero throughput gain, because my tps is the constraint and it's nowhere near 1,200.

Ages badly: the 8-hour overnight Ralph loop

Same diagnosis. The whole point of running 8 hours of agent loops while you sleep was that 8 hours of 40-tps generation produced something nontrivial. 8 hours of 1,200-tps generation produces a backlog you cannot review in your remaining waking hours. The orchestration scaffolding — checkpoints, kill switches, budget caps — was load-bearing because the runs were long. They're not long anymore. They're minutes. The scaffolding is now overhead on a thing that doesn't need scaffolding.

The new ergonomics question

If review velocity is the bottleneck, the question every CLI tool needs to answer reframes:

Old questionNew question
How many agents can I run in parallel?How fast can a human reject the wrong diff?
How long can I leave it running unattended?How tight is the inner loop with the model in front of me?
What's the wall-clock for a 10k-LOC change?What's the median time to a green test suite per commit?
How big can the PR be?How small can I force the diff to be?

Concrete shifts I'm making in SageCLI this week:

  • Standing pre-commit hooks become default-on. Lint, typecheck, fast tests, diff size guard. They were opt-in because they cost wall-clock at 40 tps. They cost nothing at 1,200 tps.
  • Max diff guard at ≤150 LOC per agent turn. Bigger diffs are not a feature, they're a review-blocker. I'd rather the model do five 100-LOC turns than one 500-LOC turn, because I can review five 100-LOC diffs and I cannot review one 500-LOC diff.
  • /review is no longer an opt-in subcommand. Every sage send ends with a structured review summary — what changed, what got tested, what was skipped — written for human eyes, not for the next agent.
  • Demote --pattern fan-out to a flag, not a recommended workflow. Keep it for the genuine embarrassingly-parallel cases (porting a feature across 30 microservices). Stop pitching it as the default.

What I'd tell anyone shipping a CLI agent this quarter

If your tool's headline feature is "run N agents in parallel" — re-pitch around review velocity, not generation throughput. Your competitors who do this in Q3 will look like they read the room. Your competitors who don't will look like they shipped the right product for 2024.

If your tool externalizes state to git and disk — you're already aligned with where the puck is going. Don't rip that out. The faster the model gets, the more the externalization matters, because you're cycling through more clean-context runs per hour.

If your tool's pitch deck has a "swarm" screenshot with 12 agents on a kanban board — keep the screenshot, change the caption. The capability is still there. It's just no longer the headline.

And if you're a user, the smallest change with the biggest payoff is the one Cerebras buried in Tip #4: cap yourself at 2–3 sessions. Not because the tools can't run more. Because you can't review more, and reviewing is now the work.

The bottleneck didn't disappear. It moved into the chair.
