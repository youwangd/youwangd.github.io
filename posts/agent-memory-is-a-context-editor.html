Your Agent Doesn't Need a Memory Layer. It Needs a Context Editor.

My agent memory directory is 40MB on disk. The slice I inject into any given prompt is 55KB. That's a 1000× compression ratio, and it is the product. The storage layer that everyone is currently racing to build is the easy half of the problem — and solving only the easy half is going to make agents worse, not better.

The bottleneck was never storage

I keep watching "memory for agents" get framed as a storage problem. Bigger databases, better embeddings, cleverer graphs, vector stores, hybrid retrieval. All of it in service of the same verb: remember more.

But if storage were the bottleneck, the jump from 200K to 1M token context windows would have solved it. It didn't. My agents didn't get smarter when the window got bigger — they got more confused. They started quoting preferences I'd already overridden. Citing stale file paths from six months ago. Blending facts from unrelated projects. The larger the window, the worse the signal.

This is the part the "remember more" framing gets backwards. Every extra token you put in the window pushes something else out — usually the newest, most relevant thing, because attention has real recency problems and retrieval has real precision problems. Giving your agent more memory without a selection policy is giving it a bigger haystack and calling it a search engine.

The bottleneck is not persistence. It's eviction: which 5% of what the agent has ever seen belongs in the next prompt, and what gets thrown away to make room.

What actually moved the needle when I built mine

I ship a workspace-scoped memory layer as part of MeshClaw — persistent user preferences, per-project notes, a decaying history log, a semantic key-value store. On disk it's a few directories of markdown plus a SQLite file. Roughly 40MB today across a year of real use.

Building the storage took a weekend. Making the agent behave differently took six months of tuning three rules, none of which are about how to store and all of which are about what to refuse to inject.

Rule 1: hard cap on injected history, set well below any model's window

# mesh_claw/context.py
_MAX_CONTEXT_CHARS = 55_000

Opus has a million-token window. I inject ~55K chars of memory, full stop. The cap is not a cost decision — it's a quality decision. Past about that size, signal dilutes faster than new information helps. The model starts confusing projects, quoting old preferences I already overrode, citing stale paths. The cap exists because it forces everything above it to actually decide.

Rule 2: decaying weights so recent > old

History entries get exponentially down-weighted past 90 days. A correction from yesterday outranks a correction from last October by a large margin. Without decay, the memory layer is a museum — every artifact present, none prioritized. With decay, it's a working desk: the thing you touched most recently is the thing you find first.

Rule 3: workspace scoping so one project's noise doesn't leak into another

Different workspaces hold different memory. What I learned about a fee-domain service at work does not surface when I'm working on a quant repo. This is obvious in retrospect and almost nobody ships it — because "universal memory" sounds better in a README than "per-directory sandbox." Universal memory is also how agents start hallucinating relevance between unrelated codebases.

40MB on disk, 55KB injected. The 1000× compression is the product. Every hour I've ever spent tuning it has been spent on the compression, never on the storage.

Call it a context editor

The primitive we actually need isn't a memory layer. It's a context editor: a small, opinionated component that runs before the big model and answers one question — given the next turn's intent, what gets left out?

Concretely, an editor looks like this:

# the layer that's missing from almost every "memory for agents" design
def edit_context(memory_store, next_turn_intent, budget_chars):
    candidates = memory_store.all_relevant(next_turn_intent)
    # Score by recency, scope match, and past-turn usefulness
    scored = score(candidates, intent=next_turn_intent)
    # Aggressive eviction — default to DROP, not KEEP
    kept = []
    for item in scored:
        if item.score < THRESHOLD: continue
        if len("".join(kept)) + len(item) > budget_chars: break
        kept.append(compress_if_needed(item))
    return kept

Three things that editor does and a storage primitive doesn't:

  • Defaults to drop. Retrieval systems default to return — they hand you the top-k. An editor defaults to omit and makes you justify inclusion. Different posture, very different outputs.
  • Compresses in-place. A 2000-token transcript becomes a 120-token note before it ever reaches the big model. The editor owns the compression so the model never sees the raw.
  • Budgets hard. Every context slice is bounded. If the budget is 55K, the editor's job is to make the best possible 55K — not to give the model "everything relevant" and hope attention sorts it out.

The editor is a small model or a rule engine. It doesn't need to be smart. It needs to be opinionated, tunable, and visible — three properties that vector stores, graph databases, and retrieval pipelines all structurally fail to provide.

Why this keeps mattering

Every token in your prompt is a token you pay for, a token the model's attention has to dilute across, and a token that could have been something more useful instead. That arithmetic doesn't change when the window gets bigger. It doesn't change when your memory store gets fancier. It only changes when something in your stack has the authority to throw things away.

(This week a developer cancelling his Claude subscription hit #2 on HN — 946 points, 565 comments. The debate wasn't about model quality. It was about exactly this: a token layer that silently evicted his cache, silently doubled his bill, and silently pushed the model into cheaper, lazier outputs. Same arithmetic, different surface. If we build "memory for agents" as a category of storage without editors, we'll rebuild that exact failure mode in our own infrastructure — just with our names on it.)

The next primitive is not a better embedding, a bigger graph, or a cleverer schema. It's a small, visible, tunable component whose entire job is to throw things away. Build the editor. The storage is the easy part.
