Your Agent Doesn't Need a Memory Layer. It Needs a Context Editor.

My memory directory is 40MB. The slice I inject into any given prompt is 55KB. That's a 1000× compression ratio, and it's the entire product. Three "open-source memory layer for agents" projects launched on HN this week. None of them talk about the compression. All of them talk about the 40MB.

The Claude cancel post is about tokens, not quality

Monday's #2 story on Hacker News was a developer explaining why he cancelled his Claude subscription. 946 points, 565 comments. Read the debate and you'll notice what it isn't about: nobody seriously argues Opus got dumber. The thread is about token transparency.

The author's complaint, in one quote:

"When you get back to work… your conversation cache is gone and the model starts reading your codebase again. Cost-wise this is smart. But experience-wise? You paid tokens for the initial load and, after a forced break… you pay again for the same load."

Then he quotes Opus's own thinking log mid-session, after the cache evaporated:

"Rather than editing every slider in JSX, I'll add a generic initializer… This is clearly bad practice."

The model literally narrated that it was picking a lazy fix to conserve its output budget. This is the actual grievance: not that the model is worse, but that the token layer became visible and visibly adversarial. Cache invalidation double-charges you. An undocumented monthly limit kicks in mid-conversation. The agent silently cheapens its own work to make its bill smaller. The frustration is that you can't tell what you're paying for.

The memory-layer gold rush is solving the wrong problem

The same week the Claude post hit, three projects launched framing themselves as "memory for agents":

  • wuphf (110 pts) — "a Karpathy-style LLM wiki your agents maintain"
  • stash (83 pts) — "open source memory layer so any AI agent can do what Claude.ai and ChatGPT do"
  • MenteDB (16 pts) — "an open-source memory database for AI agents, in Rust"

All three ship storage primitives. None of them ship a selection primitive. Their READMEs talk about schemas, embeddings, vector stores, hybrid retrieval. Zero talk about eviction policy, recency decay, or what to drop.

This is a category error. If the bottleneck were storage, raising context windows from 200K to 1M would have fixed everything. It didn't. Every extra token you put into the window pushes something else out — usually the newest, most relevant thing, because attention has real recency problems and retrieval has real precision problems. Giving your agent more memory without a selection policy is giving it a bigger haystack and calling it a search engine.

And here's the direct line back to the Claude cancel post: every token your "memory layer" injects is a token you pay for. A memory layer without an editor is exactly what the HN author is angry about — invisible infrastructure silently charging him for context he didn't ask to load.

What actually moved the needle when I built mine

I shipped a workspace-scoped memory layer as part of MeshClaw — persistent user preferences, per-project notes, a decaying history log, a semantic key-value store. On disk it's a few directories of markdown plus a SQLite file. Roughly 40MB today across a year of use.

The lessons that moved agent quality had almost nothing to do with how I stored things. They were about what I refused to inject.

Rule 1: hard cap on injected history, set well below any model's window

# mesh_claw/context.py
_MAX_CONTEXT_CHARS = 55_000

Opus has a million-token window. I inject ~55K chars of memory, period. The cap is not a cost decision — it's a quality decision. Past about that size, signal dilutes faster than new information helps. The model starts confusing projects, quoting old preferences I already overrode, citing stale file paths. The cap is a forcing function for the selection layer above it.

Rule 2: decaying weights so recent > old

History entries get exponentially down-weighted past 90 days. A correction from yesterday outranks a correction from last October by a lot. Without decay, your memory layer is a museum — every artifact present, none prioritized. With decay, it's a working desk.

Rule 3: workspace scoping so one project's noise doesn't leak into another

Different workspaces have different memory. What I learned about a fee-domain service does not show up when I'm working on a quant repo. This is obvious in retrospect and nobody ships it — because "universal memory" sounds better in a README than "per-directory sandbox." Universal memory is also how agents start hallucinating relevance between unrelated codebases.

40MB on disk, 55KB injected, and the 1000× compression is the product. The storage was the easy half.

Call it a context editor

The primitive we actually want isn't a memory layer. It's a context editor: a small, opinionated component that runs before the big model and answers one question — given the next turn's intent, what gets left out?

Concretely, an editor looks like this:

# pseudocode — the layer that's missing from every "memory for agents" project
def edit_context(memory_store, next_turn_intent, budget_chars):
    candidates = memory_store.all_relevant(next_turn_intent)
    # Score by recency, scope match, and past-turn usefulness
    scored = score(candidates, intent=next_turn_intent)
    # Aggressive eviction — default to DROP, not KEEP
    kept = []
    for item in scored:
        if item.score < THRESHOLD: continue
        if len("".join(kept)) + len(item) > budget_chars: break
        kept.append(compress_if_needed(item))
    return kept

Three things that editor does and a storage primitive doesn't:

  • Defaults to drop. Retrieval systems default to return — they give you the top-k. An editor defaults to omit and makes you justify inclusion. Different posture, very different outputs.
  • Compresses in-place. A 2000-token transcript gets summarized to a 120-token note before it goes in. The editor owns the compression, so the big model never sees the raw.
  • Budgets hard. Every context slice is bounded. If the budget is 55K, the editor's job is to make the best possible 55K — not to give the model "everything relevant" and hope attention sorts it out.

Cloudflare's Code Mode made a sibling argument this week from a different angle: agents should inhabit state machines, not regenerate text. Same insight, one level up — the durable win is in not re-materializing context, not in storing more of it. An agent that codes against a stable state machine is an agent that doesn't need to re-load its situation into the prompt on every turn. Less context is the feature, not a compromise.

Why this matters for the Claude cancel post

The HN author's anger isn't really at Anthropic. It's at the fact that the layer deciding what ends up in his prompt is opaque, adversarial, and billed. He can't see it, can't tune it, and the vendor has every incentive to evict his cache to save their own money.

If "memory for agents" becomes a category, and every project in it ships storage without an editor, we'll have recreated the exact failure mode — just in our own infrastructure this time. A memory DB happily dumps 200K tokens of "relevant history" into every turn, and you pay for it, and the model loses the thread, and you don't know which half to cut.

The next primitive is not a better embedding, a bigger graph, or a cleverer schema. It's a small, visible, tunable component whose entire job is to throw things away. Build the editor. The storage is the easy part.
