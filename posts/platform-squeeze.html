2026-04-11 open-source agents platforms

The Platform Squeeze Is Here

On April 4, Anthropic told Claude Code subscribers they'd need to pay extra to use OpenClaw. One week later, steipete posted "it's gonna be harder in the future to ensure OpenClaw still works with Anthropic models" — and got 4,600 hearts. The open-source AI tooling ecosystem just hit its first real inflection point.

What Actually Happened

Three things converged in the same two-week window:

  1. Anthropic locked out third-party harnesses. Claude Code subscribers can no longer use their subscription limits for OpenClaw or other external tools. Want to use OpenClaw with Claude? Pay separately, per-token. Boris Cherny, Anthropic's head of Claude Code, framed it as "engineering constraints" and "sustainable growth." Peter Steinberger, OpenClaw's creator (who'd already joined OpenAI), called it out: "First they copy some popular features into their closed harness, then they lock out open source."
  2. Anthropic shipped Agent Teams. Official multi-agent orchestration, built into Claude Code. One session acts as team lead, coordinates work, assigns tasks. The docs are polished. It does what external orchestrators like sage, claude-flow, and Claude Squad have been doing — but natively, with no extra tooling.
  3. The MCP vs Skills debate exploded. A blog post titled "I Still Prefer MCP Over Skills" hit 262 points and 227 comments on Hacker News. The author's core argument: Skills require CLIs and local execution, while MCP provides clean remote APIs with auth, discovery, and portability. Meanwhile, someone ported Anthropic's skill-creator to OpenCode — making it work with any model, not just Claude.

These aren't isolated events. They're the same story: the platform is pulling up the ladder.

The Playbook We've Seen Before

This is the oldest pattern in tech platforms:

  1. Platform launches with open APIs and encourages third-party tools
  2. Third-party tools prove the market and discover what users want
  3. Platform builds native versions of the most popular tools
  4. Platform makes it harder (or more expensive) to use the third-party versions

Twitter did it to third-party clients. Apple did it to Sherlock. Salesforce did it to countless AppExchange partners. Now Anthropic is doing it to OpenClaw.

The sequence is textbook. OpenClaw proved that developers want an open-source, extensible harness for Claude. Anthropic watched, learned what features resonated (skills, multi-file editing, agent orchestration), built Claude Code with those features, and then made it economically painful to use the open-source alternative.

Steinberger saw it coming. He joined OpenAI in February. By April, Anthropic had locked the door.

What This Means for Agent Tooling

If you're building tools that depend on a single AI provider, you're building on rented land. This isn't a hypothetical risk anymore — it happened, publicly, to the most popular open-source AI coding tool in the world.

The implications cascade:

Orchestrators that only work with Claude are vulnerable. Claude Squad, claude-flow (now ruflo at 31K stars) — if Anthropic decides Agent Teams is good enough, they can make external orchestration progressively harder. Rate limits, API changes, pricing tiers. They don't need to block you outright. They just need to make the native option frictionless and the external option annoying.

MCP is safer than Skills — for now. The HN debate crystallized something important. MCP is a protocol — it works across providers. Skills are often tied to a specific agent's execution environment. When david.coffee writes "I don't want a future where every single service integration requires a dedicated CLI and a markdown manual," he's describing the fragility of provider-specific tooling. MCP servers work with Claude, GPT, Gemini, local models. A Claude Code skill works with Claude Code.

Runtime-agnostic is no longer a nice-to-have. It's a survival strategy. When I built sage as a pure-bash orchestrator that works with Claude Code, Cline, Gemini CLI, and any ACP-compatible agent, the "why not just use Claude Code's built-in features?" question was fair. Now the answer is obvious: because Claude Code's built-in features come with Claude Code's built-in lock-in.

The Three-Layer Defense

If you're building in the AI agent ecosystem — whether you're an individual developer or a startup — here's how to think about platform risk:

Layer 1: Protocol over product. Build on MCP, ACP, and A2A — not on Claude Code APIs or OpenClaw plugins. Protocols survive platform shifts. Products don't. MCP works with every major model provider. ACP works with 25+ agents. If one provider squeezes you, you swap the backend without rewriting your tools.

Layer 2: Own your orchestration. Don't let the model provider decide how your agents coordinate. If Anthropic's Agent Teams is your orchestration layer, Anthropic controls your workflow. If your orchestration is a bash script that dispatches to any runtime, you control your workflow. The orchestrator should be dumber than the agents — just routing, scheduling, and state management. No LLM calls in the control plane.

Layer 3: State lives on your disk. Agent context, task history, results, memory — all of it should be files you own, in directories you control. Not in a provider's cloud. Not in a session that evaporates when the process dies. When sage stores task state as JSON files in ~/.sage/agents/, that's not a limitation of bash — it's a deliberate architectural choice. Your data survives any provider change.

Where This Goes

The next 12 months will sort the AI tooling ecosystem into three categories:

Platform-native tools that live inside Claude Code, Codex, or Gemini. Fast to adopt, impossible to migrate. You're a tenant.

Provider-aligned tools like OpenClaw (now OpenAI-backed) or Copilot extensions. Better than native, but still gravitationally bound to one ecosystem.

Protocol-native tools that speak MCP/ACP and work with any backend. Harder to build, harder to market ("works with everything" is a weaker pitch than "works perfectly with Claude"). But they're the only ones that survive a platform squeeze.

The OpenClaw situation is a gift to the third category. Every developer who got burned by the Anthropic lockout is now asking: "What tools do I use that won't do this to me?" That question didn't exist two weeks ago. Now it's the first thing people think about.

Steinberger built something remarkable with OpenClaw. Tens of thousands of developers relied on it daily. And one email from Anthropic changed the economics overnight. That's not a failure of OpenClaw — it's a structural vulnerability of building on a single provider's goodwill.

The lesson isn't "don't build open source." The lesson is: build open source that doesn't need permission from the platform to exist.
