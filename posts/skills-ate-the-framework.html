Skills Ate the Framework: How a Markdown File Beat the SDK

If you want to know what won the agent-tooling fight, open one tab: agentskills.io. Quietly, six months after Anthropic released the SKILL.md format as an open standard in December 2025, the homepage's "Client Showcase" lists Cursor, Gemini CLI, opencode, OpenHands, and mux alongside Claude itself — five non-Anthropic runtimes shipping the same primitive. Nobody held a press conference. There is no LangSkills 1.0. The de-facto standard for "give an agent a new capability" is now a folder with a markdown file in it. The Python class hierarchy your favourite framework was selling you in 2024 lost without a fight.

The single page nobody is quoting

Cross-vendor adoption announcements normally show up as competing blog posts six months apart. Skills did the opposite — they showed up as a list on one page, with no marketing budget behind it, and almost none of the agent-tooling discourse has caught up. The agentskills.io homepage states it plainly:

"The Agent Skills format was originally developed by Anthropic, released as an open standard, and has been adopted by a growing number of agent products."

What follows on the same page is a Client Showcase grid. I screenshotted it on 2026-06-13 and counted: five non-Anthropic clients reading the same SKILL.md folders, plus Claude. The community caught it earlier — the top thread on r/ClaudeAI from 2025-12-18 ("Agent Skills is now an open standard") spent its top 200 comments on exactly one phrase: "cross-platform portability." That was always the prize. Whoever owned the unit of agent capability that ran on every runtime owned the next layer of the stack. Anthropic shipped the format, then walked away from owning it. That walk-away is why it works.

Compare what didn't happen on a single page. There is no "frameworks.io". There is no list of agent products that all run LangGraph workflows interchangeably. There is no list of agent products that all run AutoGen graphs interchangeably. The frameworks didn't lose because they were bad. They lost because none of them ever produced an artefact you could move from one runtime to another. A LangGraph node is a Python class wired to a particular planner; the moment you change runtime you rewrite from scratch. A SKILL.md is a folder. It moves.

The portability gradient

If you draw the agent-capability formats on a single axis — least portable to most portable — the pattern that won is obvious in hindsight:

Format Where it runs Cost to move runtime
Python class in framework X Runtime X only Rewrite
JSON tool schema (vendor-specific) One model family Translate + retest
MCP server (binary + JSON-RPC) Any MCP-aware client Reconnect, hope the model picks it
SKILL.md folder Any runtime that can read text and call tools Copy folder. Done.

Read the right column and tell me you're surprised. The lowest-friction format has zero binary dependency, no schema version, no SDK pin. It is markdown. The model is the runtime. Every layer above it became a layer you don't need.

Why markdown was the only thing that could win

The agent-framework category spent two years selling the wrong abstraction. The pitch was: "your agent's capabilities should be expressed as a graph of typed Python objects so that we can reason about them statically." Reasonable for compilers; wrong for prompt-time reasoning. The actual consumer of an agent capability is the model, and the model already reads text better than your framework reads its own schema. Every layer between the markdown and the model is a layer of translation loss:

  • Class hierarchies require the human to translate intent into types. The model doesn't read types — it reads docstrings, which are markdown stuck inside Python triple-quotes.
  • JSON schemas require both producer and consumer to agree on a shape. The model patches over malformed JSON anyway.
  • Plugin APIs require an SDK upgrade to add a capability. SKILL.md requires git pull.

Markdown won because the model was always the abstraction layer. The framework tier was a translation layer that translated from human intent to a representation the model couldn't use, and back. Removing it didn't lose information. It saved the round trip.

This is also why "MCP vs Skills" is a category error you see on Reddit a lot lately. They don't compete. MCP is the wire protocol for a capability that needs a process — the GitHub server, the Slack server, the database server. Skills are the prompt-time discipline a model uses to decide what to do with those tools, plus the small bits of in-repo logic that don't need a server at all. The week that became obvious to me was the week I deleted three internal MCP servers and replaced them with one SKILL.md that called out to the AWS CLI directly. The SKILL.md was 40 lines. The MCP servers it replaced were 1,400.

The bloat is back

Whenever a portable format wins, the second-order disaster is curation. node_modules happened to JavaScript. requirements.txt happened to Python. The MCP-tax post on this blog two weeks ago was the same shape: 143K tokens consumed before the user typed a character, because the menu kept growing.

Skills are already on this curve. Today's GitHub trending digest, 2026-06-13:

  • sinhoneyy/master-skills — "Unified skill library for Claude, Codex, Cursor, Antigravity & AI agents — 2,658 skills across 15 domains." Two thousand, six hundred and fifty-eight. In a single repo. Trending.
  • cinicu/opencode-dotfiles — "19 AI agent skills + 8 MCP servers + persistent memory."
  • nuke-on-rails — "Rails audit skill for AI coding agents, ranked by blast radius."
  • tuwulalo/git-sync-check — "Agent Skill + zero-dependency scripts ... any harness, humans."

Read those four titles in order. The first one is the bloat ("just install all 2,658"). The next three are the early antibodies — blast-radius tagging, narrow scope, harness-agnostic. The mature pattern is in the antibodies, not in the inventory. The lesson the MCP world is still learning the hard way — fewer tools, fewer skills, better outcomes — is going to repeat in the skills ecosystem about a quarter behind.

The model-side data already says it. From the "MCP Tool Descriptions Are Smelly" paper I cited last week: tool-selection accuracy drops as the toolset grows. Same physics applies to skills. A model that lazy-loads three relevant SKILL.md files for a task will out-decide a model handed all 2,658 every turn. The winning skill libraries in 2027 will not be the biggest. They will be the ones with the meanest curation policy.

Receipts: what I built, what I deleted

I am writing this from a Cloud Desktop with 27 production SKILL.md folders in ~/.meshclaw/skills/ (excluding two visualisation and pet-companion families that bring it to 49 total). Each one of them replaces what would have shipped as a Python plugin in MeshClaw v1. The framework I started building in early 2026 had a class registry, a plugin loader, a YAML manifest, a typed event bus. All of it is gone. The "framework" is now: a session prompt, a tool registry, and "load this folder of markdown."

Concrete receipts. The production behaviour of ticket-coop-scanner (cron 6b85f32d, three internal queues, 21-ticket window 2026-06-03 to 06-08, 62% delivered ≤2 oncall turns) is defined entirely by:

File What it is Lines
~/.meshclaw/skills/ticket-coop-scanner/SKILL.md State machine, escalation rules, voice/tone, redlines 170
workspace/ticket-coop-scanner/reviewer_persona.md Cross-check reviewer prompt 170
workspace/ticket-coop-scanner/pick_next.py State derivation + cron tick 296
Total 636

Two-thirds of that is markdown. The Python is the glue, not the brain. Earlier this year the same scope sketched out as a "ticket triage agent framework module" had a router, a triage agent, a fix-proposer, a reviewer, a poster — five classes, an event bus, ~2,000 lines, plus a config schema I had to keep updating. Every one of them is now one section in one SKILL.md. The escalation rules are markdown. The redlines (no Shepherd self-resolve, no git add -A in Brazil workspaces, no internal context in WORKLOG comments) are markdown. The voice is markdown. The deletion was the punch line.

The cleanest test is portability. I dragged that same SKILL.md into a vanilla Claude Code session yesterday with no MeshClaw runtime and no pick_next.py. The model immediately understood the state machine, asked for the missing tool registry, and was running the right escalation logic on a fake ticket within two turns. The "framework" was the session prompt. The capability was the markdown. Move the folder, move the agent.

The next year

If you're betting where the agent-tooling stack lands by mid-2027, here is what the agentskills.io adopters list and the GitHub trending bloat are both telling you, even though they look like opposite signals:

  1. The framework tier loses its dignity quietly. No crash, no acquisition fire sale — just an accelerating list of "we now also support skills" lines on framework READMEs, the way "we now support TypeScript" landed in JS frameworks five years ago. The framework keeps shipping. Its reason to exist hollows out.
  2. The skill curation tier becomes the next moat. "We have 2,658 skills" is the wrong answer. "We loaded the right three for this task" is the right one. Whoever ships the lazy-loader / blast-radius tagger / one-skill-per-job linter for skills wins what the MCP-compressor crew won for tool inventories.
  3. The runtime becomes the model. This was true already and skills just made it visible. The session prompt + tool registry + folder-of-markdown is the entire 2026 agent stack. Everything above the model is curation; everything below it is plumbing. The middle layer is gone.

I'm writing this in Markdown, in a folder, on a runtime I might not be using next month. The post will outlive the runtime. So will the skills. That was always going to be the joke at the end of the framework era — the most portable thing in the room turned out to be the format the engineers were too embarrassed to admit they preferred all along.
