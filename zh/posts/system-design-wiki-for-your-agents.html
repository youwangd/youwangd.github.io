2026-04-23 知识系统 智能体 架构

给 AI 编一本系统设计百科，它就不再瞎编架构了

Agent 一本正经地让我在一条写多读少的路径前面加 Redis 缓存。没给任何出处。它不知道 Netflix 两年前发过一篇复盘，讲得很清楚：这套做法在他们的量级下会怎么塌。跟它争没用。我直接给它编了一本 wiki。

Agent 并不懂系统设计

它只是对「系统设计」附近经常出现的 token 有感觉。和懂，不是一回事。

2025 年大家不太在意，反正让它写的多半是 CRUD，写对的概率挺高。到了 2026 年，它开始整个服务地写。「差不多对」的含义变了：熔断没有抖动、分片键全打在热点、事件 schema 没有版本演进方案。每一条都是一次真实的线上事故。每一条在 Netflix、Uber、Stripe、Cloudflare 的工程博客里都写过。训练时它读过，但记不住。

第一反应是上 RAG。我试过，结果比想象中差。切片取回来全是脱离上下文的碎片，rerank 又会编造相关性，最后 agent 用 Meta 的视频编码博客来论证数据库复制方案。RAG 负责检索，不负责编译。

Karpathy 那篇 LLM wiki 说了另一种办法：不去检索原始文档，而是把它们蒸馏成一张互相链接的 markdown 知识图。Agent 读的是 wiki，不是语料。wiki 是产物，语料只是原料。

我照这个思路给软件系统设计编了一本。两个月，3210 页，280 个精选来源，每 6 小时自动入库，静态站点跑在 Cloudflare Pages 上。站点：sysdesign-wiki.pages.dev。

整套东西塞得进一个 cron

没有数据库，没有向量库。就三个目录，一条规矩：raw 不可变，wiki 是产物，不存在第三样东西。

sysdesign-wiki/
├── raw/      // 抓来的原文，按公司分目录
├── wiki/     // 编译出的知识库，就是产物
│   ├── concepts/ patterns/ systems/ companies/
│   └── sources/   // 每篇入库文章一页
└── tools/    // 抓取、入库、lint、构建

发布流水线是 cron 里跑的 30 行 bash：

#!/usr/bin/env bash
set -euo pipefail
cd /home/dyouwang/sysdesign-wiki
exec 9>/tmp/sysdesign-wiki.lock
flock -n 9 || exit 0

python tools/rss_pull.py              # 29 个源 → raw/
python tools/ingest.py --incremental  # raw/ → wiki/sources/
python tools/gen_category_indexes.py
python tools/gen_landing_page.py

git add -A
git diff --cached --quiet || {
  git commit -m "auto-ingest $(date -u +%FT%H:%MZ)"
  git push
}

推一次，Cloudflare Pages 自己重建，90 秒后站点就更新完了。策展 agent 跑在 sage 后台，每来一篇原文就写一页 source：一段话的提炼、3 到 5 条 takeaway、几条 [[wiki-link]] 连到它涉及的概念和模式页。凡是不 trivial 的论断都带出处。两家公司观点冲突时，加一段 ## Contradiction，两边立场都写清楚。不假装共识。

同一个 prompt，两次结果

我拿同一个题目跑过两次：「设计一条每秒 10 万事件的写入路径，每一步都要有业界先例撑腰」。

RAG 那次拉回来 6 个切片：一条 LinkedIn 讲消费者延迟的，一段 Stripe 幂等 key 的节选，一份 AWS 调优文档，再加两篇完全无关的 Netflix 视频编码博客，只因为「高吞吐」这几个字都出现过。最后的方案读起来很权威，内部逻辑是散的。我追问某个选择的依据，它引了那篇视频编码的文章。

Wiki 那次打开了 patterns/write-ahead-buffering.md、systems/kafka.md、concepts/backpressure.md。本来就互相链好。每条论断本来就带出处。它写了四段话，六条引用全都对得上。我追问依据，它指着一段 ## Contradiction 说：「LinkedIn 和 Uber 在这点上有分歧，我选了 LinkedIn 的，因为你的流量画像和他们的 case 更接近。」

两次用的是同一个模型。区别在于，wiki 那边，思考工作已经做完了。

编译出来的知识，才是护城河

语料不会复利。语料再多也只是草垛更大。wiki 会复利。Uber、Square、Shopify 都写过幂等，patterns/idempotency.md 不会因此变长四倍，只会更锋利。从「Stripe 是这么做的」，变成「业界收敛到了 X，因为 Y；到了 Uber 这个量级得换成 Z」。后面所有 agent 任务，白捡这份锋利。

OpenAI 的 Frontier 团队讲过他们如何零手写代码地交付一百万行。办法之一是把 AGENTS.md 写成目录，而不是百科。百科放在结构化的 docs/ 里——那其实就是一本 wiki，只是没这么叫。真正沉淀下来的东西，不是 agent 交付的那份代码，而是人提前编译好的那套脚手架。有了它，agent 才能查阅，而不用硬背。

所以认真在用 agent 干活的，护城河不在模型、不在 prompt、不在 harness，而在 agent 动手前读的那份编译好的知识。再大的上下文窗口救不了你。只有一份编译好、带引用、持续更新的知识库救得了。

最小可用版本

3210 页是不必要的。先写十页就够了。

mkdir -p wiki/{concepts,patterns,systems,sources}
cat > AGENTS.md <<'EOF'
你负责维护关于 <你的领域> 的知识库。
先读 wiki/index.md。不 trivial 的论断都要带出处。
来源冲突时，加一段 ## Contradiction。
EOF

手写十页概念，每页 500 字左右：一句话定义、3 条要点、3 条引用。把 agent 指向这个目录，它就不再瞎编了。等你写页写烦了，再去自动化 ingest。

编译知识是人的活，使用知识是 agent 的活。别再跟它争那个熔断器该不该加。页写出来。
