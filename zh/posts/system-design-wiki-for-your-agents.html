2026-04-23 知识系统 智能体 架构

我给 AI 建了个系统设计 wiki，因为它总在瞎编架构

那个 agent 信誓旦旦地建议我在写密集路径前面塞一个 Redis 缓存。没有任何引用。它完全不知道 Netflix 两年前发过一篇事故复盘，把这种做法在他们那种规模下为什么崩溃写得清清楚楚。它不知道，是因为它一年前把互联网整本压进了权重里,现在权重一薄就开始瞎编。我懒得再跟它争了,给它建了个 wiki。

AI 编码这波讨论里,大家都不愿意说的一个前提

你的 agent 不懂系统设计。它懂的是「系统设计」这几个字附近经常出现哪些 token。这是两码事。

2025 年 4 月大家不在意这个区别——你让它写的大多是 CRUD,它确实基本写得对。2026 年 4 月的 agent 在写整个服务,「基本对」意味着你的 circuit breaker 没加抖动、你的 sharding key 打到热点上了、你的事件 schema 没有版本演进方案。这每一条,都是正在等着把你拉去半夜应急的生产事故。而这每一条,Netflix、Uber、Stripe、Cloudflare 的工程博客里都写过。模型训练时读过这些文章。它就是记不住。

直觉反应是上 RAG:把工程博客塞进向量库,按需检索。我试过,比你想的还糟。切片回来的是脱离上下文的碎片,reranker 会编造相关性,结果你的 agent 拿 Meta 一篇视频编码的博客去论证数据库复制该怎么设计。RAG 是检索,不是编译。

Karpathy 最近推了一个不一样的思路,叫 LLM wiki:别去检索原始文档,把它们蒸馏成一个持久的、互相链接的 markdown 知识图。Agent 读的是这个 wiki,不是原始语料。wiki 是产品,语料只是它被编译出来的原料。

我按这个思路给软件系统设计建了一个。两个月了,3210 页,280 个精选 source,每小时自动入库,部署在 Cloudflare Pages 上。站点在这:sysdesign-wiki.pages.dev。下面讲讲我踩了什么坑,为什么我认为这就是 AI 编码时代知识该有的形态。

三层架构,一条规矩

架构特意做得很无聊:

sysdesign-wiki/
├── raw/         // 抓来的原始文章,不可变,一个公司一个子目录
│   ├── netflix/ uber/ meta/ aws/ cloudflare/ stripe/
│   └── _feeds.yaml
├── wiki/        // 编译后的知识库——真正的产品
│   ├── concepts/    // sharding、consensus、CAP、CRDT、backpressure
│   ├── patterns/    // circuit-breaker、saga、CQRS、bulkhead
│   ├── systems/     // Kafka、Spanner、Dynamo、TAO、Haystack、Envoy
│   ├── companies/   // 每家公司的技术栈 + 经验总结
│   └── sources/     // 每篇文章一个 source 页
└── tools/       // scraper、RSS poller、入库、lint、压缩

规矩就一条:raw 不可变,wiki 是产品,没有第三样东西。没数据库。没向量库。没搜索后端,就是 mkdocs 自带的那个。我想知道「Netflix 的 regional failover 怎么做的」,不查 embedding。我打开 wiki/concepts/regional-failover.md,三段话,四条引用,每条都链回 raw/netflix/。就这样。

任何不是显然成立的论断,都要引用 source:(Source: [[sources/2026-01-15-netflix-chaos]])。两家公司意见不一致的时候——这种情况天天有——就加一个 ## Contradiction 段落,把两种立场都点名。不假装大家其实达成了共识。

整条入库流水线就是 30 行 bash

我把全文贴出来,因为大家现在特别爱给一个放得进 cron 的问题搞一套 Kubernetes 的方案。

# cron_publish.sh —— 每 6 小时跑一次
#!/usr/bin/env bash
set -euo pipefail

cd /home/dyouwang/sysdesign-wiki
exec 9>/tmp/sysdesign-wiki.lock
flock -n 9 || exit 0   # 不重叠执行

python tools/rss_pull.py              # 29 个 RSS 源 → raw/
python tools/ingest.py --incremental  # raw/ → wiki/sources/
python tools/gen_category_indexes.py  # 自动生成索引页
python tools/gen_landing_page.py      # 首页统计刷新

git add -A
git diff --cached --quiet || {
  git commit -m "auto-ingest $(date -u +%FT%H:%MZ)"
  git push
}

剩下的交给 Cloudflare Pages。一个 push 触发构建,mkdocs 渲染静态站,搜索索引被后处理到 25 MiB 的单文件上限以下,90 秒后站点就上线了。没服务器。没守护进程。这整套系统最贵的部分是 GitHub 的免费额度。

入库脚本里真正有意思的部分:拉 29 个工程博客的 RSS,对于那 16 个屏蔽 feed reader 的站(Meta、Google、LinkedIn 是老面孔),用一个带住宅 UA 的 HTML 爬虫补上。每篇文章落成一个 markdown。然后一个 curator agent——跑在 sage 后台——读这个 markdown,写出一页结构化的 source:标题、一段蒸馏、3-5 条 takeaway、一串 [[wiki-link]] 指向它触及的 concept/pattern/system 页。

curator 做的一件比其他所有事都更重要的事,是入库前先过 scope filter。跑题文章直接拒。伪装成工程博客的产品发布稿。套着基建外壳的 ML 论文。厂商 PR。全跳过,一行理由写进 wiki/log.md。没有这一层,wiki 一周内就被噪音淹死了。

为什么这比 RAG 更适合架构决策

下面这个 prompt 我跑了两次,一次用 RAG 查原始语料,一次让 agent 读编译好的 wiki:

我要做一个服务,每秒吃 10 万条事件,落主库。设计写路径。每一个选择都要有业界先例支撑。

RAG 版本捞回来 6 个 chunk:LinkedIn 一篇 Kafka consumer lag 的博客、Stripe 幂等性文章的一个片段、AWS DynamoDB 调优指南的两段、两篇完全不相关的 Netflix 视频编码博客——就因为「high throughput」这个词在它们里都出现过。agent 写出来的设计听起来很权威,内部互相矛盾。我追问某个选择的理由,它引了视频编码那篇。如果我当时没盯着,我就把这玩意儿合进主分支了。

wiki 版本打开了 wiki/patterns/write-ahead-buffering.md、wiki/systems/kafka.md、wiki/concepts/backpressure.md。每一页已经互相链好了。每一个论断已经带了 citation。agent 给出四段话、六条引用,每一条都对得上号。我追问某个选择,它指着一个 ## Contradiction 段落,点出 LinkedIn 的立场 vs Uber 的立场,然后说「我选 LinkedIn 那种,因为你的流量画像更接近他们的 case study」。

差别不在模型。两次同一个模型。差别在于 wiki 已经把思考做完了。RAG 捞碎片。wiki 直接把编译好的答案递过来。

没人聊的复利

语料和编译后的知识库最根本的区别,是语料不会复利。更大的语料就是更大的草堆,每次检索都从零开始。

wiki 会复利。每次入一篇新 source,不是往里加噪音,而是被蒸馏进已有的页。我第一次加 Stripe 幂等 key 的文章时,新建了 wiki/patterns/idempotency.md。到第四次——Uber、Square、Shopify 都发过同一个 pattern——这一页不是长到原来的四倍。它变更锋利了。分歧被点出来了,经典权衡被写得更干净了。从「Stripe 是这么做的」演化成「业界收敛到这个做法是因为某某原因,除了 Uber 那种规模得换 Z」。这是一种语料给不了你的知识增益。

而 agent 读这一页的时候,就顺便继承了这种变锋利。curator 的活只做了一次。之后每个 agent 任务都受益。

2026 年做 agent 的人,这意味着什么

我想把话说得直接一点,因为我觉得整个行业在这件事上权重给低了。认真用 AI 编码 agent 做事的人,护城河不是模型,不是 prompt,不是 harness。是 agent 开工之前读的那本编译好的知识库。

OpenAI 的 Frontier team 零人类代码写出百万行的办法,是把 AGENTS.md 写成目录,不是百科全书。百科全书放在一个结构化的 docs/ 目录里。那个目录就是一本 wiki。他们没这么叫,但就是这个东西。他们说得很清楚:团队真正攒下来的资产,不是 agent 交付的代码。是 scaffolding——质量分、execution plan、design doc。这些东西是人类编译出来、让 agent 能导航而不是背诵的。

我认识的认真跑 agent 的人,都在往同一个形状上收敛:

  • 一份短的 AGENTS.md,是地图,不是手册(100 行,不是 2000 行)
  • 一个结构化的知识库,agent 需要深度的时候去读(wiki、docs/、design-docs/)
  • 一条入库流水线,不需要人盯着也能让知识库一直新
  • 一个 lint,能抓到代码和知识库不一致

如果你在发 agent,但上面这几样一样都没有,你的 agent 现在就在瞎编架构,而且正在往生产发。模型救不了你。harness 救不了你。更大的上下文窗口救不了你。只有一本编译过的、带引用的、保持更新的知识库能救你。

最小可行版

不用搞 3210 页。你只需要对你系统最关键的那 10 个概念。从这儿开始:

# 20 分钟搭一个 wiki
mkdir -p wiki/{concepts,patterns,systems,sources}
cat > AGENTS.md <<'EOF'
你是 <你的领域> 知识库的维护者。
先读 wiki/index.md。不是显然成立的论断必须给 source。
source 之间冲突就加一段 ## Contradiction。
EOF
echo "# Index" > wiki/index.md
# 手写 10 页 concept,每页大约 500 字
# 每页: 一行 summary、3 条 takeaway、3 条 citation
# 这比你会搭的任何 RAG 都更有用

就这。10 页,手写,带 source。把 agent 指向这个目录。看着它停止瞎编。等你写烦了,再去自动化入库。

Agent 的工作是用知识。你的工作是编译知识。工具很便宜,权重不是瓶颈。瓶颈是 agent 在写下第一行代码之前读到的那个东西,有没有真正反映出你对自己系统是怎么崩的那份了解。

别再跟 agent 吵 circuit breaker 该怎么加了。把那一页写出来。
