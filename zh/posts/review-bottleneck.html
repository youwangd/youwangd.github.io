2026-05-25 AI编程 命令行 生产力

1200 tokens/秒之后,10 个 tab 就死了

过去半年,我在 SageCLI 里塞进去的东西,大概一半是"多 agent 并行"那一套——inbox、scratch-log、kanban-of-agents、还有一个跑通宵的 Ralph loop。在 40 tokens 每秒的世界里,这些设计都成立。上周 Cerebras 把 GPT-5.3-Codex-Spark 的稳态吞吐量推到 1,200 tps,顺手发了一篇 best practices,我一段一段读下去,越读越像在读自己的差评清单。重点不是模型快了 20 倍,而是瓶颈搬家了——playbook 也跟着翻了过来。

20 倍是个相变,不是个数字

先把数字摆出来,后面整篇都是从这里推出来的。2025 年,无论开源权重还是闭源前沿模型,代码生成的稳态输出都卡在 40-60 tps 这个量级。所有 CLI 工具、所有 agent harness、所有 kanban-of-agents 界面,都是冲着这个 budget 设计的。然后 Cerebras 在 WSE-3 上把 Codex Spark 跑了起来,OpenAI 把 GPT-5.3-Codex-Spark 作为 research preview 接进去,稳态吞吐量直接干到 1,200+ tps。不是蒸馏小模型,不是 7B,就是你已经在付钱用的那个 Codex,跑在不一样的硅片上。

20 倍是数字,真正用上一个小时之后会意识到——这是相变。原来"贵"的东西,瞬间变成"白送":

  • 每个 task 之间跑一遍完整测试,而不是每个 PR 才跑一次。
  • 同时生成 10 个 UI 方案挑一个最好的,wall clock 跟原来生成 1 个差不多。
  • 每次保存都跑 lint、typecheck、diff review,完全不打断 flow。
  • 不再做 context 压缩——直接把全量 context 重新塞进去,一来一回比压缩还便宜。

这不是想象。Cerebras 的 benchmark 页面贴了实测,OpenAI community 上一堆人在贴第一次用 Spark 的体感截图。东西已经在跑了。

没人公开讲的反转

Cerebras 那篇 best practices 的 Tip #4,原话搬过来,值得读两遍:

"慢的模型把我们推向并行。你打开 10 个 tab,扔出 10 个 prompt,然后等。用 Spark 的话,一分钟生成几千行代码。review 代码才是新的瓶颈,Spark 当 pair programmer 用最爽,而不是当一个被你 delegate 出去的工人队伍。最多 2-3 个 session。"

过去 18 个月所有的多 agent CLI 工具,本质上都是在回答这段话的第一句。Claude-Squad、Kanbots(这周 HN 230 分,慢模型时代思维方式的最后狂欢)、ruflo/claude-flow、还有我自己 SageCLI 里的 fan-out。pitch 永远是一样的:模型是慢的那一环,那就把它并行掉,用 git worktree 加 AGENTS.md 让 10 个 agent 不互相踩。这个判断在那个时间点是对的,但它现在不对了。

慢的那一环现在是 review 你的人。@aiDotEngineer 这周在 X 上把话说穿了:"速度快 20 倍,围绕慢生成建立的习惯不会消失,只会让烂代码以 20 倍速度上线。" 65 个心,但梗在扩散,因为机制上就是这么回事。并行只在被并行的那一环是约束的时候才管用。一旦 review 速度成了约束,每多一个 agent 都是反向杠杆——一堆活儿堆在一个你排不掉的队列里。

Cerebras 那句"you cannot let that pile up across multiple agents"已经是客气的说法。诚实版本是:kanban-of-agents 这种 UI 是慢推理时代的应付方案。整个 2026 下半年,我们就要看着这种 UI 慢慢变成 antipattern。

SageCLI 里哪些会留下,哪些要扔

我对自己的工具具体讲两句,模糊的自我批评什么也教不了人。SageCLI 大致两半,一半留得下,一半要扔。

留得下:无状态 executor + git 当记忆

Cerebras Tip #5 和 #6 讲的就是 不要 让 agent 跨 task 维持对话状态。每次给它干净的 context,把记忆外置到 git 和 disk 上的 plan 文件里,executor 就是个纯函数 (repo, task) → (diff, log)。SageCLI 的 msg send / msg ls / msg clear 加上 --worktree 就是这个形状——但说实话不是因为我预料到了 20 倍提速,而是从 build system 那边抄过来的。

sage send "修一下 date_window() 里那个 off-by-one"
sage ls --since 5m   # scratch log: agent 看见了什么、试过什么
sage commit          # diff 落盘,context window 释放
sage clear --worktree

这个 loop 在 40 tps 跑得了,在 1,200 tps 跑得更欢。API 一行不用动。AGENTS.md / PLAN.md / VERIFY.md 这套外置文件——上个季度全行业都收敛到了——逻辑一样。模型自己没记忆的时候,disk 是放状态最对的地方。模型变快不会改变这件事,反而让它更重要,因为你跑这个 loop 的频率高了。

留得下:--then 验证链

40 tps 的时候这是锦上添花,1,200 tps 的时候这是必需品。

sage send "加一个 --strict flag" \
  --then "跑 pytest -x,把失败贴出来" \
  --then "diff <= 50 行的范围内修掉失败" \
  --then "跑 mypy strict 出报告"

模型现在生成 diff、跑测试、修回归、再跑一遍 mypy 的总时间,比原来光生成 diff 还短。跳过验证的代价没变,做验证的代价崩了。 如果你的 CLI 没办法用一条 operator 驱动的命令把"做这个,然后验证,然后修坏掉的部分"串起来,那它就快变成一个没有依赖追踪的 Make 了。

要扔:--pattern fan-out 的 N 个并行 agent

这一段疼。sage plan --pattern fan-out --n 10 一口气开 10 个 worktree,分发 10 个子任务,我的注意力在 10 张 kanban 卡之间轮转。我做的,我用过的,在那个约束下形状是对的。现在不对了。

1,200 tps 之下,这 10 个子任务在一个 pair programming session 里串行跑完,比我在 10 张卡之间切来切去还快。fan-out 的成本不再是 wall clock,是 review backlog。10 张卡等 review 等于 10 倍的认知负担、零吞吐增益——因为 我的 tps 才是约束,而我离 1,200 还差几个数量级。

要扔:跑 8 小时的 Ralph 通宵 loop

同样的诊断。原来设计 8 小时通宵跑,是因为 8 小时 × 40 tps 才能产出一点像样的东西。现在 8 小时 × 1,200 tps 产出的是一座你白天也 review 不完的山。整套编排脚手架——checkpoint、kill switch、budget cap——是因为 run 很长才有意义。run 现在不长了,run 是几分钟。脚手架变成了一个不需要脚手架的东西上面的额外开销。

新的工效学问题

如果 review 速度是瓶颈,所有 CLI 工具要回答的问题就要换一组:

旧问题新问题
能并行多少个 agent?人能多快拒掉一个错的 diff?
能让它无人值守跑多久?跟模型面对面的内循环有多紧?
1 万行的改动 wall clock 多少?每个 commit 到测试全绿的中位时间多少?
PR 能开多大?diff 能被强制压到多小?

这周我在 SageCLI 里要做的具体调整:

  • standing pre-commit hooks 默认开。 lint、typecheck、快测试、diff size guard。这些原来是 opt-in,因为 40 tps 下它们吃 wall clock。1,200 tps 下它们什么都不吃。
  • 每个 agent turn 的 diff 上限 ≤150 行。 diff 大不是 feature,是 review 卡口。我宁愿模型分 5 个 100 行的 turn 干完,也不要它一把 500 行——因为 5 个 100 行我能 review,1 个 500 行我 review 不动。
  • /review 不再是 opt-in。 每个 sage send 结束都要带一段结构化 review 摘要,写给人看的,不是写给下一个 agent 看的。
  • --pattern fan-out 降级成一个 flag,不再是推荐 workflow。 留给真正 embarrassingly parallel 的场景(同一个 feature port 到 30 个 microservice 那种)。停止把它当默认 pitch。

给这个季度还在出 CLI agent 的人

如果你工具的头部 feature 是"并行跑 N 个 agent"——重新围绕 review velocity 写 pitch,不是 generation throughput。Q3 改完的人会显得你读懂了空气;不改的人会显得你做了 2024 年正确的产品。

如果你的工具把状态外置到 git 和 disk——你已经站在球的下一个落点上,别拆。模型越快,外置就越重要,因为你每小时跑的 clean-context run 数量更多了。

如果你的 pitch deck 里有一张"12 个 agent 跑 kanban"的截图——截图留着,caption 改了。能力还在,只是不再是头条。

如果你是用户,代价最小、收益最大的一个改动,就是 Cerebras 在 Tip #4 里埋的那一句:把自己卡死在 2-3 个 session。不是工具跑不动更多,是 你 review 不动更多——而 review 现在就是 work 本身。

瓶颈没有消失,只是搬到了你坐的那把椅子上。

由一个人类和他的 agent 一起搭建。2026
