Agent SDK 早就躺在你硬盘里了

HumanLayer 创始人 Dex Horthy 把官方 claude -p 扔了,写了个叫 Shannon 的 200 行 wrapper:tmux 里起一个 交互式 claude,然后 tail -f 那个 JSONL transcript 文件——完事。一周 196 个 star。没 SDK,没 bidi-stdio,没 streaming buffer。三个 90 年代的 Unix 工具。Twitter 上 24 小时砸了 688 个心,而那条只有 17 颗心的回复——"actually getting bidi stdio streaming is non trivial"——才是这件事真正的重点。

那条把 SDK 神话拆穿的推

周六上午,一个靠 orchestrate coding agent 吃饭的公司创始人,发了这么一条:

"hey surprise — you can just launch interactive in tmux and then tail the jsonl. shipped a small wrapper... ralph loop iterating to full parity rn."

— @dexhorthy,2026-05-14 · 24 小时 688 ❤️

紧接着的那条,心数没第一条多,但是更扎心:

"everything llm land is like that. context overflow signaling, retries, etc. it's very sad and very horrible."

— @dexhorthy,2026-05-14

这哥们儿专业搞 agent 基础设施,公开承认官方 SDK 是个税,绕过去的方法是个 bash 脚本。196 star,11 fork,几天前的事。这不是网红抖机灵,这是铸造车间内部人员的招供书。

Shannon 到底干了啥(以及它拒绝干啥)

翻 repo 看架构,反高潮到像在搞笑。整个控制流就这么几行:

# Shannon 的执行顺序,一行不落:
1. tmux new-session -d -s agent-7              # 起个 detached tmux
2. tmux send-keys -t agent-7 "claude" Enter    # 启动交互式 claude
3. tmux send-keys -t agent-7 "fix the bug in src/auth.go" Enter
4. tail -f ~/.claude/projects/<hash>/<session>.jsonl | jq .
   # ↑ 这就是它的 SDK。

这流程里没有的东西才是重点。它没有:

  • 没有 Anthropic API client。Shannon 自己一次 HTTPS 都不发。claude 自己发,跟它没关系。
  • 没用 claude -p headless。Dex 故意挑了 interactive 模式——headless 才是官方 SDK 包的那个东西,他绕开了。
  • 没有 bidi-stdio 协议。没 stream 多路复用,没半缓冲 stdout reader,没那种"模型说完了没还是还在想"的玄学判断逻辑。
  • 没有 streaming 框架。没 SSE parser,没 JSON chunk 重组,没 backpressure。

因为 CLI 已经把活儿干了。每一轮对话、每一次 tool call、每一个 Read 结果、每一条 error,Claude Code 都会作为正常运行的副作用,一行行写进 ~/.claude/projects/<hash>/<session-uuid>.jsonl。Shannon 只是个 200 行脚本,它说的话是:"我直接读硬盘上已经有的不就完了?"

你要是从来没打开过那种文件,现在就开一个看看:

你的 harness 需要的每一个字段,都已经在那儿了。在硬盘上,免费的。一次 SDK 调用都不用。

bidi-stdio 抽象的层次错了

SDK 模式在你脑子里大概长这样:

┌─────────┐    stream    ┌──────────────┐    stream    ┌─────────┐
│   你    │ ───────────→ │  Agent SDK   │ ───────────→ │ 模型 API│
│ harness │ ←─────────── │ (bidi stdio) │ ←─────────── │         │
└─────────┘              └──────────────┘              └─────────┘
                                ↑
                       重试、backpressure、framing、
                       parsing、生命周期——
                       全是你在中间这个盒子里写的。

你花一个月搭这个中间盒。debug streaming 边界情况。给半截 JSON chunk 写测试。处理 stderr 比 stdout 早 flush 的那种破事。凌晨两点的时候你终于搞明白 line-buffered 和 block-buffered 是啥区别。

Shannon 的图长这样:

┌─────────┐  send-keys  ┌────────┐
│   你    │ ──────────→ │  tmux  │
│ harness │             │  pane  │
└─────────┘             │   ↓    │
     ↑                  │ claude │ ──→ Anthropic
     │                  │   ↓    │
     │   tail -f        └───┬────┘
     └──────────────────── ─┘
                            ↓
                   ~/.claude/.../session.jsonl
                   (已结构化,已经完整,啥都不缺)

三个盒子,三个 Unix 抽象,没一个比 1991 年还年轻:

  • tmux 给你进程生命周期。spawn、attach、detach、kill、断网继续跑。已解决问题。
  • tail -f 给你 streaming。append-only file + inotify = 计算机里最干净的事件流。已解决问题。
  • jq 给你 parsing。结构化 JSON 进,你想要啥出。已解决问题。

所谓"agent SDK",是你忘了你的电脑是台 Unix 机器之后才会需要的东西。OS 就是 SDK。agent CLI 就是个长跑的进程,顺便把日志写成结构化的。你 harness 跟 agent 之间的"协议",就是一个文件路径加一个 inotify watch。

说实话,这就是我从第一天写 SageCLI 就押的注,当时被各种"你应该用 SDK"的反馈淹过。SageCLI 的 send / ls / clear 全是 bash + tmux + JSONL,没了。没框架,没 SDK,没 streaming 库。原因不是我多聪明——是我懒。CLI 已经把我要的全干完了,再用 Python wrapper 把它再实现一遍这事明显不对。Shannon 只是从一个完全独立的方向、一个声量大得多的舞台,把同一个直觉验证了一遍。

那 2026 年的"agent SDK"会变成什么

随便挑一个今年发布的 coding agent,在它源码里 grep 一下 session,你会看到这样的路径:

~/.claude/projects/<hash>/<uuid>.jsonl       # Claude Code
~/.codex/sessions/<date>/<uuid>.jsonl        # OpenAI Codex CLI
~/.cursor/transcripts/<workspace>/*.jsonl    # Cursor
~/.aider/.aider.chat.history.md              # Aider (是 markdown,也能 parse)
~/.opencode/sessions/*.jsonl                 # OpenCode
~/.kiro/.../sessions/*.json                  # Kiro

每一个都是 agent 的 transcript,作为正常运行的副产品被写出来。每一个都是结构化的。大部分是 JSONL。读它们没一个需要 SDK。那个你被告知必须学的协议——bidi-stdio framing、MCP transport、ACP envelope——其实是在你 transcript 已经免费存在的那一层之上一两层。

这倒不是在打 ACP 或 MCP 的脸。它们解决的是真问题:把编辑器和 agent 之间的协议标准化,这样 JetBrains 不用为 22 个 CLI 写 22 套接入。这是 LSP 那一层,N×M 复杂度的解药——花协议成本是值的。

但是你自己的 harness——你的 cron 任务、你的后台 worker、你的 CI bot、你的自定义 orchestration 循环——你不是 N×M。你是 1×1。你需要的不是协议,是一个文件路径。正经对比一下:

层SDK / bidi-stdio文件系统(Shannon, SageCLI)
启动library APItmux new-session
送入输入stream 写入tmux send-keys
读取输出stream parser、framing、重试tail -f session.jsonl
持久化你自己实现已经在硬盘上
崩溃后恢复你自己实现cat 那个文件
审计日志你自己实现git add 那个目录
多 agent 并发线程池 + 锁N 个 tmux pane,N 个文件
到第一次能 ship 的时间几周200 行 bash

对未来 12 个月我的判断是:每个还能起来的"agent SDK"会悄悄变成 on-disk transcript 上面那层薄薄的礼貌封装。Anthropic 自己的 SDK 不会再被推荐成 drive Claude Code 的标准方法。厂商会发transcript schema,而不是 SDK client。能跑出来的 wrapper——Shannon、SageCLI、还有十几个我们还没见到的——会在用 JSONL 干啥这件事上竞争,而不是怎么把 JSONL 拿到。

你一直在等的那个协议早就有了。它叫文件系统。1969 年就发布了。uptime 一直很好。

一句话总结

如果你的 agent harness 里有个"client" 模块,删了它,把 tail -f 指向 CLI 已经在写的那个 JSONL,然后把你本来要拿去 debug stdio framing 的那一周时间,做成用户能真正用上的功能。

由一个人和他的 agent 共同建造。2026
