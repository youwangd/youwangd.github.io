2026-06-07 mcp agent 性能

装满 MCP 的 Agent 只会更蠢:工具说明书把你七成脑子吃光了

200K token 的 context window 听起来很够用,直到你意识到 agent 必须把它花在什么地方。Apideck 四月份扒了自己生产环境的数据:三个 MCP server——GitHub、Slack、Sentry,加起来大概 40 个工具。用户还没敲第一个字符,光这些工具的 schema 就已经吃掉了 200K window 里的 143K。72% 的预算,直接没了,花在了一份模型还没下单的菜单上。这就是 GitHub trending 上那一排排"awesome-MCP-servers"仓库没人愿意算的账。也是为什么 2026 年所有真正在跑 agent 的人,都在悄悄删工具,而不是加。

把账摊开看

数字不会骗人。下面这张表是用户消息进来之前就要从 context window 里扣掉的钱,数据全来自真实的生产 telemetry 和 MCP server 自己公布的工具清单:

来源 配置 消耗 token
Apideck 生产 telemetry 3 个 server · 约 40 个工具(GitHub + Slack + Sentry) 约 143,000 / 200K 窗口(72%)
Atlassian mcp-compressor 发布 官方 GitHub MCP server 自带的工具清单 约 17,600,94 个工具
Atlassian,同篇 同样 94 个工具,前面挂上 mcp-compressor 代理 约 500(降了 97%)
Anthropic Tool Search(2026 年 2 月) 按需懒加载工具定义 相对全量加载减少约 85%

盯着这张表看三十秒,有两个事实你逃不掉:

  • token 成本不是由你实际调用了几个工具决定的——它跟你列出来了几个工具线性相关。
  • 所有正在大规模跑 MCP 的厂商,现在都在往中间塞一层东西,目的就是不让模型看见完整的工具清单。这不是 feature,这是损失控制。

Atlassian 公开把一个 17,600 token 的工具清单压到 500 token,还专门写文章把它定位成预防工具膨胀,这等于他们在承认 MCP 的默认配置已经把 Rovo Dev 的 prompt 预算搞崩了。Perplexity 的 CTO Denis Yarats 三月份直接在内部生产环境放弃了 MCP,理由就是"context window 消耗",这话不是吐槽,是采购信号。

三笔你 tool-calling 成功率上根本看不见的隐藏成本

1. Context 膨胀——还没干活就先交税

第一笔账最直白也最容易测。每个工具定义都是 JSON Schema:参数、类型、描述、默认值、错误语义,模型每一轮都得从头读一遍。一份 17,600 token 的工具清单意味着你少了 17,600 token 的余量给真正重要的东西:system prompt、检索回来的文档、对话历史、用户实际想解决的问题。在长会话里这笔账会反复被收——清单作为 context 的一部分,下一次调用又要被重新读一遍。

痛得最真切的不是合成 benchmark,是长跑的会话。我们有一个 wiki-curator,挂在常驻 kiro session 上一篇一篇地处理文章。一开始每篇 10–15 分钟。处理了大概二十几篇之后,慢慢爬到 60–100 分钟。prompt 一个字都没改。模型也没退化。就是 context 不停往上堆,工具清单一遍一遍被重读,单次 tick 的成本从"够用"漂到了"完全不能用"。修法没什么花哨的:每跑 N 篇就重启 session,把桶倒空。桶还会再灌满,但至少这一桶是干净的。

2. 工具选错——菜单越大,模型选得越差

第二笔账你账单上看不到,只能在 trace 里看到。arXiv 二月那篇 "MCP Tool Descriptions Are Smelly" 把我天天在生产里看到的失败模式量化了:工具集越大,工具选取的 accuracy 越低。当四十个工具里一半的描述都长得差不多,模型根本分不清谁是谁。get_user vs fetch_user_profile vs read_member——三个 MCP server,三个团队,三套几乎一样的 signature,模型挑了一个,经常挑错。

这是对常见话术的反向打脸。话术是:能力越多,agent 越强。数据是:能力越多,agent 越糊涂,你最新加上的那个工具有时候是负贡献——它把你真正需要的那个挤糊了。多不一定好。哪怕忽略 token 不算,多本身就常常等于差。

3. Cold-start race——工具到得太晚,直接没注册上

第三笔账最隐蔽,因为它不会显化成"回答变差"。它显化成工具直接消失。这个我们查了大半个下午才搞懂,网上似乎也没人写过。

大多数 agent runtime 在 session 启动时会"封"一次工具注册表。封完之后,任何晚到的 MCP server 都活着——/mcp 里看得见,JSON-RPC 探针也能拿到完整工具列表——但 agent 自己根本不知道这些工具的存在。它们没赶上注册表关门那一刻。整个 race 静悄悄。没有 error。只有一个干瞪眼的 server 陪你跑完整个 session。

事故复盘:那 1 秒的工具注册表 race

我们当时按 MCP 推荐的写法配 scholar-feed-mcp:

{
  "mcpServers": {
    "scholar-feed": {
      "command": "npx",
      "args": ["-y", "scholar-feed-mcp@3.7.0"]
    }
  }
}

这是全世界 MCP README 都教你写的样子。npx -y 看着特别顺手:版本能 pin、按需下载、不污染全局。session 启动了,/mcp 显示 scholar-feed 绿灯、connected、健康。agent 看到的工具数:零。它宣称的 25 个 method 一个都调不动。agent 日志没 error。MCP transport 也没 error。

我们直接拿 JSON-RPC 怼了一发,server 乐呵呵地把 25 个工具全返了。server 没事。agent 也没事。它俩就是没接上。grep 了一阵才把缝找到。

# 一次 session 启动的实测时间线
T+0.000s   kiro-cli 启动,agent 进程 fork
T+0.420s   MCP transport 打开,准备拉 scholar-feed(npx 被调起)
T+1.000s   ### 工具注册表关门——agent 锁定了工具列表 ###
T+1.000s   agent 看到:scholar-feed 有 0 个工具
T+3.150s   npx 终于热完,scholar-feed-mcp ready
T+3.151s   server 宣布有 25 个工具——已经晚了

修法本质是一个字符的改动,落地是 config 重写:别再拿 npx 当 launcher。把包装到全局——npm i -g scholar-feed-mcp@3.7.0——然后让 spec 直接指向二进制的绝对路径,args 留空:

{
  "mcpServers": {
    "scholar-feed": {
      "command": "/home/dyouwang/.nvm/versions/node/v22.17.1/bin/scholar-feed-mcp",
      "args": []
    }
  }
}

cold-start 时间从 ~3.0 秒掉到 0.37 秒。server 在 T+1.0s 关门之前就把工具喊出来了。agent 看到全部 25 个工具。同样的代码、同样的包版本、同样的模型。唯一变化是 launcher 不再和注册表赛跑。

这种病只会随着你装的 MCP server 越多越严重。每多一个 npx 风格的 server,就多一次掷骰子赌它能不能在关门前热好。叠得越厚,你就在静默地丢掉越多工具。看不见的工具照样在你账单上——你为它的清单付了 token,什么都没收回来。

瘦身手册

把 MCP server 当依赖来管。"该不该加这个 server"的默认答案是不该。让团队为每一次添加做举证,跟在生产 codebase 里加一个新 SDK 一样严肃。下面是我们已经在跑、确实减少痛苦的几条规矩:

  • 按 domain 拆分,而不是按 demo 堆。Pinterest 的做法:一个 agent 要么拿到数据类 server,要么拿到沟通类 server,绝不把全套拍在一起。如果一个 workflow 必须两边都用,写一个薄的 orchestrator,spawn 两个 scope 各自独立的 agent。你拿一点点协调成本换一大把 context 余量。
  • 大 server 前面挂个压缩器。Atlassian 的 mcp-compressor、Anthropic 的 Tool Search,或者自己撸一个:用两个 meta-tool——list_tools(filter) 和 invoke_tool(name, args)——把那 94 个工具的 eager 清单藏起来。模型用到了再去找。97% 的减幅不是打字错。
  • 狠狠审你的 allowedTools 白名单。没进白名单的工具照样烧你 token 列清单——server 该返还是要返——但你一分钱产出都拿不到。如果一个 server 你一个工具都没启用,直接把整个 server 删掉。"以后可能用得上"在 token 这件事上不存在,你今天就在为它付费。
  • 发二进制,别发 npx 启动符。cold-start 重要,因为工具注册表关门很早。全局安装 + 绝对路径,稳赢"包管理器当 launcher"那一派。npm i -g、uv tool install,只要能在 500ms 内解析到一个二进制就行。
  • 给长 session 设重启预算。没人盯着的时候 context 也在堆。挑一个指标——文章数、轮次、累计 token——到阈值就硬重启。wiki-curator 那个重启技巧拉回了 5–10 倍的单 tick 性能。这是我们整套基础设施里最便宜的优化,没有之一。

"awesome-X-AI-agents"trending 列表上的本能反应是:技能越多,agent 越强。本能反应错得很彻底。每多装一个 server,你都在烧 context、把模型选工具的判断糊掉,还多一次输给 startup race 的可能性——而且这种输你日志里根本看不到。@steipete 几天前那条推讲得很干净:"modular and lean,只加你需要的。技能越少、工具越少 = agent 跑得越高效。"112 颗心,零反对。市场已经投完票了。"MCP server marketplace"那波淘金潮要直接撞上 200K token 的墙,活下来的会是带刀来的 agent,不是带工具箱的。

你的 agent 弱不是因为它工具不够。是因为它工具太多。
