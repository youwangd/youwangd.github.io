你的 AI 智能体正在烧钱：Token 优化的三层策略

Token 成本是 AI 智能体的隐性税。每次 git diff、每条错误日志、每个文件读取，都在往你的账单里塞垃圾。三层压缩策略可以让你在不损失质量的情况下省下 40-70% 的钱。

痛点：Token 账单的真相

很多团队只盯着 prompt 优化，却忽略了真正的 Token 杀手：工具输出。

看看这个数字。一个重度使用工具的 session，工具输出能占到 40% 的 Token 量。一次 git diff 能产生上千个 Token。一个带调用栈的错误日志，几百个 Token。这些东西每轮对话都要重新处理，成本在不断累积。

更要命的是，这些输出大部分都是噪音：

  • 重复的样板文字
  • 无意义的空行和格式
  • 冗长的时间戳
  • 相同的错误信息
  • 巨型的 JSON 响应

智能体需要的是信号，不是噪音。但现在我们把噪音全塞进上下文窗口，让模型去筛。这就像让 CEO 逐字读垃圾邮件。

第一层：源头压缩

最有效的压缩是在工具输出进入上下文窗口之前就动手。几个立竿见影的工具：

RTK 专门用来压缩命令行输出。看看这个效果：

从 200 Token 压缩到 10 个。20x 的压缩比。

Laconic：Markdown 瘦身

Laconic 专门清理 Markdown 文档：

  • 去掉无用的 badge
  • HTML 表格转 CSV 格式
  • 规范化空白字符
  • 压缩重复的分隔线

结构化压缩

对于日志和结构化输出，你可以用更激进的策略：

智能截断

对于超长输出，保留头部 70% + 尾部 20% 的内容。为什么？错误信息通常在末尾，而上下文在开头。中间的重复信息价值最低。

第二层：历史墓碑

随着对话进行，旧的工具结果价值在递减。「墓碑策略」的名字来自操作系统——在删除文件时留下一个小标记，表示「这里曾经有什么」。

我们对智能体记忆也用同样的思路：

  • 0-2 轮：保留完整结果
  • 3-5 轮：压缩为摘要预览
  • 6+ 轮：最小墓碑 [工具已调用: git status - 5 files changed]

注意：压缩时必须保留 toolResult 的 API 结构，否则模型 API 验证会报错。你需要在保持结构完整的前提下缩短内容。

# 完整版（0-2 轮）
{
  "toolResult": {
    "tool": "git_status",
    "output": "On branch main\nYour branch is up to date...",
    "status": "success"
  }
}

# 墓碑版（6+ 轮）
{
  "toolResult": {
    "tool": "git_status", 
    "output": "[墓碑: 5 files changed, clean working tree]",
    "status": "success"
  }
}

第三层：分级模型

不是所有任务都需要 Claude Sonnet 或 GPT-4 的智商。关键洞察：后台任务不需要前沿智能。

任务分层

  • 对话 + 工具调度 → 贵的模型（Sonnet, GPT-4）
  • 记忆提取、摘要生成 → 便宜的模型（Haiku, GPT-3.5, Flash）

成本差异有多大？Haiku 比 Sonnet 便宜 12 倍。Flash 比 GPT-4 便宜 15 倍。用便宜模型处理压缩和摘要，成本立刻下降一个数量级。

# 架构示例
primary_agent = ClaudeAgent(model="sonnet")  # 主对话
memory_compressor = ClaudeAgent(model="haiku")  # 后台压缩

# 主智能体处理用户请求
response = primary_agent.process(user_input)

# 便宜模型在后台清理历史
async_task: memory_compressor.compress_history()

三层叠加效果

每一层都能带来显著节省，但真正的魔法在于叠加：

原始成本：     10,000 Token × $0.01 = $0.10
源头压缩：      4,000 Token × $0.01 = $0.04 (60% 节省)
墓碑策略：        200 Token × $0.01 = $0.002 (98% 节省)
分级模型：        200 Token × $0.001 = $0.0002 (99.8% 节省)

综合节省：70-90% 的 Token 成本降低。

在生产环境，这意味着：

  • 月度 API 账单从 $1000 降到 $200
  • 相同预算下能支撑 5x 的用户量
  • 更快的响应速度（更少 Token 意味着更少延迟）

踩坑提醒

这些策略不是银弹。诚实地说几个坑：

工具兼容性有限

很多代理压缩工具（比如 Kompact）只支持特定的 API 格式。RTK 目前只压缩 bash 输出。要针对你的工具栈定制压缩逻辑。

有损压缩的风险

过度压缩会丢失智能体需要的关键上下文。症状：智能体开始问一些之前已经回答过的问题，或者遗漏重要细节。

缓解方法：保守起步，逐步调激进程度。监控智能体性能指标。

压缩放错位置

在错误的地方做有损压缩会毁掉准确率。错误日志、代码 diff、API 响应——这些通常需要保持完整性。先压缩样板文字和格式，再考虑语义压缩。

调试变难

当智能体行为异常时，追踪压缩后的上下文会比较困难。建议保留原始日志的副本，用于事后调试。

开始行动

Token 优化不是一次性的工作，是持续的系统工程。建议的实施顺序：

  1. 测量现状：统计你的智能体 Token 消耗模式。工具输出占多少比例？
  2. 低风险压缩：从 RTK、Laconic 这种成熟工具开始。先压缩样板内容。
  3. 墓碑机制：实现渐进式历史压缩。小心保持 API 结构完整性。
  4. 分级调度：把后台任务切换到便宜模型。监控质量指标。
  5. 持续优化：定期审查压缩效果，调整策略。

你的智能体账单里有 40% 是噪音。三层策略能帮你把这些噪音变成节省的成本。在这个 AI 烧钱的时代，这不是可选项——是必选项。
