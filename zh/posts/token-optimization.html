你的 AI 智能体正在烧钱：Token 优化的三层策略

跑了几个月智能体之后，我发现一个反直觉的事实：真正吃掉你 API 预算的不是你精心调教的 prompt，而是那些你根本没在意的工具输出。一坨 git diff，一页搜索结果，一份报错日志——这些东西默默地把你的上下文窗口塞满，然后每一轮对话都跟着你走，轮轮收费。

钱都花哪儿了？

先看一组数据。拆开一个典型的、工具用得比较狠的智能体 session，Token 的去向大概是这样的：

  • 工具输出 ~40% —— git diff、文件内容、搜索结果、API 响应，都算这里
  • 对话历史 ~30% —— 之前聊过的内容，每轮都要带上
  • 系统提示 + 上下文 ~20% —— 你精心写的 system prompt 和各种注入的背景信息
  • 当前请求 + 格式开销 ~10% —— 用户这轮说了啥，加上 JSON 的各种花括号

看到了吗？大头是工具输出，而且这 40% 基本全是噪音。你的智能体跑了一个 npm test，输出 500 行，真正有用的就 3 行——哪个 test 挂了。它抓了一个网页回来，50KB 的正文，答案在第一段。

但这些垃圾不会自己消失。它们会老老实实待在上下文窗口里，每一轮对话都跟着，轮轮付费。一次搜索结果 5000 Token，后面再聊 10 轮，这就是 5 万 Token 的成本——而那个搜索结果可能第 3 轮就没人看了。

大部分团队把精力花在打磨 prompt 上，从 200 Token 抠到 150 Token，节省了 25%——的那 20%。与此同时，工具输出那 40% 的领地完全放养。这就像装修房子时认真挑窗帘颜色，但漏水的屋顶不管。

第一招：出口压缩——垃圾别进门

最划算的 Token 是根本不进入上下文的 Token。在工具输出返回的那一刻就拦截它、压缩它，而不是等它进了对话再想办法。

命令行输出：用 RTK

RTK 是个 Rust 写的小工具，专门吃命令行输出然后吐出精华。来感受一下：

# 原始输出
$ git push origin main
Enumerating objects: 15, done.
Counting objects: 100% (15/15), done.
Delta compression using up to 8 threads
Compressing objects: 100% (8/8), done.
Writing objects: 100% (9/9), 1.23 KiB | 1.23 MiB/s, done.
Total 9 (delta 4), reused 0 (delta 0)
remote: Resolving deltas: 100% (4/4), completed with 2 local objects.
To github.com:user/repo.git
   abc1234..def5678  main -> main

# 过一遍 RTK
✓ Pushed 9 commits to main (1.23 KiB)

200 Token 变 10 个。你的智能体只需要知道「推成功了」，不需要知道 delta compression 用了几个线程。

文档和网页：用 Laconic

Laconic 干的事是给 Markdown 和 HTML 瘦身。那些 GitHub README 里的 badge 图标、花哨的表格、重复的分隔线——全是给人眼看的装饰，对模型来说纯粹是噪音。Laconic 能做到 40-68% 的无损压缩，表格转 CSV，badge 去掉，空白归一化。

通用套路：折叠重复、砍时间戳、保头保尾

对于 RTK 和 Laconic 覆盖不到的场景，几个通用规则就能搞定大部分情况：

# 日志里 100 行一模一样的 INFO？折叠
[10:23:45-10:24:30] Processed 94/100 files [INFO x94]
ERROR: file 95 - Permission denied

# 截断？别只保留头部
# 保头 70% + 保尾 20%（报错信息通常在最后！）
# 中间塞一句 [... 省略 2847 tokens ...]

# 同一个命令跑了三遍？直接引用
[同上次 ls /home/user 的结果，15 个文件]

最后那个「去重」别小看——有些智能体真的会用完全相同的参数反复调同一个工具。返回一个引用就好，别让模型吃第二遍。

第二招：对话瘦身——让旧信息慢慢「退场」

源头压缩解决了「进门」的问题，但对话越聊越长怎么办？5 轮前的 git diff 结果，到第 10 轮大概率没人 care 了，但它还在那儿占着坑位。

思路很简单：信息的价值会随时间衰减，压缩力度应该跟着走。

  • 刚出炉（0-2 轮）：完整保留，一个字不动
  • 有点凉了（3-5 轮）：压缩成摘要，保留关键信息——「npm test 跑了 47 个用例，2 个挂在 auth 模块」
  • 彻底凉了（6 轮以后）：只留一行记录——「[执行过: npm test → 2 failures]」

这招在行业里有时候被叫做「tombstone」——像墓碑一样，原来的内容已经「入土」了，但留个碑告诉你这里发生过什么。智能体回头翻的时候还能知道上下文，但不会被一大坨过期内容撑到。

实操时有个坑必须注意：很多模型 API（比如 Bedrock Converse、Azure OpenAI）会严格校验 tool result 的 JSON 结构。你不能把 toolResult 直接替换成一段纯文本，那样 API 会报错。得在保持结构完整的前提下缩短 content：

// 第 1 轮：完整结果
{
  "toolResult": {
    "toolUseId": "toolu_abc",
    "content": "FAIL src/auth.test.js\n  ● login flow\n    ✕ should validate token (24ms)\n      Expected: valid\n      Received: expired\n\n  47 tests, 2 failures",
    "isError": false
  }
}

// 第 7 轮：一行够了
{
  "toolResult": {
    "toolUseId": "toolu_abc",
    "content": "[压缩] npm test: 47 tests, 2 failures in auth",
    "isError": false
  }
}

结构没变，内容从 1200 Token 变成 20 个。模型 API 不会叫，智能体也不会失忆。

第三招：杀鸡别用牛刀

到这一步，该压的都压了，该瘦的都瘦了。但还有一个维度被大部分人忽略了：不是所有活儿都值得请最贵的模型来干。

想想看，一个智能体 session 里其实有两类工作：

  • 需要脑子的——理解用户意图、决定调哪个工具、组织回复、处理复杂逻辑。这些活儿交给 Sonnet、GPT-4 这种大脑袋，没毛病。
  • 纯体力活——把一大段工具输出压缩成摘要、从对话里提取值得记住的信息、生成 tombstone。这些任务模式固定、不需要创造力，用 Haiku 或者 Flash 绰绰有余。

成本差多少？Haiku 大概是 Sonnet 价格的 1/12，Flash 是 GPT-4 的 1/15。把后台任务全切到便宜模型上，光这一步就能省下一个数量级。

# 伪代码，意思到了就行
main_model = "sonnet"      # 处理对话、做决策
background_model = "haiku"  # 压缩、提取、打杂

# 用户问了个问题 → 大模型出马
response = call_model(main_model, user_input + context)

# 后台默默干活 → 小模型就够
summary = call_model(background_model, "把这段工具输出压缩成一句话: " + tool_output)
memories = call_model(background_model, "从对话里提取值得记住的事实: " + recent_exchanges)

这个思路 Alma 团队也在用，他们管它叫「Tool Model」模式——控制面（决策）用贵的，数据面（处理）用便宜的。和微服务架构里分离 control plane 和 data plane 是一个道理。

三招叠加的效果

这三层不是三选一，是叠加的。来算一笔账：

场景：智能体调了一个工具，原始输出 10,000 Token

第一招（出口压缩）：10,000 → 4,000 Token    省了 60%
第二招（5 轮后瘦身）：4,000 → 200 Token      省了 95%（相对原始）
第三招（后台用便宜模型）：处理成本再降 12x

一个 session 20 轮，重度工具使用：
  没优化：~50,000 Token，全用贵模型
  三层拉满：~15,000 Token，60% 跑在便宜模型上
  综合省下：70-85% 的账单

月底看 API 账单的时候，你会感谢自己的。

别太猛——几个踩过的坑

说完好处，聊聊现实。这些策略不是无脑上的，有几个坑值得注意：

RTK 只管命令行。搜索结果、浏览器输出、MCP 工具返回的内容，它管不着。你得针对不同类型的工具输出分别处理，没有银弹。

代理式压缩工具水土不服。市面上有些工具（Kompact、Janus 之类）是以代理的方式拦截 API 请求做压缩的。如果你用的是 OpenAI 的标准 API，它们能用；但如果你走的是 Bedrock、Azure、或者任何自建端点，这类工具直接废了。压缩逻辑得内建到你的智能体架构里，而不是外挂。

过度压缩比不压缩更糟。有一种经典翻车：第 2 轮的工具结果在第 3 轮就被压成了一行，结果第 4 轮用户问「刚才那个报错的具体信息是什么？」——智能体答不上来。衰减速度要调，宁可保守一点，不要激进。症状很好辨认：如果智能体开始重复问已经回答过的问题，多半是压过头了。

有些东西不能压。错误日志的 stack trace、代码 diff 的具体改动、API 返回的结构化数据——这些压了就废了。规则很简单：先压格式和样板，语义内容留到最后再动。

调试会变难。出了 bug 你想回溯上下文，看到的全是压缩后的摘要，查不到原始数据就抓瞎了。建议：原始输出单独存一份日志，压缩后的进上下文。出问题翻原始日志就行。

怎么开始

不用一步到位，从性价比最高的地方切入：

  1. 先搞清楚你的钱花在哪。给工具输出加个 Token 计数器，按工具类型统计。大概率你会发现某一两个工具是大头。
  2. 装上 RTK 和 Laconic。两个都是即插即用的，前者管命令行，后者管文档。花 10 分钟配好，立刻见效。
  3. 加上头尾截断。超过一定长度的输出，保留头部 70% 和尾部 20%，中间省略。这是最安全的有损压缩。
  4. 再上对话瘦身。实现渐进式压缩，注意保持 API 结构。
  5. 最后考虑分级模型。把后台任务切到 Haiku/Flash，主对话保持旗舰模型。

光前三步就能干掉 40-60% 的工具相关 Token 成本。后面两步是进阶优化，等你跑起来再说也不迟。

Token 这东西很有意思——你不数不知道，一数吓一跳。但好消息是，浪费越多，优化空间越大。先量，再压，然后看着账单一点点掉下来。
