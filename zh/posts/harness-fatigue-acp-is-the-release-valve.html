工具一堆，没一个我想用：ACP 才是救命稻草

周六早上 r/LocalLLaMA 上一个帖子,标题就四个字 "I am overwhelmed by Harnesses"——我被这堆 harness 搞崩了。开头楼主列了 19 个他装过的 coding agent CLI,3 个还在付费,真正每天用的那个他不肯说。两个小时上首页。底下回复才是关键:大家不是被选择压垮的,是被耦合捆死的。每一个 harness 都是一坨没有 wire format 的黑盒,接管了你的 session、permission、MCP 配置——还有,你接下来一年的工具决策。ACP 是第一个真正能把这些东西拧开的扳手。

到底是哪种累

把楼主电脑上装的列出来,因为这就是现在每个人电脑上的状态:claude-code、opencode、codex、gemini、goose(现在叫 aaif-goose)、cline、aider、continue、kilo、roo、junie、cagent、fast-agent、kimi、qwen-code、vt-code、openhands、crystal(现在叫 Nimbalyst),再加上他们组上 sprint 自己撸的那个。这里面 6 个在过去 60 天内改名了。crystal → Nimbalyst。block/goose → aaif-goose。ruvnet/claude-flow → ruflo。steveyegge/gastown → gastownhall。GitHub 上现在有个 awesome-cli-coding-agents 的 curated 列表,311 star 67 fork。已经有目录的目录了。

大家(包括半年前的我)的本能反应都是"挑一个,死磕到底"。这建议是错的——而且为什么是错的,正是这篇文章的全部重点。

你"挑一个 harness"的时候,到底挑了什么

你以为你挑的是一个二进制。你实际挑的,按"想换的时候有多痛"排序,大概是这些:

  1. session 模型。父 session 的权限会不会传给子 session、external_dir 黑名单会不会继承、崩了之后 resume 还能不能拿回上下文。
  2. MCP 配置布局——更糟糕的是,MCP 错误处理风格。有的 OAuth 失败直接吞掉,有的无限重试,有的把上游报错原样吐出来。三种风格三种 debug 体验。
  3. 权限审批 UX。Tab 同意?敲 y?点弹窗?approve-once vs approve-forever vs session-scoped——每家选的都不一样,肌肉记忆是带不走的。
  4. tool schema 清洗逻辑。某些模型(Kimi、Moonshot)拒绝带 oneOf 或者递归 ref 的 schema。每家 harness 自己手撸一套 normalize,而且 normalize 得不一样。
  5. workspace 路径约定。绝对路径还是相对路径、symlink resolve 不 resolve、repo 挪窝以后会怎样。

这五条没有一条是模型的选择。没有一条本来该是不可逆的。但今天每一条都是不可逆的——因为 agent 和编辑器/shell/IDE 之间根本没有协议,harness 本身就是协议,所以协议 bug 都以 patch release 的方式发出来。

OpenCode 的 changelog 就是物证

给 SageCLI 写 weekly intel 的时候我一直开着 OpenCode 的 release feed 这个窗口。看一眼 6 天里发的版本,v1.14.27 → v1.14.33,2026-04-27 到 2026-05-02:

版本第几天修了啥本质是啥
1.14.28+1子 session 保留 external_dir deny权限继承 bug
1.14.29+2规整 MCP OAuth 错误错误形状不一致
1.14.30+3相对 workspace 路径可移植路径约定 bug
1.14.31+4远程 MCP URL 错误包装错误形状不一致
1.14.32+5给 Kimi 清洗 tool schemaschema 规整
1.14.33+6给 Moonshot 清洗 tool schemaschema 规整(又来一遍)

6 天 6 个版本。每一条都是该住在协议层、整个 ecosystem 一次性修掉的东西。OpenCode 还算工程做得好的——他们在修;别家是同样的 bug 不修。而推上现在正在宣发的n+1个新 harness,马上就要把这 6 个 bug 从零再写一遍。

这就是 Reddit 那条帖子说不出口、但所有人都感受到的累。累不是因为用 harness 累,是因为眼睁睁看着 20 个团队各自重新发明一遍权限弹窗 UX、MCP 重试策略、tool schema 清洗、session resume——然后用户用升级 churn 的方式买单。

ACP 就是 LSP 时刻

2016 年微软和红帽搞出 LSP。LSP 之前每个编辑器自己写每种语言的插件:Python-mode、Java-mode、Go-mode,各自实现 find-references、自动补全、hover、跳定义。Language server 一升级插件就坏。编辑器开始专科化——VSCode 服务 TS、IntelliJ 服务 Java、Vim 党"我学一下 ctags 算了"。听着耳熟吗?

LSP 把矩阵拍扁了。O(编辑器 × 语言) 变成 O(编辑器) + O(语言)。今天每个像样的编辑器都说 LSP,每个语言都发 server,"哪个编辑器配哪门语言"基本只剩口味问题,不再是能力问题。

ACP——Agent Client Protocol——就是 coding agent 版的同一招:

  • 2025 年 8 月: Gemini CLI ↔ Zed 第一次跑通,两方 JSON-RPC,验证回路。
  • 2025 年 10 月: JetBrains 和 Zed 宣布 open collaboration。这一刻它就不再只是 Zed 的玩具了。
  • 2026 年 5 月: 公开的 Zed ACP registry 上挂了 22 个 agent(AgentPool、Augment、Blackbox、Claude、Cline、Codex、cagent、fast-agent、Gemini CLI、Copilot、Goose、Junie、Kimi CLI、Minion、Mistral Vibe、OpenCode、OpenHands、Pi、Qoder、Qwen Code、Stakpak、VT Code)和 12 个编辑器(Zed、JetBrains、AionUi、Emacs、marimo、Neovim、Sidequery、aizen、DeepChat、Tidewave、Web AI SDK、Obsidian)。

The Register 2026-10-07 那篇标题挺到位:"JetBrains 力挺一项可能蚕食 VS Code 霸权的开放 AI 编程标准"。JetBrains 之于 ACP,就像红帽之于 LSP——一旦一家非微软的主流 IDE 站台了,网络效应就开始倾斜。

两边都说 ACP 之后,白送你的东西:

  • 权限委托。 协议自带"agent 想跑 X 你同不同意"的 handshake。编辑器爱怎么渲染弹窗就怎么渲染。再也不用记 harness 自家那套 approve-once-vs-forever。
  • session 可移植。 Session 是协议对象,有稳定 ID。任何 client 都能 resume。任务跑一半从 Zed 切到 Neovim。Harness 可以崩,session 在编辑器里活着。
  • tool 输出流式渲染。 OpenCode 反复打补丁的那个 bug——tool 输出片段渲染——是协议原语。修一次,所有人都好。
  • schema 在边缘 normalize。 Agent 用已知 schema 声明 tool。编辑器可以居中调度。Kimi 和 Claude 的 schema 怪癖变成一次性 adapter,而不是每个 harness 跑一遍。

这才是"LSP 时刻"在工程上的意思。不是"大家口头上同意了一个 buzzword",而是每个 harness 必须自己扛的表面积塌方,导致切换成本塌方,导致 harness 这一层商品化,导致你终于可以"想用哪个模型挑哪个 agent、想在哪个编辑器里就在哪个里",矩阵不再粘合。

SageCLI 这边怎么看

认账:SageCLI 一开始我就是 bash + jq + tmux 拼起来,刻意不上 framework,就是因为 2025 年我们已经看到 harness fatigue 的悬崖,不想跟谁结婚。这决定 2025 年看着像犟,现在看着像运气好。我们押了两个 bet:(a) wire format 一定会到,(b) 价值住在 harness 周围的 corpus 里,不在 harness 二进制里。两个都在兑现。

具体来讲,我们把"可移植(值得投,换 harness 不会丢)"和"会蒸发(别耦合)"分得很清:

层可移植?原因
AGENTS.md + 项目规范是。 去年统一文件名就是为了这一刻。每个说 ACP 的 agent 都读它。
MCP server 清单是。 MCP 之于 tool 这一侧,就像 ACP 之于编辑器一侧。同一组 server,换 harness,tool 还在。
Slash command / prompt 模板大半可以。 触发语法是 harness 的,内容是你的。prompt 是你的,trigger 是它的。
session 模型 / 权限 UX不可以。 别投资,别背 harness 的怪癖。ACP 会吃掉这一层,正在吃。
内置工具(web fetch、文件编辑等)不可以。 改用 MCP server,工具就是你的,不是 harness 的。"内置"是穿了便利外衣的锁定。

关键是认清你有两类投资面:协议形状的(AGENTS.md、MCP、prompt)和厂商形状的(其他一切)。在协议形状的那边花钱,拒绝给厂商形状的那边交学费。下一波改名落地——我赌 30 天内必有下一波——你应该毫无感觉。换二进制,留 corpus。

不会说 ACP 的那些 agent 怎么办

顺带说:就算你最爱的 agent 暂时不会说 ACP,registry 里还有 Pi 这个 ACP-bridge,能把不会 ACP 的 CLI 拉进网络。和 2017 年 LSP-vs-LSIF 的 adapter 同一个把戏。你不用等所有厂商,你只需要每家有一座桥就够了。"我必须挑一个"这个框架预设了一个封闭集,实际情况是每个 agent 离"可互换"只差一个 ACP shim。

那这个楼主到底该听啥

那条帖子下面一堆"挑 Claude Code 就完了"和"Cursor 就够用"。两个建议都是错的——错法跟 2015 年"挑 VSCode 就完了"一模一样。正确的回答是:

  1. 别再把"选 harness"当成单向门。wire format 的全部意义就是可逆。
  2. 优先选会说 ACP 的 harness,或者离一座桥的 harness。看 Zed registry,那现在就是事实上的兼容列表。
  3. 把投入挪到 AGENTS.md、MCP server、prompt 模板。这三个能熬过接下来 4 轮改名。
  4. 把 harness 二进制当成 shell prompt 主题来对待。立场可以鲜明,但要随时能换,周二下午就能换。

"被 harness 淹没了"这个感觉是真信号。但解药不是挑一个然后死撑。解药是认清:harness 下面那一层——协议层——终于到了,把投资往下挪一层,挪到那个不再天天 churn 的地方。经历过编辑器大战的人都知道接下来的剧本。插件输了,协议赢了。当年早早上船的厂商留住了用户,没上船的被改名改进了坟墓。

Crystal 现在叫 Nimbalyst。Goose 现在叫 aaif-goose。阀门已经开了,放点压力出来吧。

由人类与他的 agent 共同打造。2026
