依赖能不升就别升:Mitchell Hashimoto 喊了十年的"歪理"又火了

一个 moving 的 Docker tag 在生产环境里直接往回退了。litellm:main-latest 拉下来,版本是 1.82.6。我们正打算升级的那个容器,本来跑的是 1.83.4。"保持最新"的工具链,给我们送了一个比原来更老的版本。这是周二发生的事。到了周五,Mitchell Hashimoto 那条"永远不要升级依赖"的推已经 5436 个心,评论区彻底炸了。

这条 5436 心的推,其实是十年前那一条

2026-05-20 这天,Mitchell Hashimoto——Ghostty 作者、HashiCorp 前创始人、当年 Vagrant 和 Terraform 的那个人——发了这条推:

"Fork 你的依赖,把它裁剪到只剩你用得上的部分,除非用户那边坏了否则永远不要更新。这话我说了十多年了。我一直觉得,升级带来的风险远大于潜在 bug 的风险——潜在 bug 是可以追踪的,CVE 是可以监控的。"

12 小时 148 条回复。大部分人震惊,一部分人愤怒,还有意外多的人——悄悄表示同意。这条推真正值得说的地方是:他从 Vagrant 那个年代就在讲这套。推文不是新的,听众才是。

十年前你听这话,会觉得是 Twitter 上的怪人。在 2026 年再听,这是关键运维建议——因为支撑你所有有意思应用的那一层底座(agent 栈、LLM 工具链、扩散模型 pipeline),已经把依赖树变成了雷区。diffusers 在 import 的时候硬性检查 peft>=0.17.0,而它甚至没在 requirements 里声明这个依赖。transformers 4.51.3 需要 huggingface-hub<1.0,>=0.30.0,但默认 pip resolver 会给你装 1.13.0,生产环境直接 import 失败。半打"agent SDK"各种各的 vendor 方式,每次发布都是抛硬币。

"为了安全总是要升级"这个福音,正在被它自己的工程学压垮。能稳定交付的团队,都是 2014 年就读过 Hashimoto 的那些人。

"总是升级"是工程剧场

整个行业的默认立场——每一个 dependabot 配置、每一个"你有 47 个过时包"的 lint 提示、每一个 supply-chain 厂商的 marketing slide——都在告诉你新的就是更安全的。这是一个范畴错误。它把 known-bad 和 known 混为一谈了。

Hashimoto 的说法很精准:潜在 bug 是可追踪的。CVE 会被提交,会进扫描器,有严重等级,你可以做决定。但你 upstream 下个 minor release 里出现的 regression?这是一个出现在你没写的代码里、你不知道存在、却把你用户依赖的功能搞挂了的 bug——你只会在 Slack 里有人问"API 是不是挂了?"的时候才发现。这种 bug 没扫描器,只有爆炸半径。

"总是升级"派假装不存在的三个 tradeoff:

1. 升级风险 vs 潜在 bug 风险。已经在生产里跑了几个月没冒头的潜在 bug,统计上明天冒头的概率很低。但新的代码,从来没在你的环境里跑过。"新发布版本里有 regression"这件事的基础概率并不低——去看任何一个活跃项目 minor 升级后那一周的 bug tracker 就知道了。可我们居然把升级当默认安全选项。

2. Maintainer 的 churn vs 你的 churn。upstream 每改一次 API,这次迁移的账单是你来付。Maintainer 做了一个免费的决定,你被收了费。Upstream maintainer 不欠你稳定性——他们欠自己一个能维护下去的代码库。这个错配是结构性的。Fork 是对它唯一诚实的回应。

3. 供应链安全是可读性,不是新鲜度。xz-utils 的后门,被 ship 得最快的项目就是那些 auto-pull 的项目。防御从来不是"24 小时内打补丁",防御是"你知道你的树里有什么"。一个被冻结、vendor 进来、审过的依赖,比一个跨六层 transitive 信任链自动升级的依赖,要难被攻陷得多。读你的依赖才叫安全。刷新它们只是在祈祷。

来一份我们自己基础设施的账单。Flux2 推理容器的 pin 矩阵:

diffusers==0.37.1          # Flux2Pipeline 在 0.36.0 引入;0.38 会引发 safetensors 连锁冲突
transformers==4.51.3       # DLC 默认版本;锁定 huggingface-hub 范围
huggingface-hub==0.36.2    # 1.x 会让 transformers import 挂掉;pip 默认会装 1.13
peft==0.17.1               # diffusers 在 import 时做 dep_version_check —— 没在 requires_dist 里声明
safetensors==0.6.2         # DLC 默认;升 diffusers 0.38 才需要 0.8;不要动

这里每一个 pin,都是用生产环境调试时间换来的。一次轻松五个小时。把这种数量级的脆弱乘以 agent 栈里大概四十个等量级的包,数字就很难看了。冻结的 lockfile 不是偏执——它是给未来的你的一封备忘录,告诉你哪一些 pin 是用血换来的。

"Fork、裁、冻"在 2026 长什么样

Hashimoto 的"歪理"不是一种 vibe,是三个具体的模式——每个模式都通过消除一整类失败来证明自己复杂度的合理性。下面是我们真正在用的三种,按承诺程度从低到高:

Vendor 给前端用。Showcase 那个 wizard 前端用了 Alpine.js。Alpine.js 有 CDN。我们不用。我们把 alpine.min.js 直接丢进 /static/vendor/,完事。代价:一个文件,大概 40KB,进 git。收益:wizard 不会因为 CDN 出 regression 而炸,不会被 CDN 攻陷影响,不会因为某个 Cloudflare 的人改了路由而坏掉。今天跑的 JavaScript 跟我们 ship 那天跑的是同一份。这就是全部。

Pin 给运行时用。SageMaker 的 async endpoint 跑在私有 VPC 里,出网完全不通。pip install 直接 timeout——网络到不了 index。包能进容器的唯一方式是 vendor 一个 wheelhouse,版本写死,加上 --no-index --find-links /opt/ml/model/code/wheelhouse。听起来很严苛。是很严苛。但这也意味着我们的 endpoint 不可能因为 transitive 升级而被悄悄搞坏。Air-gapped 基础设施强加给你的约束,正好是 Hashimoto 推荐你主动接受的约束。

Branch 给产品用。MeshClaw——我在 Cloud Desktop 上跑的 agent runtime——是一个 fork。upstream maintainer 不想要 Secretary 邮件功能。我们想要。所以 feature/secretary-email-support 就在本地存着,upstream 出了真正重要的东西的时候 rebase 一下,其它时候原地不动。下次 upstream ship 一个 breaking refactor 的时候,我们的 agent 还在跑。代价就是一个 branch,加上不要假装自己在往上游推的自律。收益是这个功能存在,而且按我们的节奏存在。

三种模式的共同点是同一个:你是 integrator,不是 consumer。你的工作不是追 upstream 的 release notes。你的工作是知道此刻你生产环境里跑的是什么代码,而且只在你的用户需要的时候才动它,不是 npm/pip/cargo 醒了决定该动它。

反共识的那个结论

大部分工程师读 Hashimoto 会觉得他在装老顽固。他不是。他只是在运维上诚实地说出了大部分团队不愿意承认的事实:你 import 一个包的那一刻,你已经把你可靠性的一部分外包给了一个素未谋面、对你的客户没有任何义务的陌生人。

升级是一种押注——押你的 upstream maintainer 关心你的 use case 多过关心他自己的路线图。有时候这是个不错的押注。很多时候不是。2026 年的 agent 栈大部分时间是"很多时候不是"——项目太多、改动太多、太多 maintainer 在追下一个范式。litellm:main-latest 之所以倒退,是因为某个人、在某个地方、对一个 tag 指针做了决定。那个决定不是为我们做的。任何一个都不是。

悄悄说的真相:fork、裁、冻,是当底层比你 roadmap 跑得还快的时候,"交付"长成的样子。它不是 2026 才发明的。Hashimoto 十年前就发现了——他看着基础设施工具生态烧穿自己的贡献者。他一直在说,因为没人听。这条推文今天能火,是因为听众终于积累出了能听懂这话的伤疤。

如果你正在 agent、LLM 或者任何 2024 年之后的工具生态上面构建严肃的东西:读你的依赖,fork 关键的那部分,vendor 小的那部分,pin 运行时那部分,你的用户不逼你之前一切不要动。把每一个 upgrade 命令当作一次 deploy 看待——因为它就是——并且按 deploy 的标准 gate 它。

"总是升级"派会说这不负责任。问问他们上一次读他们刚 bump 的那个包的 changelog 是什么时候。诚实的答案,几乎永远是从来没有。那才是真正的不负责任。Hashimoto 只是从 2014 年起就有勇气把这话说出来。
