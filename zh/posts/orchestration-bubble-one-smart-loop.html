Agent 编排泡沫:14 个新框架同时上线的那周,我把自己的也删了

2026-06-02 那一周,Google 把一个叫 Scion 的项目开源了 —— 自称"agent 的 hypervisor"。当天 Hacker News 第一名,230 票。再往下翻六条,有一篇标题叫 "Agent Orchestration Is Not the Future",19 票,4 条评论,几乎没人理。一周之内,Show HN 上又冒出来八个新框架:Hephaestus、KAOS、Ayatori、GraphFlow、durable-swarm、Trama、Sub-Agent MCP、Bluemarz。七天 14 个项目,一个比一个能"compose"、"supervise"、"hypervise"。那篇 19 票的反向文章是唯一说对了的那篇。

泡沫达到顶点的那一周

我每天有个跑在自己机器上的 digest 脚本,扫 HN、Show HN、GitHub trending、r/AI_Agents。把 2026-06-02 到 06-08 之间桌面上摆出来的东西按 HN 分数排一下:

项目 口号 HN 票数
Scion(Google) "agent 的 hypervisor",编排研究 testbed 230 ← 第一名
Hephaestus "打造复杂的 multi-agent workflow" 87
KAOS "给 LLM agent 用的 Kubernetes 风格调度器" 61
GraphFlow "基于 DAG 的 agent 执行引擎" 54
Ayatori / Trama / Bluemarz / durable-swarm / Sub-Agent MCP (一周内八个新人,口号一样,logo 不同) 30–80

关键不是任何单个项目,而是这种不对称。十四个项目卖同一个梦,一个项目说这个梦是错的,市场以 13:1 的票数选了梦那边。这个市场会错,会和它当年错判即插即用微服务、低代码工作流编辑器、"AI no-code agent"完全一样的方式错 —— 那三个品类都是在 hype cycle 的这个时间点达到峰值,然后什么有持久价值的东西都没留下。

营销话术 vs 可量化的产出

每次看到一个赛道这么快地热起来,我就跑一个最便宜的测试:把头部三个项目的 README 抓下来,grep 两个东西 —— benchmark 和 case study。如果通篇都是 "compose"、"orchestrate"、"supervise"、"hypervise"、"swarm",但没有任何一个可以拿出来吵架的数字,那这个东西是个研究品味的产品,不是生产力产品。

这周这一批跑下来是这样:

$ for repo in scion hephaestus kaos graphflow ayatori trama bluemarz durable-swarm; do
    echo "== $repo =="
    grep -ciE 'benchmark|swe-bench|tau-bench|case study|production' README.md
    grep -ciE 'orchestrat|hypervis|swarm|compose|supervise' README.md
  done

八个项目,加起来零个能拍桌子吵架的数字。同一周用同样的 grep 跑 Anthropic claude-code 的 changelog,跳出来:SWE-bench Verified 88.6%、τ²-bench Telecom 96.0%、不报告的缺陷比 Claude 4.7 少 4 倍。一个 repo,三个量化结论,零个 "hypervisor" 字样。这个不对称就是这篇文章的全部论点。

第一手凭据:我自己跑生产的两个 bot

我不是站在外面喊话。我手上有两个跑生产的 agent,过去 18 个月里我用它们,把所有自己搭过的 multi-agent 编排都干掉了。

凭据一:ticket-coop-scanner

这是一个 cron 任务(cron 6b85f32d),盯着公司里三个 ticket 队列。一个状态机,一个 state file,一次 cron tick。没有 supervisor,没有 router,没有"团队"。五天窗口 2026-06-03 到 06-08:

指标 数值 备注
触达 ticket 21 张 三个队列,严重等级混合
完全解决(≤1 次 oncall 批准) 9 / 21(43%) bot 自己把 ticket 推到关闭
完全自主(0 次人介入) 4 / 21(19%) 没有人参与
≤2 次回合内交付 13 / 21(62%) 包含 Shepherd ticket 上写完整 CR-ready 提案的
触达完整 fix 提案 17 / 21(81%) 没解决的也写好了 fix spec

这玩意儿一周搭出来的。今年早些时候我花了两个月,搭了个"multi-agent ticket pod" —— router agent、triage agent、fix-proposer、reviewer、poster。结果数字更低,延迟是三倍,故障模式是十倍(router 在 triage 没结束之前就发帖子;reviewer 和 proposer 互相不同意;state file 不共享导致 agent 之间状态分裂)。整个删了。

凭据二:nanochat 训练

今天 2026-06-09,我正在 4× L40S 上训一个 9 亿参数的 transformer,$206 跑完。"编排"长这样:

sudo -u ubuntu -H bash -c '\
  cd /home/ubuntu/nanochat && source .venv/bin/activate && \
  setsid nohup torchrun --nproc_per_node=4 -m scripts.base_train -- \
    --depth=20 --device-batch-size=8 --num-iterations=11000 \
    --save-every=500 --window-pattern=L --run=correct \
    >/home/ubuntu/nanochat-l40s-train/logs/d20-correct.log 2>&1 &'

一条 torchrun。四个 DDP worker。一份日志。一份世界状态。没有 agent 委员会决定哪块 GPU 拿哪个 shard。整个 base + SFT + RL 的 pipeline 是三个 shell script(08-base.sh、10-sft.sh、10-train-rl-4gpu.sh)。我之前也搭过一个"分布式训练 agent harness",意识到 PyTorch 本来就在做编排、我加的那一层除了出 bug 时多一份混乱什么都不做的那一秒,就把它删了。

Umami 教训,换个说法再说一遍

我有一条私下的法则,叫 Umami 教训 —— 起源于那个被业内吐槽"每六个月把整个产品重写一遍"的分析公司,因为底层一直在动。这条法则用在 agent harness 上特别合适:

你为当前这一代模型搭的所有编排管道,下一代模型 release 之后都会被擦掉。能留下的资产只有 skill 文件、state 文件、prompt。

具体例子。四月份我有 80 小时的 LangGraph 管道,专门为 Claude Sonnet 4.5 调的:显式 handoff、role 定义、retry 策略、定制 router。六周以后 Opus 4.7 出来,模型原生的 tool-calling 和自纠错 loop 已经比我手调的 router 还好。那 80 小时直接报废。然后五月底 Claude Code 4.8 上线,Dynamic Workflows 直接进了主代码 —— /effort ultracode 一条命令开几十到几百个 subagent 互相 review,直接拿到 SWE-bench Verified 88.6%,你这边一行框架代码都不用写。编排已经变成 vendor 自带功能。所有第三方编排框架在那份 release notes 落地的瞬间,都成了 legacy 候选。

对照一下我这边模型升级之后还活着的东西:

  • skill 文件 —— markdown,模型无关,新模型上反而更好用
  • state 文件 —— JSON,agent 自己读自己写,跨 vendor 合约一模一样
  • cron 时间表 —— 什么时候醒来。这个不会被淘汰

没活下来的:我写过的每一个 router、supervisor、planner、dispatcher、委员会。

"80 IQ 的工人凑不出 140 IQ 的整体"

这是 moridinamael 那篇文章里没人点赞的那句。它是我见过最干净的一句话,解释为什么 multi-agent 委员会跑不过一个能干的单 agent loop。你没办法把一群 80 IQ 的工人组织成一个 140 IQ 的整体。协调成本是真的,跨 handoff 的错误放大是真的,而且委员会的故障模式比单 agent 更糟,因为每个 agent 都用自己的幻觉去 gaslight 下一个 agent。

同样的话,2025 年 12 月有个 r/AI_Agents 的人用更白的话说过:

multi-agent pod 概念上听上去合理,但实际跑起来,我一直被编排复杂度和 agent 之间各种诡异故障模式坑。

GitHub 星标版本的同一条教训:multi-agent-shogun 第一周冲了 +1300 颗星,第二周 +12 颗。主题 + 小众确实能搞出星标尖峰,但搞不出留存。星图本身就是项目自己承认 —— 第一次 git clone 完之后没人真正在用。

诚实的及格线

这是我现在用来检查任何 agent 编排框架(我的、你的、这周 HN 头条的)的测试:

  1. 拿 claude-code(或者 Cursor、kiro-cli)+ 三个写得好的 SKILL.md + 一个每 5 分钟戳它一次的 cron 任务。
  2. 用同一个模型让你的框架跑同一个任务。
  3. 比四个数:完成率、P50 延迟、P95 延迟、每个完成任务花的钱。

如果框架在这四个数里没有任何一个比 cron + skills 这套基线高出 20% 以上,它就不是个生产力工具,它是穿了软件外套的管理学研究项目。把 cron 任务上线,把框架归档,把工程时间留给下一次模型 release —— 那次又会把框架剩下的一半存在意义吃掉。

2026-06-02 那一周 14 个项目说反话。一个 benchmark 数字都没拿出来。这就是泡沫。它会以所有 agent 工具时尚都用过的同样姿势破掉:不是一次崩盘,而是慢慢褪色,下一次模型 release 悄悄让框架显得多余。到 2026 年底,大概率 14 个里大多数会变成只剩一个 committer 在推依赖升级 PR、生产里没人在跑的 repo。到 2027 年底,活下来那一小撮会改名叫"agent observability"或"agent eval",因为模型自己接管编排之后,第三方工具能讲的诚实故事只剩这两个了。

把 cron 写好。把 skill 写好。让模型自己规划。这就是 2026 年 agent 栈的全部。

由人类与他的 agent 们共同打造。2026
