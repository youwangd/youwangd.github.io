别再给 Agent 加记忆了，它缺的是一把剪刀

我的 agent memory 目录在硬盘上 40MB。每次塞进 prompt 的那一片是 55KB。1000 倍的压缩比，就是产品本身。大家正在抢着做的那个存储层，是这整个问题里最简单的那一半——而只解决简单的那一半，会让 agent 变差，不是变好。

瓶颈从来不是存储

我看了一年的 "agent 记忆" 话题，一直在被当成存储问题讨论。更大的数据库、更好的 embedding、更花的图结构、向量库、hybrid retrieval。全部都在服务同一个动词：记更多。

但如果真的是存储瓶颈，context window 从 200K 涨到 1M 早就把问题解决了。并没有。我的 agent 没有因为窗口变大而变聪明——反而变得更糊涂。开始引用我早就推翻的 preferences、贴出半年前就不存在的文件路径、把互不相干的项目的事实揉在一起。窗口越大，信号越差。

"记更多" 这套框架把方向搞反了。你往窗口里多塞一个 token，就挤出去另一个——而且挤出去的通常是最新、最相关的那个。因为 attention 对 recency 有硬伤，retrieval 的 precision 也有硬伤。给 agent 加更大的 memory 但不做筛选，等于给了它一个更大的草垛，然后管这叫搜索引擎。

瓶颈不是 persistence，是 eviction：agent 历史上见过的所有东西里，哪 5% 应该进下一个 prompt，其他的该怎么扔出去腾位置。

我自己写的时候，真正管用的三条规则

我在自己的 agent 里做了个 workspace 级的记忆层：持久化 preferences、每个项目独立的 notes、带衰减的 history、语义 key-value store。硬盘上是一堆 markdown 加一个 SQLite 文件。一年用下来现在大概 40MB。

把存储层搭起来花了一个周末。让 agent 的行为真的变好，花了六个月调这三条规则——没有一条是在讲"怎么存"，全部在讲"拒绝塞什么进去"。

规则 1：硬上限，远低于任何模型的窗口

# mesh_claw/context.py
_MAX_CONTEXT_CHARS = 55_000

Opus 有 100 万 token 的窗口。我最多塞 55K 字符的 memory，不多塞一个字。这个上限不是省钱决策，是质量决策。过了这个体量，信号被稀释的速度比新信息带来的价值快。模型开始混淆项目、引用我早就推翻的老 preference、给出早已挪窝的文件路径。上限存在的理由是——它逼着它上面那一层必须做决定。

规则 2：衰减权重，新的压老的

History 条目过 90 天开始指数级降权。昨天的一条纠正远远压过去年十月的一条。没有衰减，记忆层就是个博物馆——每件文物都在，但没一件被排到前面。有衰减，它才是一张工作台：你最近动过的那个东西，就是你最先找得到的那个东西。

规则 3：workspace 隔离，一个项目的噪声不污染另一个

不同 workspace 各自有各自的 memory。我在 fee 相关服务里学到的东西，不会在我搞 quant repo 的时候冒出来。事后看这条很 trivial，但几乎没人做——因为"通用记忆"在 README 里比"按目录沙盒"听起来高级。"通用记忆"也正是 agent 开始在无关 codebase 之间胡乱串味的方式。

硬盘 40MB、注入 55KB。这 1000 倍压缩才是产品本身。我在这套东西上花过的每一个小时，都花在压缩上，没有一个小时花在存储上。

它应该叫"上下文剪辑器"

我们真正需要的原语不是"记忆层"，是一个上下文剪辑器(context editor)：一个小而有主张的组件，跑在大模型之前，只回答一个问题——根据下一轮要干什么，这次不带什么？

具体长这样：

# 几乎每个"agent 记忆"设计里都缺的那一层
def edit_context(memory_store, next_turn_intent, budget_chars):
    candidates = memory_store.all_relevant(next_turn_intent)
    # 按 recency、scope match、历史有用性打分
    scored = score(candidates, intent=next_turn_intent)
    # 默认丢，不默认留
    kept = []
    for item in scored:
        if item.score < THRESHOLD: continue
        if len("".join(kept)) + len(item) > budget_chars: break
        kept.append(compress_if_needed(item))
    return kept

剪辑器做三件存储原语不做的事：

  • 默认是扔。Retrieval 的默认动作是"返回"——top-k 给你端上来。剪辑器的默认动作是"省略"，你要让它加进去得给理由。姿势不一样，结果差别很大。
  • 原地压缩。一份 2000 token 的对话历史，在塞进窗口之前先被摘成 120 token 的笔记。压缩由剪辑器负责，大模型永远看不到原文。
  • 预算硬卡。每一片 context 都有 budget。如果是 55K，剪辑器的任务是做出"最好的 55K"——而不是把"所有相关的东西"都扔给模型然后祈祷 attention 自己能排序。

剪辑器可以是一个小模型，也可以是一个规则引擎。它不需要聪明，它需要有主张、可调、可见——这三条是向量库、图数据库、retrieval pipeline 结构上都给不了的。

为什么这件事一直重要

你 prompt 里的每一个 token，都是你在付钱的 token、是模型 attention 必须摊薄的 token、是本可以用来放更有用的东西的那个位置。窗口变大、这个算术不变。你的 memory store 变花、这个算术不变。只有当你栈里有某个组件被授权扔东西，这个算术才变。

(本周一个开发者退订 Claude 的帖子冲到了 HN 第二热，946 分 565 条评论。吵的不是模型质量，吵的就是这件事——一个看不见的 token 层，悄悄驱逐了他的缓存、悄悄让他账单翻倍、悄悄把模型推向更便宜更懒的输出。同一个算术，换了个表面。如果"agent 记忆"变成一个"只做存储、不做剪辑"的品类，我们就是在自己的基础设施里把同一套失败模式重新搭一遍——只不过这次牌子上写我们自己的名字。)

下一个原语不是更好的 embedding、更聪明的图、更花的 schema。是一个小的、可见的、可调的组件——它唯一的工作就是把东西扔掉。把剪辑器做出来。存储那半，是简单的那半。
