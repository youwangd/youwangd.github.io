别再给 Agent 加记忆了，它缺的是一把剪刀

我的 memory 目录在硬盘上 40MB。每次塞进 prompt 的那一片是 55KB。压缩比 1000 倍,整个产品就是这件事。本周 HN 上同时上线了三个「开源 agent 记忆层」,没有一个在讲这件事。它们都在讲那 40MB 怎么存。

退订 Claude 那篇,吵的是 token,不是模型

周一 HN 第二热是一个开发者解释为什么退订了 Claude。946 分、565 条评论。你把整条讨论翻一遍就会发现——没人在认真讨论 Opus 变笨了。大家吵的是 token 的透明度。

作者的核心抱怨,一句话:

"你回到工位继续干活,对话缓存已经没了,模型又要把你的代码库从头读一遍。从成本角度这很聪明。但从体验角度?你花 token 加载了一次,强制中断之后……同一份东西你要再付一次钱。"

然后他贴了 Opus 自己思考日志里的一段——缓存蒸发之后,模型是这么想的:

"与其一个个改 JSX 里的 slider,不如加个通用初始化器……这明显是个坏习惯。"

模型亲口告诉你:我正在挑一个偷懒方案,因为我要省自己的 output token。这才是真正激怒用户的点:不是模型变差,是 token 这层变得肉眼可见,而且可见地在对着你玩。缓存失效让你重复付费,不公开的月度限额对话中途弹出来,agent 偷偷降低自己的工作质量来压账单。用户的愤怒是——我搞不清楚自己在为什么付钱。

记忆层淘金潮,全跑偏了

Claude 那篇刚火,同一周 HN 上蹦出来三个「agent 记忆」项目:

  • wuphf(110 分)—— "Karpathy 那种 LLM wiki,让 agent 自己维护"
  • stash(83 分)—— "开源记忆层,让任何 AI agent 都能做到 Claude.ai 和 ChatGPT 能做的事"
  • MenteDB(16 分)—— "给 AI agent 用的开源记忆数据库,Rust 写的"

三个都在做存储原语。没有一个在做筛选原语。README 里谈的是 schema、embedding、向量库、hybrid retrieval。关于驱逐策略、recency decay、什么该扔掉——零字。

这是方向错了。如果瓶颈是存储,那 context window 从 200K 涨到 1M 早就把问题解决了。并没有。你往窗口里多塞一个 token,就会挤出去另一个——而且通常挤出去的是最新最相关的那个,因为 attention 对 recency 的处理有硬伤,retrieval 的 precision 也有硬伤。给 agent 加更大的记忆但不做筛选,等于给了它一个更大的草垛,然后管这叫搜索引擎。

这事和退订 Claude 那篇是同一条线:你「记忆层」往窗口里塞的每一个 token,你都要付钱。一个没有剪辑器的记忆层,正是 HN 作者愤怒的那种东西——一层看不见的基础设施,默默把一堆他没主动要求加载的 context 塞进来,然后向他收费。

我自己写的时候,真正管用的三条规则

我作为 MeshClaw 的一部分做了个 workspace 级的记忆层:持久化 preferences、每个项目独立的 notes、带衰减的 history、语义 key-value store。硬盘上是一堆 markdown 加一个 SQLite 文件。一年用下来现在大概 40MB。

真正提升 agent 质量的那几条教训,几乎都跟「怎么存」无关。都跟「拒绝塞什么进去」有关。

规则 1:硬上限,远低于任何模型的窗口

# mesh_claw/context.py
_MAX_CONTEXT_CHARS = 55_000

Opus 有 100 万 token 的窗口。我最多塞 55K 字符的 memory,不多塞一个字。这个上限不是省钱决策,是质量决策。过了这个体量,信号被稀释的速度比新信息带来的价值快。模型开始混淆项目、引用我早就推翻的老 preference、给出早已挪窝的文件路径。上限本身就是逼着上游筛选层动起来的一种约束。

规则 2:衰减权重,新的压老的

History 条目过 90 天就开始指数级降权。昨天的一条纠正远远压过去年十月的一条。没有衰减,你的记忆层就是个博物馆——每件文物都在,但没一件被排到前面。有衰减,它才是张工作台。

规则 3:workspace 隔离,一个项目的噪声不污染另一个

不同 workspace 各自有各自的 memory。我在 fee 相关服务里学到的东西,不会在我搞 quant repo 的时候冒出来。事后看这条很 trivial,但没人在自己的产品里做——因为「通用记忆」在 README 里比「按目录沙盒」听起来更高级。通用记忆也正是 agent 开始在无关 codebase 之间胡乱串味的方式。

硬盘 40MB、注入 55KB,这 1000 倍压缩才是产品本身。存储那半是简单的那半。

它应该叫「上下文剪辑器」

我们真正需要的原语不是「记忆层」。是一个上下文剪辑器(context editor):一个小而有主张的组件,跑在大模型之前,只回答一个问题——根据下一轮要干什么,这次不带什么?

具体长这样:

# 伪代码 —— 每个「agent 记忆」项目都缺的那一层
def edit_context(memory_store, next_turn_intent, budget_chars):
    candidates = memory_store.all_relevant(next_turn_intent)
    # 按 recency、scope match、历史有用性打分
    scored = score(candidates, intent=next_turn_intent)
    # 默认丢,不默认留
    kept = []
    for item in scored:
        if item.score < THRESHOLD: continue
        if len("".join(kept)) + len(item) > budget_chars: break
        kept.append(compress_if_needed(item))
    return kept

剪辑器做三件存储原语不做的事:

  • 默认是扔。Retrieval 的默认动作是「返回」——top-k 给你端上来。剪辑器的默认动作是「省略」,你要让它加进去得给理由。姿势不一样,结果差别很大。
  • 原地压缩。一份 2000 token 的对话历史,在塞进窗口之前先被摘成 120 token 的笔记。压缩动作由剪辑器负责,大模型永远看不到原文。
  • 预算硬卡。每一片 context 都有 budget。如果是 55K,剪辑器的任务是做出「最好的 55K」——而不是把「所有相关的东西」都扔给模型,然后祈祷 attention 自己能排序。

本周 Cloudflare 的 Code Mode 从另一个角度论证了同一件事:agent 应该住在状态机里,而不是每轮重新生成文本。高一层看是同一个洞察——长期的胜利在于不重新 materialize context,而不是「存更多」。一个直接对着稳定状态机编程的 agent,就是一个每轮不需要把处境重新加载进 prompt 的 agent。更少的 context 本身就是 feature,不是妥协。

回到退订 Claude 那条线

HN 作者真正气的不是 Anthropic。他气的是——决定什么东西进他 prompt 的那层,是不透明的、对着他玩的、还要收钱的。他看不到、调不了,而且供应商有一切动机为了省自己的钱去驱逐他的缓存。

如果「agent 记忆」这个品类火起来,而进来的每个项目都只做存储、不做剪辑——我们就会在自己的基础设施里把同样的失败模式重跑一遍。一个 memory DB 每轮欢快地往 prompt 里灌 200K token 的「相关历史」,你为此付钱,模型丢了主线,而你根本不知道该砍掉哪一半。

下一个原语不是更好的 embedding、更聪明的图、更花的 schema。是一个小的、可见的、可调的组件——它唯一的工作就是把东西扔掉。把剪辑器做出来。存储那半,是简单的那半。
