Agent 写 Go 比写 Python 还猛，是因为 Go 早在 2012 年就"预埋"了 AI 需要的东西

周一 Mitchell Hashimoto——就是那个一手搞出 HashiCorp 全线产品、这几年满世界安利 Zig 的人——发了条推特：他又开始写 Go 了，而且说 go doc 和 gopls 是 agent 的"超能力"。2398 个赞。推文背后真正有意思的一句没说出来：他不是在夸 Go，他是在描述一门 2012 年设计的老语言，它那套设计决策恰好就是 LLM 写代码时最需要的接口。没人计划过这件事。所以才值得聊。

那条 2398 赞的推特

原文照抄：

"我又开始写 Go 了……go doc 和 gopls 简直是 agent 的超能力。开箱即用的情况下，agent 写 Go 的生产力高得离谱——比我用过的其他语言（包括整个 JS 生态）都猛。另外 Go + Zig 是很好的搭配。" — @mitchellh, 2026-04-21

仔细读一下。Hashimoto 不是在说 Go 是门好语言——这个他早就知道。他是在说：agent 在 Go 上的表现好到让他自己都吃惊，而且是零配置、开箱即用。对比对象是 JS——训练数据多了一个数量级、库多、Stack Overflow 答案多。让他惊讶的不是 Go 的质量，是这个差距。

我点进他那个 go-libghostty 仓库看了下他到底怎么搞的。.agents/skills/ 里有两个自定义 skill：discovering-upstream-apis 和 upstream-api-feedback。两个都是围绕"怎么查 Go 的工具链"写的。没有 prompt engineering，没有 RAG，就是告诉 agent：哪个 go 子命令回答哪个问题。

agent 一整天到底在干什么

先把 benchmark 分数放一边。你盯着一个 agent 改真实代码库的时候，它的内循环每一轮都长这样：

  1. 找到要改的函数。
  2. 看函数签名和文档——它承诺的是什么？
  3. 找所有调用方——我改了会破坏什么？
  4. 改。
  5. 确认还能编译。
  6. 跑相关测试。

六次查询。每次查询都得以 agent 能消化的形式返回结果。重点问题不是"这门语言工具链强不强"——但凡正经语言都有工具链。重点问题是：一个没做任何配置的 agent，能不能用一条 CLI 命令就拿到答案？

老实排一下：

六个操作。Go 每一个都是一条标准命令，随语言工具链自带，零配置。而 Python——AI 库最多的那门语言——agent 要先猜出这个项目用的是哪个 linter，才能开始 lint。

2012 年的设计决策，2026 年开始收租

Rob Pike 和 Ken Thompson 在 2012 年根本没想过 LLM。他们想的是 Google 那种巨型单体仓库，还有怎么让新员工快速上手。但他们做的那些决策——无聊得近乎强迫症——恰好就是一个"上下文极度匮乏"的 agent 需要的东西：

  • 代码格式只有一种。 gofmt 是强制的，不可配置。agent 根本不需要推断代码风格——因为"风格"这个决策点根本不存在。
  • 文档是一等公民。 Godoc 注释就写在函数上面，go doc 直接读。不存在"这个团队用的是 Sphinx 还是 MkDocs 还是啥都没用"这种问题。
  • 一条命令构建。 go build。不是 make，不是 cmake，不是 ./gradlew --refresh-dependencies clean build -x test。
  • stdlib 扁平、好发现。 150 个包，每个包解决一个显而易见的问题。net/http，而不是 @tanstack/router-react-async-experimental。
  • 错误作为返回值。 agent 能看见错误路径，因为它就在函数签名里。没有 try/except 那种隐藏的控制流。

这些决策在 2012 年都是在牺牲表达力。每一条都让代码读起来无聊、但机器看起来清晰。当年是审美选择，现在是对齐选择。

Python 的反驳，以及为什么没看起来那么站得住脚

最明显的反驳：agent 是在 Python 上训练的，Python 多 = agent 写 Python 厉害。这话没错，但这是一次性优势。训练数据是固定成本。工具链的"机器可读性"是每次查询都要付的成本。你的 agent 每次重新进入代码库，都要再付一次。

这事我在工作里天天体验。SageCLI 是 Python + shell。公司的生产 monorepo 里有个 Kotlin CDK 包。我让 agent 改 Python 函数的时候，它要读 3-5 个文件才能拼出调用图。我让它改 Kotlin CDK 的时候，gradle dependencies 加上 Kotlin 显式的 import 声明，让它只读一个文件就能干活。Python 那边的 agent 烧 4 万 token 探索，Kotlin 这边 8 千 token 就发 PR 了。

Armin Ronacher——写 Flask 那位——去年十月在 Pragmatic Engineer 上也提过类似观点：生产级 AI 栈正在分叉。TypeScript 和 Rust/Go 负责基础设施（agent、runtime、协议），Python 被推回到"库板凳"的位置。看看现在在发车的东西：Cloudflare 的 agent SDK？Go。Anthropic 的 mcp-go？Go。HashiCorp 全线？Go。MCP 协议现在有十个官方 SDK——Go 和 Python、TypeScript 一起在第一梯队。

DHH 当然不同意。他 4-20 那条推特（652 赞）说："AI 模型还没好到能在大型关键应用上一发入魂生成可以合并到 master 的代码。"他说得没错，也没说到点子上。没人在搞"一发入魂"。我们讨论的是迭代的内循环，不是最终产物。Ruby 的问题不是语言差，而是 bundle、rails、rspec、rubocop、sorbet——五个独立工具五套配置。agent 得先学会这个项目的"方言"才能开工。

如果你用的语言不是 Go，该怎么办？

我们大多数人没得选语言。我就没得选。SageCLI 必须是 Python，因为它得跑在用户已经有 Python 的地方。那怎么办？

自己造那条语言没给你造的"元命令"。具体来说：

  1. 一条 project doctor 命令，把 format、lint、类型检查、测试全部聚合进去，输出成 agent 能查的单一格式。Makefile target、just recipe，什么都行——只要是一条命令，输出稳定。
  2. 一个标准依赖图查询。 如果 agent 要在 pipdeptree、uv tree、poetry show --tree 之间做选择，你就已经输了。挑一个，起别名，写文档。
  3. AGENTS.md 指向这些命令。 不要写成散文式的"代码库介绍"。要写成一份命令手册，agent 能 grep 的那种。

上周我在公司一个 Kotlin CDK 包上就是这么干的。写了 AGENTS.md，跑了 8 轮 AutoSDE 静态分析，累计 40+ 条 comment，一条一条都改了。那份文档的内容没那么重要——这个过程本身才重要：它逼着代码库变得可被 agent 查询。每一条"agent 搞不清楚"的 comment 背后，都是一个"光靠 CLI 一个人类也搞不清楚"的信号。

这就是 Hashimoto 那条推特里藏着的道理。Go 赢不是因为 Rob Pike 预见到了 LLM。Go 赢是因为他讨厌歧义，而 LLM 正好是对歧义最极端的压力测试。那门"替你做最少决策、而且把做过的决策用干净的 CLI 暴露出来"的语言，就是你的 agent 感觉"生产力离谱地高"的那门语言。

剩下的人——我们这些没赶上 2012 年设计决策的人——就得自己造这条 CLI 了。
