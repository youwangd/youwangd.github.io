2026-06-05 评测 agent 测试

你的 Agent 评测器其实在撒谎 —— 不喂错代码,你永远不知道它瞎不瞎

前两天 HumanLayer 的创始人 dexhorthy 在 X 上贴了一句听来的话,本来应该贴到所有做 agent 评测团队的会议室门口:"你没法给一个你自己都不懂的东西做评测。"35 个赞,没人接茬,没人深究。但他这话扎到一个 99% 的 benchmark 都忽视的点:你的 oracle(评测器)在偷偷给错代码判 PASS。不是"可能"会,是真的会。我们这边刚写完一个 20 个任务的 agent benchmark,验下来三个 oracle 是瞎子。bug 率 15%。如果当时不验那一遍,后面跑出来的每一个 PASS 都是凭信仰打分。

"听到一句:'你没法给一个你自己都不懂的东西做评测。'"

—— @dexhorthy,2026-06-03

没人量化的那个故障模式

2026 年 ACM Computing Surveys 那篇关于 test oracle 自动化的综述,把这个故障模式写得最清楚:"假阴性,指 oracle 对一个本应不正确的程序状态返回了真,也就是漏报了一个故障。"翻译成 agent benchmark 的语言:假阴性 = 你的评分器对错代码判了 PASS。Agent 没靠本事拿到这一分。是评测器白送了。

这不是空想。arXiv 2601.05542 那篇 《Understanding LLM-Driven Test Oracle Generation》 里专门记录了一种反复出现的 bug:LLM 生成的 oracle 用没重写的 equals() 比对,或者干脆走默认引用比较,结果错代码也照样过。oracle 根本没在检它该检的属性,它检了一个旁边的属性,而那个属性碰巧在错代码上也是真。

每个 benchmark 都有这个攻击面。SWE-bench、HumanEval、MBPP、你自己内部那个 agent 评测、你下周要写的那个,一个不漏。圈里讨论的都是这个 agent 跑的什么模型,几乎没人讨论oracle 问对问题没。这个空档,就是一切公信力翻车的地方。

Trap-positive:评测工程里最便宜、被跳得最多的一步

修法一段话讲完。在你给任何一个真实 agent 提交打分之前,把 benchmark 里的每一个任务,喂一份你明知道是错的实现给它的 oracle,要求 oracle 必须返回 FAIL。它要敢说 PASS,那不是候选人写得好,是 oracle 漏了。

具体到 bash,一个任务的 trap-positive 检查长这样:

# 任务:用一行 shell pipeline 实现 `head -n 1`。
# Trap-positive:故意写错的实现 —— 不止打印第一行,而是全打。
TRAP_IMPL='cat'   # 全文输出,跟 head 完全不一样

# 把陷阱实现喂给 oracle。
verdict=$(./oracle.sh "$TRAP_IMPL" task.json)

# 这里 oracle 必须说 FAIL。说 PASS 就是 oracle 自己有 bug。
if [ "$verdict" != "FAIL" ]; then
  echo "ORACLE BUG: trap-positive 返回 $verdict,期望 FAIL"
  exit 1
fi

就这么六行,每个任务跑一次,corpus 上线之前过一遍。秒级抓到坏 oracle,不用等论文挂出去之后再被人捅穿。

反过来讲:如果你压根构造不出一个让 oracle 拒掉的 trap-positive,那你没有 oracle,你有的是一枚加了点装饰的硬币。后面所有的 PASS 都跟那枚硬币一样不靠谱。

三个对我们撒过谎的 oracle

下面这三个 bug 是这台机器上真实发生的。我们给一个叫 goal-autopilot 的 agent benchmark 写了 20 个 trap-task oracle,验完发现 3 个有问题,15% 的瞎子率。每一个不修,都会在某个 PASS 上骗过我们。

1. grep-head-1:oracle 在抄自己的答案

任务:写一段脚本,找出包含某个子串的第一行。oracle 跑完候选脚本、抓到 stdout,然后跑 grep 在整个工作目录里找期望的那一行。陷阱实现什么都不打印。oracle 还是判了 PASS。

// 错的 oracle(简化版)
candidate_output=$(bash "$candidate_script")
if grep -r "$expected_line" "$WORK_DIR"; then
  echo PASS    // 直接 match 到 WORK_DIR 里的 ground-truth 文件
fi

// 为什么坏:WORK_DIR 里有 fixture,包括把 expected_line
// 当作 ground truth 的 task.json。oracle 是在 grep 自己的
// 答案文件,根本没看候选脚本输出。

// 修好的 oracle
candidate_output=$(bash "$candidate_script")
if printf '%s\n' "$candidate_output" | grep -qx "$expected_line"; then
  echo PASS
fi

这个 bug 写起来太顺手了,看起来也太合理了。语法没问题,逻辑读起来也对,CI 永远绿灯。它判出来的每一个 PASS 都是凭运气。

2. concurrent-counter:CPython 的 GIL 把 oracle 该抓的 race 给挡掉了

任务:写一个线程安全的 counter。陷阱实现是个故意没加锁的 counter,self.value += 1 不是原子操作。oracle 跑 100 个线程,每个加 1000 次,断言最后值是 100,000。

# trap-positive 提交(没锁,本意是要它失败)
class BrokenCounter:
    def __init__(self):
        self.value = 0
    def inc(self):
        self.value += 1   # 不是原子操作,这就是陷阱本身

# oracle 的断言
assert counter.value == 100_000   # 在 CPython 3.11 上每次都通过

为啥?CPython 字节码解释器在普通整数自增的 load-add-store 序列上抓着 GIL 不放。race 在 spec 上是存在的(算 UB),但实际跑 CPython、跑这种纯 int 操作,这个 race 根本不会被触发。oracle 在它实际部署的 runtime 上检不出它本来要检的 bug。

修法:把陷阱负载换成 GIL 会主动释放的操作 —— 边迭代边改 list,边读边写 dict,断言改成 GIL 不保护的数据结构 invariant。换了个 bug,同样的后果(数据损坏),但这次 oracle 真的会响。

这里的教训是:oracle 跟运行平台是耦合的,哪怕你以为没耦合。陷阱跑在 Python 上、oracle 又指望抓 C 层的 race,那你测的是 CPython 的 GIL,不是候选实现。

3. safe-eval-arith:$? 抓到的是错的进程退出码

任务:写一个沙箱化的算术求值器,要拒掉 shell 注入。oracle 把恶意 payload pipe 进候选脚本,然后查 $? 看候选有没有非零退出(代表它拒了)。

# 错的 oracle
echo "$malicious_payload" | bash "$candidate"
if [ $? -ne 0 ]; then echo PASS; fi

# 在 pipeline 里,$? 是最后一个命令的退出码,也就是
# `bash "$candidate"`。看起来对吧?对,前提是你真的 pipe 了。
# 但我们 trap-positive 那份提交是用 `read` 读 stdin 的,
# `read` 读到东西就返回 0,哪怕 read 之后的逻辑做了错事。
# 候选退出 0,oracle 就……PASS 了。

# 我们以为在检"沙箱有没有拒这个 payload"。
# 实际上在检"`read` 是不是返回了 0",而它几乎永远返回 0。

# 修好的 oracle
output=$(echo "$malicious_payload" | bash "$candidate" 2>&1)
rc=$?
# 既要求退出码非零,又要求输出里出现 "rejected" 字样。
# 静默退出 0 的候选过不了。
if [ "$rc" -ne 0 ] && printf '%s' "$output" | grep -q "rejected"; then
  echo PASS
fi

这一个,定位花了 40 分钟,改只用了 90 秒。诊断时间和打补丁时间的比例,oracle bug 基本都是这个调调,这也正是大家跳过这一步的原因 —— 看着贵。直到 benchmark 上线,某个带着审视眼神的人开始问尖锐问题,才知道之前省下的那点时间多么不值。

你应该先发制人挡掉的那个 reviewer 攻击

设想你出了一篇论文、一篇博客或者一份内部上线 deck,标题是 "agent X 在 benchmark Y 上得 73 分。"现在再设想一个眼神锋利的读者 —— 隔壁实验室、刁钻的 PM、NeurIPS 评审 —— 一句话回过来:"你 oracle 做过 trap-positive 验证吗?"

答案如果是"没"或"什么是 trap-positive",你那个数字就成了氛围,而且整个评论区都看到了。如果答案是 "做了 —— 我们写了 20 个 oracle,验出 3 个有 bug,这是 patch log," 故事就完全反过来。你刚刚拿出证据,展示自己的 benchmark 在认知严谨度上比这领域 95% 的榜单都高。这个数字不是氛围,这个数字有一个你真的核过的置信区间。

cline 团队这周那个 "Don't Build Slop" talk 总结得挺好:agent 成熟度的衡量,现在已经下沉到评测层,不在模型层了。Agent 跑哪个模型,那是牲口;给它打分的 oracle 是宠物。模型一周一换,oracle 要是瞎,它会污染你迄今为止以及今后发表的每一个结果。

一天的 trap-positive 工程,这就是定价。买回的是你 benchmark 当下凭信仰透支掉的所有公信力。我见过的每一个跳过这一步的团队,迟早都得做一次公开或半公开的"我们之前那个数有点问题"的认错。每一个在 corpus 上线前老老实实跑过一遍 trap-positive 的团队,从此再没出过这种事。

把 oracle 写出来,然后故意把它打趴一次。它要还判 PASS,扔了重写。这事儿就这么简单。
